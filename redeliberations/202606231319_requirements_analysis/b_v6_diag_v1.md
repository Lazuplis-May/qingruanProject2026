## 问题清单

### 问题1：管理员智能管理功能缺少用户对话端点（严重）

- **问题描述**：第4.9节描述管理员通过"自然语言对话界面"操作数据库，其底层机制为 Dify Agent（admin-manager-agent）接收自然语言后经 Text2SQL 工具回调 `/api/admin/execute` 执行 SQL。但第6节的 API 接口规格中，管理员与 admin-manager-agent 之间的对话端点完全缺失。`POST /api/admin/execute`（6.10节）仅为 SQL 执行端点（供 Dify Agent 的 Text2SQL 工具回调使用），不是管理员输入自然语言的用户对话端点。第6.11节的 `POST /api/dify/agent/:agent_id` 标注为"内部代理"接口，不应直接暴露给前端。对比 AI 智能助手模块——其用户对话端点 `POST /api/assistant/chat`（6.9节）与内部代理 `POST /api/dify/agent/:agent_id`（6.11节）是明确分离的两层——管理员模块缺少等价的用户对话端点。实现者无法确定管理员前端应向哪个 URL 发送对话消息。
- **所在位置**：缺失项，应位于第6.10节（管理相关 API），参照6.9节 AI 助手端点的模式
- **严重程度**：严重
- **改进建议**：在第6.10节新增管理员对话端点，如 `POST /api/admin/chat`（SSE 流式响应，认证要求为"是 + admin"）。该端点接收管理员的自然语言消息，Express 服务端代理转发至 Dify admin-manager-agent，SSE 流透传策略与6.4/6.9节一致。请求体参数至少包含 message（string，必填）和 conversation_id（string，可选）。

### 问题2：life_advice 表的写入机制不完整 —— AI 助手对话中如何生成并持久化健康建议（严重）

- **问题描述**：第4.8节描述 AI 助手可在对话中为用户生成健康生活建议，存储于 life_advice 表。第6.11节 Dify 工作流清单中 diabetes-assistant-agent 的输出列标注"健康建议输出写入 life_advice 表（由 Agent 工具调用触发）"。然而，全文从未定义：
  1. Agent 通过哪个 Express 端点写入 life_advice 表（是通过 `/api/admin/execute` Text2SQL 路径写入，还是需要一个专用的写入端点）
  2. 若走 Text2SQL 路径，Agent 的 Text2SQL 工具如何生成 INSERT INTO life_advice 语句并保证字段完整性（标题、标签、内容均需填充）
  3. 若需专用端点，该端点未在任何章节定义
  
  对比 health-article-generator 工作流——第6.7节明确说明其输出通过"AI 内容生成持久化路径"写入 articles 表，完整性清晰。life_advice 的写入路径相比之下仅有一个箭头标注，缺乏可执行的流程描述。实现者无法确定 Express 后端需要为 life_advice 的写入准备什么代码。
- **所在位置**：第4.8节 AI 智能助手功能描述、第6.11节 Dify 工作流清单汇总表 diabetes-assistant-agent 行、缺失处为第6.9节（AI 智能助手 API）或第5节（数据路径）
- **严重程度**：严重
- **改进建议**：在第5节"AI 内容生成持久化路径"段落的覆盖范围中明确纳入 life_advice 表，说明 Agent 写入 life_advice 的具体路径（推荐方案：在 Dify Agent 中配置专用 HTTP 工具，回调 Express 的某端点执行 INSERT；或明确走 Text2SQL 路径并说明 Agent 系统提示词中已约束生成符合 life_advice 表结构的 SQL）。在第6.9节补充说明 life_advice 写入的具体端点或路径。

### 问题3：Dify Agent Text2SQL 工具回调中 user_id 动态传递的可行性未经验证（严重）

- **问题描述**：第5节定义了 `/api/admin/execute` 的双认证模式，其中 Dify Agent 回调场景要求 Agent 在回调请求体中携带 `user_id` 参数。文档假设"Express 在代理转发用户消息至 Dify Agent 时，将 `user` 参数设置为当前用户的 user_id，Dify Agent 在 Text2SQL 工具调用时将该 user_id 填入回调请求体"。这一假设依赖 Dify 平台的两项能力：
  1. Dify Agent 的 Text2SQL 工具支持在回调请求体中引用对话上下文的 `user` 参数（即支持变量模板替换）
  2. Dify 的 `/v1/chat-messages` API（或 Agent API）的 `user` 参数会被保留在 Agent 的工具调用上下文中，且可被工具配置引用
  
  以上两项能力是否为 Dify 平台的标准功能，文档未提供任何验证依据。若 Dify 不支持此能力，则整个 AI 助手的行级权限模型将失去执行基础——AI 助手场景下普通用户的 Text2SQL 操作将无 user_id 可用，行级权限约束无法实际执行。此问题在功能层面体现为安全性风险（越权访问），在开发层面体现为返工风险（架构师在设计阶段发现不可行后需重新设计认证桥接方案）。
- **所在位置**：第5节"Dify Agent 回调 `/api/admin/execute` 时的用户身份传递机制"段落
- **严重程度**：严重
- **改进建议**：（1）在开发环境搭建阶段（项目阶段3任务1）中增加一项验证任务：在 Dify 平台上实测 Text2SQL 工具是否能引用 `user` 参数进行回调请求体变量替换。（2）若验证通过，在文档中标注"已在 Dify 平台 X.X 版本验证可行"。（3）若验证不通过，提供备选方案：Express 代理层在转发用户消息至 Dify Agent 时，不依赖 Dify 的 `user` 参数传递 user_id，改为在每次用户请求时由 Express 为 Dify Agent 的当前会话维护一个服务端 session→user_id 映射表，Agent 回调时通过 conversation_id 反查 user_id。

### 问题4：POST /api/admin/execute 在 Dify Agent 回调场景下的请求体格式未定义（一般）

- **问题描述**：第5节和第6.10节从概念层面描述了双认证模式，但 Dify Agent 回调场景下 `/api/admin/execute` 的请求体 JSON 结构从未给出。浏览器直连场景的请求体（用户 JWT + 自然语言指令或 SQL）同样未定义。实现者不知道：
  1. Dify Agent 回调时请求体应包含哪些字段（user_id、query_text、sql_statement？）
  2. 浏览器直连场景（管理员通过智能管理界面操作）的请求体字段是什么
  3. 两个场景的请求体结构是否相同（仅认证方式不同）
- **所在位置**：第6.10节 `/api/admin/execute` 端点，缺少请求体参数表
- **严重程度**：一般
- **改进建议**：为 `/api/admin/execute` 新增请求体参数表，明确列出两种场景共享的字段名称、类型、必填性。推荐结构：`{ "query": "自然语言描述或SQL语句", "mode": "natural_language" | "raw_sql" }`。Dify 回调场景额外携带 `user_id`（由 Agent 工具配置自动填充），浏览器直连场景由 JWT Token 提取 user_id。标注具体 SQL 执行的安全校验规则（如禁止 DROP TABLE、禁止访问 admin_logs 表以外的不相关表等）在概要设计阶段定义。

### 问题5：iframe 间 postMessage 通信协议未定义 —— 消息类型与 payload 格式完全空白（一般）

- **问题描述**：第1.1节描述了四种跨模块通信场景（登录态共享、FAB 渲染层级、AI 助手触发导航、iframe 间数据传递），均提到使用 `postMessage` 进行通信。但全文未定义任何一条具体的 postMessage 消息格式——未定义消息类型枚举、未定义 payload 字段结构、未定义消息来源校验方式（`event.origin` 白名单）、未定义错误响应格式。以"AI 助手触发跨模块操作"为例，文档描述"AI 助手通过 postMessage 向主框架发送导航指令（含目标 Tab 标识和可选参数）"，但消息格式如 `{ type: "navigate", target: "plan", params: {...} }` 或 `{ action: "switchTab", tabId: "life-plan" }` 完全由各模块实现者自行决定，将导致不同模块使用互不兼容的消息格式，集成阶段出现大量返工。
- **所在位置**：第1.1节 iframe 跨模块通信机制
- **严重程度**：一般
- **改进建议**：在第1.1节末尾新增"postMessage 消息协议规范"段落，至少定义以下消息类型的标准格式：
  - `AUTH_SYNC`：登录态同步（{ type, token?, action: "login"|"logout" }）
  - `NAVIGATE`：AI 助手触发导航（{ type, target: "home"|"consult"|"plan"|"news"|"profile", params? }）
  - `DATA_TRANSFER`：iframe 间数据传递（{ type, source, target, payload, requestId }）
  - 明确 `event.origin` 校验方式（同源策略下为 `window.location.origin`）
  
  标注完整的消息类型枚举留待概要设计阶段补充，但核心消息的方向（谁发谁收）和必含字段应在本需求文档中明确。

### 问题6：SSE 流内 Dify 错误事件的客户端处理策略未定义（一般）

- **问题描述**：第6.4节和第6.9节定义了 SSE 代理转发策略（Express 透传 Dify SSE 事件，前端 fetch + ReadableStream 消费），第7.3节定义了 AI 服务完全不可用和 SSE 连接中断的降级行为。但 Dify API 在实际运行中可能出现一种中间状态：HTTP 连接正常、SSE 流已建立，但流内返回了 Dify 的错误事件（如 `data: {"event": "error", "message": "..."}` ）。这种情况未被任何章节覆盖——第7.3节的"SSE 连接中断"处理策略仅针对网络层面的连接断开，不适用于流内逻辑错误。实现者不知道前端应如何处理 Dify 返回的错误事件：是丢弃并静默、展示错误提示但不中断流、还是停止消费流并提示重试。
- **所在位置**：第6.4节 SSE 代理转发策略（医师对话）、第6.9节 SSE 代理转发策略（AI 助手）、第7.3节健壮性（SSE 连接中断）
- **严重程度**：一般
- **改进建议**：在第6.4节或第7.3节补充 SSE 流内错误事件的处理策略：（1）前端在 ReadableStream 消费过程中识别 Dify 错误事件（按 event 字段区分），展示对应的错误提示文案（与 AI 服务不可用提示区分）；（2）错误事件之后的流数据是否继续消费取决于错误类型——临时性错误（如 token 限流）可等待后续事件，持久性错误（如工作流执行失败）应停止消费并提示重试。具体的错误事件类型和对应处理策略留待概要设计阶段补充。

### 问题7：前端 SPA 路由结构完全未定义（一般）

- **问题描述**：文档明确平台采用 SPA 架构，底部 Tab 栏切换各功能模块（第1节、第4.1节），但全文未定义任何 URL 路由结构。以下问题均无答案：
  1. 各模块/页面的 URL 路径是什么（如风险预测页是 `/#/risk`、`/#/profile/risk` 还是 `/#/risk?step=1`）
  2. 用户直接访问深层 URL 时的行为（是否先检查登录态再渲染，还是先渲染公开部分再按需跳转登录）
  3. 浏览器的前进/后退按钮行为（SPA 是否使用 hash/history 路由以支持浏览器导航）
  4. iframe 子页面的 URL 结构（各 iframe 的 src 是什么，是否与主框架 URL 联动）
  5. 从外部链接（如通知消息）直接进入特定功能页面的深度链接支持
  
  路由结构是前端开发的第一步。缺少路由定义意味着主框架开发者无法确定路由表，iframe 子页面开发者无法确定自己的页面 URL。各模块实现者可能使用互不兼容的路由方案。
- **所在位置**：缺失项，应位于第1节（项目定位与目标）或第4.1节（系统首页）中
- **严重程度**：一般
- **改进建议**：在第1节或4.1节新增"前端路由结构"段落，至少明确：（1）采用 hash 路由（`/#/`）还是 history 路由；（2）各模块的 URL 路径映射表；（3）深层链接的登录态检查行为（路由守卫逻辑）；（4）iframe 子页面的 URL 规范（如 `/pages/risk.html#step=2`）。完整的路由表可在概要设计阶段细化，但路由方案选型和关键页面的路径在本需求文档中应确定。

### 问题8：JWT Token 过期时的前端行为未定义（一般）

- **问题描述**：第4.10节定义了 JWT 过期时间为24小时，第4.3节定义了"未登录用户访问需认证页面自动跳转登录页"。但中间状态——用户在已登录状态下使用系统、Token 在操作过程中过期——的前端行为完全未定义。具体场景：
  1. 用户正在 AI 助手对话中输入消息，Token 过期，点击发送后的行为（静默跳转登录页导致消息丢失？弹窗提示重新登录？）
  2. 用户正在填写风险预测多步骤表单的第2步，Token 过期，点击"下一步"后的行为（表单数据是否保留？）
  3. iframe 子页面中的 Token 过期，与主框架的 Token 过期检测是否同步
  
  缺少此定义将导致各模块实现者自行决定 Token 过期行为，用户体验不一致。
- **所在位置**：第4.10节用户认证与会话管理；第4.3节访问控制矩阵后的路由守卫描述
- **严重程度**：一般
- **改进建议**：在第4.10节补充"Token 过期处理"段落：（1）前端在所有 API 请求中统一拦截 401 响应（而非提前判断 Token 过期时间）；（2）收到 401 时展示非阻断性提示（如顶部 toast "登录已过期，请重新登录"），保留当前页面状态（包括未提交的表单数据），引导用户跳转登录页；（3）登录成功后回跳至过期前的页面并恢复状态（通过 URL 参数或 sessionStorage 缓存页面状态）。

### 问题9：Express 端请求体字段名与 Dify API 参数名的映射关系未注明（一般）

- **问题描述**：第6节 API 接口规格中，多个端点的请求体字段名与底层 Dify API 的参数名使用了不同的命名。例如：
  - 第6.4节医师对话端点使用 `message`，但 Dify 聊天助手 API 的标准参数名为 `query`
  - 第6.9节 AI 助手端点使用 `message`，Dify Agent API 同样使用 `query`
  - 第6.11节 Dify 代理接口的通用参数表使用 `query`（与 Dify API 一致）
  
  Express 代理层必然存在一个参数名映射步骤（`message` → `query`），但全文未在任何位置注明这一映射的存在和映射规则。实现者在编写 Express 代理层代码时，若按第6.4节的参数名直接转发请求至 Dify API，将收到参数校验错误。此外，若 Dify 工作流的 inputs 变量名与 Express 请求体字段名不一致（如第6.3节风险预测端点使用 `systolic_bp`，但 Dify 工作流 inputs 中可能使用 `systolicPressure`），同样存在映射需求但未被文档覆盖。
- **所在位置**：第6.4节、第6.9节、第6.3节
- **严重程度**：一般
- **改进建议**：在第6节导言部分或 Express 代理层描述中新增"请求参数映射说明"段落，明确 Express 代理层负责以下映射：（1）请求体字段 `message` 映射为 Dify API 的 `query` 参数；（2）各工作流/Agent 的 Express 请求体字段名到 Dify inputs 变量名的映射规则（可列表说明，或标注"各端点映射规则在概要设计阶段定义"）。至少应让实现者意识到映射层的存在，避免直接透传导致接口错误。

### 问题10：AI 助手对话中的"健康建议生成"意图识别与触发机制不明确（轻微）

- **问题描述**：第4.8节描述 AI 助手可"获取 AI 生成的健康生活建议"，第6.11节 Dify 工作流清单标注 diabetes-assistant-agent 的工具调用可将健康建议写入 life_advice 表。但触发路径存在模糊之处：AI 助手是一个通用对话 Agent，它如何判断用户当前意图是"生成一条健康建议并持久化"还是"仅仅聊一聊健康话题"？这个判断是在 Dify Agent 的系统提示词层面完成的（Agent 自主决策何时调用"生成并存储建议"工具），还是在 Express 或前端层面有显式的触发机制（如用户点击"生成建议"按钮后前端发送特定指令）？文档对 AI 助手的能力描述使用的是"可以"（capability）而非"何时"（trigger condition），实现者无法确定前端是否需要提供显式的"生成建议"操作入口，还是完全依赖 Agent 自主判断。
- **所在位置**：第4.8节 AI 智能助手功能描述
- **严重程度**：轻微
- **改进建议**：在第4.8节或第6.11节 diabetes-assistant-agent 说明中补充触发机制：明确"生成健康建议"是完全由 Agent 基于对话上下文自主决策（前端无需特殊入口），还是前端在对话界面预留"生成建议"快捷指令按钮。推荐前者（Agent 自主决策），并在 Agent 系统提示词设计要点中标注"当用户表达'给我一些健康建议'或类似意图时，调用建议生成工具并将结果持久化至 life_advice 表"。

## 审查总结

本轮审查从实现者"能否直接开工编码"的视角出发，共识别 **10 个问题**（严重 3 个，一般 6 个，轻微 1 个）。三个严重问题均会导致特定模块无法进入开发阶段：管理员对话端点缺失（问题1）使智能管理模块不可开发；life_advice 写入机制空白（问题2）使 AI 健康建议功能不可开发；Dify Agent 回调 user_id 传递未验证（问题3）可能使整个 AI 助手的行级权限模型在开发阶段返工。建议优先解决三个严重问题后再进入概要设计阶段。
