# 需求澄清文档 —— 质量审查报告（v1）

## 审查概况

- **审查对象**：`a_v1_req_v2.md`（糖尿病预治智能助手 需求澄清文档 v2）
- **审查视角**：实现者视角 —— 需求是否可直接作为开发依据、验收标准是否明确可量化、是否存在隐含的模糊地带
- **审查范围**：侧重内部审议（设计-验证循环）未充分覆盖的维度 —— 需求响应充分度、数据模型一致性、逻辑完整性、可实施性
- **迭代轮次**：第 1 次质量审查

---

## 发现的质量问题

### 问题 1（严重 — 逻辑矛盾）：users 表缺少角色字段，无法支撑角色区分的功能描述

- **所在位置**：第 5 节（数据需求）users 表定义 vs 第 4.3 节（个人中心与登录注册）角色识别描述
- **问题描述**：第 5 节定义的 users 表结构为 `用户ID、用户名、密码、头像`，经核对项目知识库中的 `db.txt` 确认该表确实不含角色字段。但第 4.3 节明确要求"登录后系统需识别用户角色（普通用户 / 管理员），根据角色展示不同的可用功能入口"，管理员菜单项的显隐也依赖角色判定。数据模型与功能需求之间存在直接矛盾 —— 实现者无法在缺少 role 字段的情况下完成角色区分逻辑。
- **严重程度**：严重。属于阻断性矛盾，直接影响数据库设计阶段的建表 DDL 和后续所有权限相关功能的实现。
- **改进建议**：(1) 在 users 表中增加 `role` 字段（如 `TEXT CHECK (role IN ('user', 'admin'))`）；(2) 或者采用独立的用户-角色关联表；(3) 在数据需求一节同步更新表结构描述。

### 问题 2（严重 — 逻辑矛盾）：punch_in 表缺少与 life_plans 的关联字段

- **所在位置**：第 5 节（数据需求）punch_in 表定义 vs 第 4.5 节（生活方案）打卡机制描述、第 4.7 节（打卡记录与分析）
- **问题描述**：第 4.5 节描述打卡操作是"在方案展示页中，每条饮食/运动项旁提供打卡按钮"，明确将打卡绑定到具体方案项。但第 5 节 punch_in 表的字段为 `用户ID、打卡时间、打卡类型、完成状态、备注`，不含任何指向 life_plans 表的外键（如 plan_id）。经核对 `db.txt` 确认该表确实无此字段。这意味着系统无法记录"用户对哪个具体方案项打了卡"，打卡分析功能（4.7 节）中基于具体方案项的依从性分析将缺乏数据基础。
- **严重程度**：严重。数据完整性缺陷，影响打卡记录与分析功能的数据可靠性和分析准确性。
- **改进建议**：(1) 在 punch_in 表中增加 `plan_id` 字段并设置外键引用 life_plans(id)；(2) 或者在功能描述中明确打卡仅按类型（饮食/运动）统计，不关联具体方案项，但需要同步调整 4.5 节中"针对具体方案项"的描述以消除矛盾。

### 问题 3（高 — 关键遗漏）：缺少用户认证与会话管理机制

- **所在位置**：第 4.3 节（个人中心与登录注册）及全文
- **问题描述**：文档描述了登录注册功能、角色区分、管理员权限校验（6.3 节），但全文未涉及以下实现者必须明确的认证基础架构问题：(1) 登录后采用何种会话保持机制（Cookie/Session、JWT Token、LocalStorage + Token 等）？(2) 前端调用 Dify API 和 Express 接口时如何携带用户身份凭证？(3) Express 服务端如何校验请求合法性、如何获取当前用户身份？(4) 密码加密存储（6.3 节提及"不应明文存储"）采用何种具体算法（bcrypt、SHA-256 + salt 等）？前后端需一致。在 SPA + iframe 架构下，iframe 子页面如何共享登录态也是必须明确的交叉问题。
- **严重程度**：高。认证体系是 Web 应用的基础骨架，缺失将导致各模块开发时各自为政、后期集成困难。
- **改进建议**：在文档中增加认证与安全章节，至少明确：会话机制选型、Token 的生成/存储/传递/校验方式、前后端鉴权交互流程、密码加密算法选择。此项与问题 8（iframe 跨模块状态管理）紧密关联，建议一并处理。

### 问题 4（高 — 关键遗漏）：life_advice 表无对应功能模块

- **所在位置**：第 5 节（数据需求）life_advice 表定义 vs 第 3-4 节（功能模块）全文
- **问题描述**：第 5 节定义了 `life_advice` 表（建议ID、用户ID、标题、标签、内容），但遍历第 3 节的功能模块概览表和第 4 节全部 9 个功能模块的详细描述，均未提及该表对应的业务功能。参考项目示例代码中底部导航栏存在"生活建议"Tab（指向 `/lifeadvice/lifeadvice.html`），但产出文档的导航结构中已无此项。该表处于"有数据定义、无功能归属"的孤儿状态 —— 实现者不清楚是要建表但不使用，还是遗漏了一个功能模块。
- **严重程度**：高。造成数据库设计与功能范围的不一致，可能浪费开发工作量或遗漏功能。
- **改进建议**：(1) 如果"生活建议"是独立功能模块，应在第 3 节功能模块概览和第 4 节功能详细描述中补充对应章节，明确其与生活方案模块的关系；(2) 如果该功能已合并到生活方案模块中，应在数据需求一节注明 life_advice 表的使用上下文，或评估是否可移除该表。

### 问题 5（高 — 关键遗漏）：缺少 API 接口规格说明

- **所在位置**：全文
- **问题描述**：文档在多处提及 API 通信 —— 第 4.9 节出现 `POST http://{IP}:3000/execute/` 的具体端点，第 1 节说明前端通过 Dify 接口间接使用 DeepSeek，但全文未提供任何系统化的 API 接口规格。实现者缺少以下关键信息：(1) 完整的前端可调用端点列表（Express 端有多少接口？各自的 HTTP 方法和路径？）；(2) 各接口的请求 body 和响应 body 的 JSON Schema；(3) 错误码定义和异常响应格式；(4) Dify 前端 SDK 的封装方式（是直接在前端调用 Dify REST API 还是通过 Express 中转？）；(5) 请求鉴权方式（见问题 3）。在前后端分离的架构下，API 契约是并行开发的基础，缺失将导致前后端对接时大量返工。
- **严重程度**：高。API 接口规格是前后端并行开发的合同，缺失导致开发停滞或集成返工。
- **改进建议**：增加 API 接口规格章节，至少列出各功能模块对应的核心端点、请求方法和简要的请求/响应数据结构。可采用表格形式汇总，具体 JSON Schema 可在详细设计阶段补充。

### 问题 6（中 — 模糊地带）：doctor_information 表的 chat_token 字段含义未解释

- **所在位置**：第 5 节（数据需求）doctor_information 表定义
- **问题描述**：doctor_information 表包含 `对话 Token` 字段，但文档未说明该字段的具体含义、格式和用途。经查阅项目任务文档（任务 3-3）确认，该字段存储的是 Dify 聊天助手的 API Key（格式为 `Bearer app-XXXX`），前端通过该 Token 调用对应医生的 Dify 聊天接口。如果不加以说明，实现者看到"对话 Token"字段无法理解其实际用途和填充方式。
- **严重程度**：中。不阻碍理解整体设计，但增加实现阶段的沟通成本和试错成本。
- **改进建议**：在数据需求一节或医师咨询功能描述（4.2 节）中补充说明 chat_token 的用途（存储 Dify 聊天助手 API Key）和格式（`Bearer app-XXX`）。

### 问题 7（中 — 模糊地带）：life_plans 的排序字段与业务时段概念的映射不清

- **所在位置**：第 5 节（数据需求）life_plans 表定义 vs 第 4.5 节（生活方案）方案内容描述
- **问题描述**：life_plans 表使用 `排序`（order）字段控制方案项的显示顺序，但第 4.5 节的功能描述使用的是业务时间概念 —— 饮食方案按"早餐、午餐、晚餐、加餐"划分，运动方案按"晨间、晚间、周末"划分。数字排序如何映射到这些语义时段？例如 order=1 是否固定代表"早餐"？还是仅作为同类型方案项之间的相对排序？这个映射关系到前端如何将 API 返回的方案项归类展示到不同的时段卡片中。
- **严重程度**：中。不影响功能理解，但实现者需自行约定映射规则，可能导致展示层的不一致。
- **改进建议**：在 life_plans 表说明或生活方案功能描述中补充 order 字段与时段概念的映射约定。也可考虑在表中增加 `period`（时段）字段使语义更明确。

### 问题 8（中 — 架构深度不足）：iframe 架构下的跨模块状态管理未涉及

- **所在位置**：第 1 节（项目定位与目标）架构描述
- **问题描述**：文档第 1 节明确采用"SPA + 底部 Tab 栏 + iframe 子页面"架构，第 4.8 节设计了全局 FAB 悬浮按钮作为 AI 助手入口。但以下跨 iframe 的实现关键问题未被讨论：(1) 父页面（index.html）与各 iframe 子页面之间的用户登录态如何共享（localStorage 同源可用但需显式设计）？(2) FAB 悬浮按钮位于父页面，点击后如何在当前 iframe 之上弹起对话窗口？(3) AI 助手通过自然语言触发的操作（如"帮我预测一下糖尿病风险"）如何导航到对应 iframe 并传递参数？(4) 各 iframe 页面之间的数据依赖（如风险预测结果用于方案生成）通过何种机制传递 —— 是通过父页面中转还是直接共享存储？这些不是纯技术可行性问题，而是直接影响前端架构设计的决策点。
- **严重程度**：中。不影响功能需求的完整性，但架构设计中未覆盖这些跨模块机制会使前端架构师在概要设计阶段还需回头补做需求层面的决策。
- **改进建议**：在第 1 节或新增的架构概览章节中补充跨 iframe 通信机制的设计原则，至少明确：登录态共享方式、FAB 的渲染层级策略、iframe 间数据传递方式（postMessage / 共享 Store / URL 参数等）。

### 问题 9（中 — 验收不足）：缺少可量化的功能验收标准

- **所在位置**：第 3-4 节（功能模块描述）全文
- **问题描述**：文档对所有功能模块采用了"描述性"而非"规格化"的写法 —— 说明了每个功能做什么，但未定义什么算"做完"。从实现者视角，以下验收标准缺失：(1) 最少配置几位医生才算医师咨询模块就绪？(2) AI 生成方案/文章的篇数是否有最低要求？(3) 风险预测功能上线前是否需要一定样本量的准确率验证？准确率目标是多少？(4) 打卡分析的 AI 输出应包含哪些分析维度（周趋势、完成率、与方案的偏离度等）？(5) 第 6.2 节对 AI 操作标注了"合理预期而非硬性约束"，使性能验收不可执行。缺少量化标准意味着开发者不知道"做到什么程度可以提交测试"，测试者也不知道"什么情况算 bug"。
- **严重程度**：中。需求文档阶段的验收标准可适当宽松，但当前几乎全部使用自然语言描述而无一条可量化指标，会导致开发过程中范围蔓延和交付争议。
- **改进建议**：为每个功能模块至少定义 1-2 条可验证的完成标准。例如："系统首页至少展示 3 位医生和 3 篇科普文章"、"风险预测至少在 10 组测试数据上输出与评分规则一致的结果"、"AI 生成操作应在 30 秒内返回完整结果或明确报错"等。非功能性需求中的 AI 响应时间应改为可测试的硬性指标，或明确标注为"不做验收要求"。

### 问题 10（低 — 引用不完整）：危险因素缺失值估算的具体公式未提供

- **所在位置**：第 4.4 节（糖尿病风险预测）
- **问题描述**：文档描述"若用户未填写腰围或收缩压，系统基于 BMI 公式估算替代值（基于任务文档的补充说明）"，但未给出具体的估算公式。经查阅任务 5 文档，腰围估算涉及 `height * 0.47/0.45`（按性别）和 BMI 调整因子，收缩压估算涉及按 BMI 分档的查表逻辑。文档仅说"基于 BMI 公式"过于笼统，实现者需要自行查阅任务文档才能获得准确的估算逻辑。
- **严重程度**：低。公式可在任务文档中找到，不影响最终实现的正确性，但降低了本文档作为独立开发依据的完备性。
- **改进建议**：在 4.4 节或附录中给出估算公式的伪代码或数学表达式，使本文档可独立指导开发。

### 问题 11（低 — 细节缺失）：文章正文的存储格式未指定

- **所在位置**：第 5 节（数据需求）articles 表定义
- **问题描述**：articles 表的 `正文` 字段存储 AI 生成的长文内容，但未说明存储格式 —— 是纯文本、Markdown、还是 HTML？这直接影响：(1) Dify 工作流的 LLM 提示词中要求输出什么格式；(2) 前端文章详情页的渲染方式（直接 innerHTML 还是需要 Markdown 解析）。
- **严重程度**：低。可通过开发过程中的约定解决，但提前明确可避免前后端返工。
- **改进建议**：在 articles 表说明或健康资讯功能描述中补充正文内容的存储格式约定。

## 补充观察（非质量问题，供参考）

1. **导航结构与参考代码的差异**：产出文档的底部 Tab 为"首页/咨询/生活方案/资讯/我的"，而项目参考代码（任务 2-2 示例）使用"首页/方案定制/生活建议/AI助手/个人中心"。文档将其描述为"基于项目任务文档推断"，但实际上任务文档仅规定了 iframe 架构而未指定具体 Tab 名称。建议在澄清说明（第 8 节）中补充一条，说明本方案的 Tab 命名与参考示例的差异及设计理由，避免实现者参照示例代码时产生困惑。

2. **原需求"在线咨询"归属的调整**：原始需求中"在线咨询"被列在"平台首页"的子功能下，而产出文档将其提升为独立的底部 Tab（"咨询"）。这属于合理的功能重组，但应在澄清说明中注明此调整，以体现对原始需求的可追溯性。

3. **密码加密算法的具体选择**：6.3 节提出"密码不应明文存储"但未指定算法。虽然此细节可在详细设计中确定，但在需求阶段指定算法选型方向（如 bcrypt）有助于技术选型的一致性。

## 整体评价

产出文档在结构组织、需求覆盖度和 AI 能力架构描述方面质量良好，尤其是第 8 节（澄清说明）中对 v1 审查意见的逐条回应体现了迭代改进的规范性。功能模块的划分清晰，各模块的业务逻辑描述基本完整。然而，从实现者"拿到文档即可开工"的标准审视，文档在**数据模型一致性**（问题 1、2）和**基础架构机制**（问题 3、5、8）两个维度存在较显著的缺口 —— 前者包含逻辑矛盾会直接导致开发阻塞，后者作为骨架性设计缺失会导致各模块并行开发时架构不一致。建议在下一版修订中优先解决严重和高严重程度的问题，使文档达到可作为概要设计直接输入的质量水平。
