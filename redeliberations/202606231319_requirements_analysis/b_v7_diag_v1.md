# 质量审查诊断报告

## 审查概况

- **审查对象**：a_v7_copy_from_v6.md（糖尿病预治智能助手需求澄清文档 v7）
- **审查轮次**：第 7 轮
- **审查视角**：需求响应充分度、事实错误/逻辑矛盾、深度与完整性（避免重复内部审议已覆盖的技术可行性维度）
- **审查方法**：对照原始用户需求 (requirement.md) 逐项核验，从实现者"能否直接开工编码"视角评估

---

## 诊断发现

### 问题 1：Dify 部署模型假定为云端服务，但原始需求暗示自建部署，存在架构前提模糊

- **问题描述**：文档第 7.4 节明确将 Dify 归类为"外部云服务，不在本地服务器上部署"，并以此为前提设计网络拓扑（Express 通过外网 HTTP 调用 Dify API）。但原始需求 requirement.md 的阶段 3 任务 1 明确列出"Dify服务搭建"作为项目环境搭建任务——在中文技术语境中，"搭建"通常指自建/自托管服务实例（Dify 社区版可 Docker 部署），而非仅开通云账号。两种部署模型对应完全不同的 API 基础地址、网络策略、服务器资源需求和成本结构，且会影响 Dify 回调 Express（`/api/admin/execute`）时的网络可达性。
- **所在位置**：第 7.4 节末段（"Dify 和 DeepSeek：Dify 平台和 DeepSeek API 为外部云服务，不在本地服务器上部署"）vs requirement.md 阶段3任务1
- **严重程度**：一般
- **改进建议**：在第 7.4 节或环境配置章节补充 Dify 部署模型的二选一说明——方案A（云服务）：`DIFY_API_BASE_URL` 指向 `https://api.dify.ai/v1`；方案B（自建）：Dify 部署在数据服务器或独立服务器上，`DIFY_API_BASE_URL` 指向内网地址。明确 Dify 回调 Express 的网络可达性要求（自建模型下 Dify 需能访问 Express 的 `/api/admin/execute` 端点），并在 7.5 节环境变量表中标注 DIFY_API_BASE_URL 的实际取值取决于部署模型选型。

### 问题 2：前端代码组织结构与 Vitest 单元测试要求存在潜在矛盾

- **问题描述**：原始需求 requirement.md 的 A4 轨道要求前端使用 Vitest 进行单元测试，核心业务逻辑 100% 覆盖。但文档第 8.1 节推荐"采用原生 HTML/CSS/JavaScript 开发，不依赖前端构建工具"，源代码结构为传统的 `<script>` 标签加载模式（`/src/js/` 目录下的独立 JS 文件通过 `<script>` 引入 HTML）。Vitest 是 Node.js 环境下的测试框架，默认依赖 ES Module（`import`/`export`）或 CommonJS 模块系统——传统的全局脚本 `<script>` 加载方式无法被 Vitest 直接导入和测试。若不引入构建工具（如 Vite），实现者将面临"如何让原生 JS 文件同时支持浏览器 `<script>` 加载和 Vitest 模块导入"的技术困境，最终可能导致两部分工作割裂：前端代码无法被 A4 轨道有效测试，或被迫在 A4 开始前重构前端代码。
- **所在位置**：第 8.1 节（前端构建方式说明）vs requirement.md A4 轨道
- **严重程度**：一般
- **改进建议**：在第 8.1 节补充 Vitest 兼容性说明——推荐方案：将核心业务逻辑 JS 文件编写为 IIFE 格式（兼容浏览器全局加载）同时提供 named export（兼容 Vitest import），或直接引入 Vite 作为轻量构建工具以原生支持 ES Module 和 Vitest。至少在第 8.1 节中标注"前端 JS 文件的模块化方式需同时满足浏览器 `<script>` 加载和 Vitest 单元测试导入需求，具体方案在概要设计阶段确定"。

### 问题 3：Dify 工作流/Agent 的核心行为定义完全空白——实现者无法从文档出发配置 AI 能力

- **问题描述**：第 6.11 节列出了 7 个 Dify 工作流/Agent/聊天助手及其 inputs 变量，但均未提供系统提示词（system prompt）、挂载的知识库文档清单、工具定义（Agent 的 Text2SQL 工具之外还需要哪些工具）或输出格式约束。这 7 个工作流是整个平台 AI 能力的核心——风险预测的准确性、方案生成的个性化程度、AI 助手的工具调用质量都直接取决于 Dify 侧的配置质量。当前文档将所有提示工程推到"概要设计阶段"或"Dify Agent 配置阶段"，但实际上需求分析阶段就应明确每个 AI 功能的行为边界和知识来源，否则后续开发中可能出现"AI 能力与需求预期不匹配"的返工风险。例如：风险预测工作流是否需要在 Dify 知识库中上传《中国2型糖尿病防治指南》全文？AI 助手 Agent 的 Text2SQL 工具如何知道各数据表的 schema？
- **所在位置**：第 6.11 节（Dify 工作流清单汇总表）全文
- **严重程度**：一般
- **改进建议**：为每个 Dify 工作流/Agent 补充最小行为约束——至少包含：（1）知识库依赖声明（需要上传哪些文档到 Dify 知识库）；（2）系统提示词的核心行为指令（3-5 条关键约束，如"风险预测工作流必须基于中国2型糖尿病防治指南评分规则进行计算，不得编造评分标准"）；（3）Agent 的工具清单（挂载哪些 HTTP 工具，各自的用途）。若实在无法在需求阶段给出完整提示词，应至少为每个工作流写一段"行为规格说明"（用自然语言描述该工作流必须完成的处理逻辑、禁止的行为和必须引用的知识来源）。

### 问题 4：风险预测评分逻辑完全委托给 AI 模型，缺乏可验证性约束

- **问题描述**：第 4.4 节声称风险评分"基于《中国2型糖尿病防治指南（2020版）》评分体系（0-51分，≥25分为高风险）"，但紧接着说"风险评分计算和结果分析由 DeepSeek 模型根据评分规则和医学知识库完成推理"。这意味着评分逻辑的正确性完全依赖 DeepSeek 模型对医学指南的"理解"——模型可能编造评分规则、遗漏关键风险因素或使用过时的标准。文档未说明：（1）《中国2型糖尿病防治指南》的评分规则是否已作为知识库文档上传至 Dify；（2）如何验证 AI 输出的风险评分与指南标准一致；（3）如果模型输出的评分不在 0-51 范围内应如何处理。从实现者视角看，他们无法测试或验证风险预测功能的正确性——因为"正确"的标准（指南评分规则）不在文档中，也不在代码中，而在 AI 模型的推理中。
- **所在位置**：第 4.4 节（糖尿病风险预测）结果展示段落
- **严重程度**：一般
- **改进建议**：（1）至少在第 4.4 节或第 5 节的知识库依赖声明中，明确列出需要上传至 Dify 知识库的医学参考资料（含《中国2型糖尿病防治指南（2020版）》中与风险评估相关的章节）；（2）在验收标准中增加"系统应输出具体的风险评分数值（0-51 分）和对应的评分依据（如年龄得分、BMI 得分、家族史得分等），而非仅输出风险等级"；（3）在健壮性规范（7.3 节）中增加"若 AI 返回的风险评分不在 0-51 范围内，应视为异常并在前端展示通用错误提示，不得展示明显的异常数值"。

### 问题 5：项目阶段任务分解与需求文档之间缺少映射关系

- **问题描述**：原始需求 requirement.md 将主线 U+ 开发分为 4 个阶段（项目准备→分析设计→开发→部署测试），并在阶段 3 中列出 10 个具体的开发任务（任务1 环境搭建、任务2 系统首页、任务3 医师咨询……任务10 智能管理）。但需求澄清文档未将其功能模块映射到这些开发任务和阶段。实现者阅读文档后无法判断：哪些功能属于同一个任务应该并行开发？哪些功能之间存在前置依赖（如风险预测必须先于生活方案生成）？各模块的开发顺序是否受原始需求任务分解的约束？这导致文档虽然功能描述完整，但缺少"开发路线图"视角的指引。
- **所在位置**：缺失项——应位于第 8 节（范围与边界）或独立节
- **严重程度**：轻微
- **改进建议**：在第 8 节末尾或新增段落中补充"功能模块与开发任务的对应关系"表，至少将每个功能模块映射到原始需求中的对应开发任务（如"4.1 系统首页 → 阶段3任务2""4.2 医师咨询 → 阶段3任务3"等），并标注模块间的开发依赖关系（如"4.4 风险预测 → 4.5 生活方案"表示生活方案的生成依赖风险预测的输出数据）。

### 问题 6：localStorage/sessionStorage 键名未统一命名，存在键名冲突风险

- **问题描述**：文档在多个章节中定义了不同的浏览器存储用途，但未约定统一的键名命名规范：
  - 第 4.10 节：JWT Token 存储在 localStorage（键名未明确）
  - 第 4.2 节：对话会话 ID 按 `conversation_doctor_{id}` 存储
  - 第 4.8 节：AI 助手对话会话 ID 存储（键名未明确）
  - 第 4.4 节：风险预测表单数据存储在 sessionStorage 的 `risk_form_data` 键
  - 第 4.11 节：免责声明确认状态存储在 localStorage（键名未明确）
  - 第 1.1 节：iframe 间通过 postMessage 传递 Token 等数据
  多键并存但无统一前缀或命名空间约定，随着功能迭代增加新键时容易冲突，且在 Token 清除（登出）时需要清理的键列表不完整。
- **所在位置**：第 1.1 节、第 4.2 节、第 4.4 节、第 4.8 节、第 4.10 节、第 4.11 节
- **严重程度**：轻微
- **改进建议**：在第 1.1 节末尾或第 7.1 节设计系统中新增"浏览器存储键名规范"段落，约定：（1）统一前缀（如 `dpa_` 代表 diabetes-prevention-assistant）；（2）列出所有已使用的存储键清单及其用途、存储位置（localStorage/sessionStorage）、数据类型；（3）定义登出时的清除策略（哪些键在登出时清除，哪些保留）。

### 问题 7：数据库 schema 变更缺少迁移策略，迭代开发中表结构调整无指引

- **问题描述**：文档第 5 节定义了 11 张数据表的完整结构，但原始需求将开发分为 4 个阶段、10 个具体任务——各阶段开发不同功能模块时会逐步建表、改表。当前文档假设一次性建好所有表（通过数据库初始化 SQL 脚本），但实际迭代开发中（如先开发首页后开发打卡分析），表结构可能在后期任务中发现需要调整（增加字段、修改约束等）。文档未提供任何 schema 迁移策略（如 SQLite 的 ALTER TABLE 限制下的安全变更方式、迁移脚本的命名和组织规范），可能导致开发过程中数据库变更操作混乱或数据丢失。
- **所在位置**：第 5 节（数据需求）
- **严重程度**：轻微
- **改进建议**：在第 5 节末尾或第 8.1 节补充 schema 变更策略说明：（1）推荐使用版本化的迁移脚本（如 `migrations/001_initial.sql`、`migrations/002_add_xxx_field.sql`）；（2）标注 SQLite 的 ALTER TABLE 限制（不支持 DROP COLUMN、不支持修改列类型等），提醒开发者在设计初版表结构时尽量完备；（3）说明迁移脚本的执行时机（开发环境手动执行 vs 部署时自动执行）。

### 问题 8：图片/头像上传端点的存储目录创建未纳入环境搭建清单

- **问题描述**：第 6.14 节定义了 POST /api/upload/avatar 端点将头像保存至 `/static/uploads/avatars/` 目录，第 7.4 节 Nginx 配置中声明了该目录的访问路径。但文档未在任何位置说明该目录应在何时、由谁创建——Express 服务启动时自动创建？部署脚本手动创建？若目录不存在时发生上传请求，Express 将写入失败并返回 500 错误。此问题虽小但在首次部署时会导致头像上传功能不可用。
- **所在位置**：第 6.14 节（文件上传）vs 第 7.4 节（部署架构）
- **严重程度**：轻微
- **改进建议**：在第 6.14 节服务端处理流程中补充"Express 服务启动时检查并自动创建上传目录（若不存在）"；或在第 7.5 节环境配置中为 UPLOAD_DIR 配置项增加说明"该目录需在 Express 首次启动前创建，推荐在启动脚本中使用 mkdir -p 确保目录存在"。

## 整体评价

该文档经过 7 轮迭代修订后，功能描述完整、API 接口规格详细、数据模型清晰、验收标准可量化，在功能层面已基本满足"实现者可据此开工编码"的要求。第 1-6 轮内部审议（设计-验证循环）已有效覆盖了技术可行性、数据路径一致性、API 参数完整性、前后端鉴权模型等维度。

本轮审查发现的主要缺口集中在三个层面：（1）**Dify 工作流的 AI 行为定义完全空白**——这是整个平台 AI 能力的核心，但全部推迟到后续阶段，导致需求文档中最关键的"AI 能做什么、怎么做、基于什么知识做"仍然是不透明的黑箱；（2）**文档与并行任务轨道（A4 测试）之间的衔接缺失**——前端代码组织方式与 Vitest 测试要求之间存在潜在矛盾，若不在需求阶段澄清，可能在 A4 启动时被迫重构；（3）**部署模型假设与原始需求暗示之间的模糊**——Dify 是云还是自建直接影响网络拓扑和 API 配置。

以上问题不影响"开始编码"的可行性（实现者可以基于当前文档开始编写 HTML/CSS/Express 代码），但会在以下节点造成阻塞或返工：配置 Dify 工作流时（无法从文档获得提示词指引）、A4 单元测试轨道启动时（原生 JS 与 Vitest 的兼容问题暴露）、以及实际部署时（Dify 部署模型未定导致环境配置不确定）。
