## 需要解决的问题

- **问题描述**：`/api/admin/execute` 端点的权限模型与第5节双路径设计存在矛盾——第5节描述该端点可同时服务管理员和普通用户（经AI助手，通过行级权限约束），但第6.10节将其认证要求定义为"是 + admin"（要求 role=admin），两条规则互斥。该矛盾源于 v3_rev1 轮次对第5节数据路径进行双路径重构时未同步更新第6.10节认证列。
- **所在位置**：SRS 第5节"数据需求"中"AI驱动的Text2SQL路径"段落；第6.10节"管理相关"API表中 `POST /api/admin/execute` 的认证列
- **严重程度**：严重
- **改进建议**：方案A（推荐）：将 `/api/admin/execute` 的认证要求修改为"是"（JWT认证即可），在Express端点内部根据Token.role进行分级鉴权；或方案B：拆分为两个独立端点分别处理管理员和普通用户路径。同步更新第6.10节认证列和第4.10节管理员鉴权描述。

- **问题描述**：users 表头像字段、doctor_information 表头像字段、diabetes_types 表图片字段、articles 表封面字段均已定义，但全文未定义图片上传API端点、存储策略、静态资源访问路径或文件类型/大小限制，前后端接口契约在头像功能上完全空白。
- **所在位置**：SRS 第5节数据需求（各表图片字段定义）；第6.2节用户相关API；第6节全文无上传端点
- **严重程度**：一般
- **改进建议**：明确头像/图片的存储策略，在第6节新增文件上传子节（至少定义头像上传端点），在7.4节部署架构中补充静态资源的Nginx托管路径说明。

- **问题描述**：三个Express端点触发Dify工作流生成AI内容后需将结果持久化写入数据库（供后续GET端点返回），但"Dify响应解析→数据结构化→数据库INSERT"的关键流程从未被明确描述，属于未命名的第三条数据路径。
- **所在位置**：SRS 第5节数据路径定义；第6.3节（风险预测API）、第6.5节（生活方案API）、第6.7节（健康资讯API）
- **严重程度**：一般
- **改进建议**：在第5节增加第三条数据路径"AI内容生成持久化路径"，为每个涉及持久化的POST端点补充服务端处理流程简述，明确Dify工作流输出与数据库表字段的映射关系或标注留待概要设计阶段定义。

- **问题描述**：AI助手Agent和管理员Agent以及各Dify工作流的API Key存储位置完全未定义。医生chat_token已建立数据库存储先例，但Agent/工作流API Key属于系统级配置，无法沿用同一模式，文档未给出替代方案。
- **所在位置**：SRS 第5节数据需求；第6.9-6.11节各代理端点
- **严重程度**：一般
- **改进建议**：在第5节或第7.4节明确系统级API Key通过环境变量或.env文件管理，给出推荐命名规范，在部署架构的环境依赖中补充Dify API Key清单。
