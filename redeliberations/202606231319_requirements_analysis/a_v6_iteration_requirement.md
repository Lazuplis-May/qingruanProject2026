## 当前审查结果

本轮审查（第6轮）对 `a_v5_req_v2.md` 从实现者视角进行评估，共发现11个质量问题，质询确认结论为 LOCATED（全部成立）。以下按严重程度列出：

### 严重问题

1. **POST /api/risk/predict、POST /api/plan/generate、POST /api/articles/generate 三个核心AI端点的Express请求体参数完全未定义**
   - 严重程度：严重
   - 所在位置：第6.3节（风险预测API）、第6.5节（生活方案API）、第6.7节（健康资讯API）
   - 问题描述：三个POST端点是平台最核心的AI功能入口，触发Dify工作流生成内容后写入数据库。第6.11节列出了各工作流的 `inputs` 变量，但这些是Dify工作流层面的输入定义，Express端点的请求体JSON字段名、类型、是否必填等规格从未给出。实现者无法确定前端应向这些端点发送什么结构的JSON数据。
   - 改进建议：为这三个POST端点各增加一份请求体参数表（格式参照6.4节表格），至少列出Express端接收的JSON字段名、类型、是否必填。对于复杂的嵌套对象（如health_info），展开列出其内部字段。若部分字段的确需要留待概要设计阶段细化，则至少给出顶层字段清单并标注"内部子字段待概要设计阶段补充"。

2. **第6.4节推荐EventSource API消费SSE流，但EventSource不支持自定义HTTP头，无法用于认证端点**
   - 严重程度：严重
   - 所在位置：第6.4节第397行（"前端使用 fetch + ReadableStream 或 EventSource API 消费流数据"），以及第6.9节第454行（"SSE 代理转发策略与 6.4 节医师对话接口一致"）
   - 问题描述：浏览器标准的EventSource API不支持自定义HTTP请求头（如 Authorization: Bearer <token>）。而 /api/chat/doctor/:id 和 /api/assistant/chat 两个SSE端点均要求JWT Token认证。若实现者选择EventSource方案，将无法携带Authorization头，导致请求被401拦截。该错误通过引用传播到了AI助手端点。
   - 改进建议：删除EventSource API选项，明确推荐仅使用 fetch + ReadableStream 方式消费SSE流。补充一句技术说明："因认证SSE端点需在请求头中携带JWT Token，而EventSource API不支持自定义请求头，故仅推荐使用fetch API实现流式消费。"

### 一般问题

3. **PUT /api/plan/adjust、PUT /api/user/profile、PUT /api/user/password 三个端点请求体完全未定义**
   - 严重程度：一般
   - 所在位置：第6.5节（PUT /api/plan/adjust）、第6.2节（PUT /api/user/profile、PUT /api/user/password）
   - 问题描述：三个PUT端点的请求体参数均未定义。PUT /api/plan/adjust 的"用户反馈"格式不明确（自然语言文字段还是结构化JSON）；PUT /api/user/profile 的可修改字段清单、类型、头像字段传递方式未说明；PUT /api/user/password 的请求体字段（old_password、new_password）未给出名称和类型。
   - 改进建议：为三个PUT端点各增加请求体参数表（格式参照6.4节），明确字段名、类型、必填性和约束规则。PUT /api/user/profile与POST /api/upload/avatar的协作流程应以简要文字说明。

4. **GET /api/risk/history 和 GET /api/articles/collections 缺少分页参数，与其他列表端点不一致**
   - 严重程度：一般
   - 所在位置：第6.3节（GET /api/risk/history）、第6.7节（GET /api/articles/collections）
   - 问题描述：第v5轮修订为5个列表端点统一补充分页参数，但遗漏了两个同样是列表类返回的端点：GET /api/risk/history（历史风险预测记录可能累积增长）和 GET /api/articles/collections（收藏文章数量可能很大）。
   - 改进建议：为这两个端点补充分页查询参数（page、pageSize），与第6.12节定义的分页响应格式保持一致。

5. **多医师场景下的会话ID（conversation_id）管理模型存在歧义**
   - 严重程度：一般
   - 所在位置：第4.2节"会话管理"段落（第84行）
   - 问题描述：文档描述 localStorage 中维护"当前活跃对话的 conversation_id"（单数），但平台至少配置3位不同科室的医生，每位医生拥有独立的Dify聊天助手和对话空间。存在两个歧义：(1) 切换医生后原对话上下文是否保留未明确；(2) localStorage的存储键命名未明确（全局单一键还是按医生ID区分）。
   - 改进建议：在第4.2节会话管理段落末尾补充分辨方案：建议localStorage按医生ID区分存储键（conversation_doctor_{id}），使得每位医生的对话上下文独立保持。或在文档中明确"每次进入医生对话默认创建新会话"是有意设计，并说明历史对话始终可通过会话列表恢复。

### 轻微问题

6. **POST /api/auth/login 请求体参数未定义**
   - 严重程度：轻微
   - 所在位置：第6.1节（认证相关API表）
   - 问题描述：POST /api/auth/login 是用户登录的核心端点，但请求体参数（如 username、password）从未显式列出。虽然从描述可推断需要用户名和密码，但作为API接口规格缺乏明确的请求体参数表。
   - 改进建议：为POST /api/auth/login和POST /api/auth/register增补请求体参数表（字段名、类型、必填性、说明），与文档中已有参数表的格式保持一致。

7. **POST /api/punch 请求体遗漏"备注"字段**
   - 严重程度：轻微
   - 所在位置：第6.6节（打卡相关API表）
   - 问题描述：POST /api/punch 的描述遗漏了备注字段，但第5节punch_in表的定义包含"备注"字段。实现者无法判断该字段是否需要在前端打卡界面提供输入入口、是否必填。
   - 改进建议：在POST /api/punch的描述中补充备注字段，标注为可选。

8. **会话列表端点措辞为"建议新增"，实现者无法确定是否为必需功能**
   - 严重程度：轻微
   - 所在位置：第4.2节第84行（"建议新增 GET /api/chat/doctor/:id/conversations 端点"）、第4.8节第203行（"建议新增 GET /api/assistant/conversations 端点"）
   - 问题描述：两个会话列表端点使用"建议新增"这一建议性措辞，但第6.4节和第6.9节的API表中已经列出了这两个端点。功能描述中的"建议"措辞与API表中的确定存在形成矛盾。
   - 改进建议：将第4.2节和第4.8节中的"建议新增"修改为"系统需提供"，与API表中的端点定义保持一致。

9. **SQLite单点故障与3服务器部署架构的可靠性矛盾**
   - 严重程度：轻微
   - 所在位置：第7.4节"部署架构"
   - 问题描述：部署架构使用3台服务器，但SQLite是文件型数据库仅存在于服务器1上。服务器2和3仅运行Nginx做反向代理和静态资源负载均衡。若服务器1宕机，3台服务器在实际数据可用性上等同于1台。文档在第7.3节可靠性中声称"打卡数据、用户信息等关键数据写入后不应丢失"，而单SQLite实例的故障风险与此承诺相矛盾。
   - 改进建议：在第7.4节末尾增加一条说明，明确指出SQLite为单实例文件数据库，不具备主从复制或自动故障转移能力。服务器1宕机时API服务和数据访问将全部中断。

10. **life_advice 缺少单个建议详情的GET端点，与articles模块的模式不一致**
    - 严重程度：轻微
    - 所在位置：第6.9节（AI智能助手相关API表）
    - 问题描述：健康资讯模块（articles）提供了完整的列表+详情端点对，但健康建议模块（life_advice）仅定义了GET /api/assistant/advice（列表），缺少GET /api/assistant/advice/:id（单个建议详情）端点。若前端"健康建议"列表页点击某条建议后需要查看完整内容，缺少对应API。
    - 改进建议：增补GET /api/assistant/advice/:id端点，或在第4.8节/第6.9节中说明"建议列表的每一项已包含完整内容字段，无需单独详情端点"，使设计意图明确。

11. **5.1节为空壳占位节，无实质内容**
    - 严重程度：轻微
    - 所在位置：第5.1节"新增 API 端点对应的数据操作"（第346-348行）
    - 问题描述：第5.1节内容仅为一句话指向第6节的索引语句，不包含任何数据操作描述、表结构补充或新增信息。作为独立小节的标题暗示有实质性内容，但实际上只是一个跳转指令。
    - 改进建议：删除5.1节编号，将内容合并到第5节导言段落的末尾（如"本章涉及的数据表对应的新增API端点的完整接口规格见第6节"），避免空壳小节。
