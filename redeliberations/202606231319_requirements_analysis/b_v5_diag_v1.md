## 问题清单

### 问题 1：POST /api/risk/predict、POST /api/plan/generate、POST /api/articles/generate 三个核心AI端点的Express请求体参数完全未定义

- **严重程度**: 严重
- **所在位置**: 第6.3节（风险预测API）、第6.5节（生活方案API）、第6.7节（健康资讯API）
- **问题描述**:
  这三个POST端点是平台最核心的AI功能入口，触发Dify工作流生成内容后写入数据库。第6.11节Dify工作流清单汇总表列出了各工作流的 `inputs` 变量（如风险预测工作流需要 age/gender/height/weight/waist/systolic_bp/family_history/pregnancy/diabetes_history），但这些是Dify工作流层面的输入定义，**Express端点的请求体JSON字段名、类型、是否必填等规格从未给出**。实现者拿到这份文档无法确定前端应向这些端点发送什么结构的JSON数据。例如：
  - POST /api/risk/predict 的请求体是 `{ "age": 35, "gender": "male", ... }` 还是其他结构？
  - POST /api/plan/generate 的 `health_info` 和 `preferences` 对象内部包含哪些具体字段？
  - POST /api/articles/generate 是在请求体中传 `category` 还是通过查询参数？

  对比来看，第6.4节（医师对话）和第6.9节（AI助手）的POST端点都有明确的请求体参数表（参数名、类型、必填性、说明），但这三个核心AI业务端点完全没有。6.11节末尾也注明"各工作流的 inputs 变量完整定义……在概要设计阶段补充"，等于将Express端点的契约推迟到了下一阶段。但实现者在此之前无法完成前端表单开发和后端接口编码。

- **改进建议**: 为这三个POST端点各增加一份请求体参数表（格式参照6.4节表格），至少列出Express端接收的JSON字段名、类型、是否必填。对于复杂的嵌套对象（如health_info），展开列出其内部字段。若部分字段的确需要留待概要设计阶段细化，则至少给出顶层字段清单并标注"内部子字段待概要设计阶段补充"。

### 问题 2：第6.4节推荐EventSource API消费SSE流，但EventSource不支持自定义HTTP头，无法用于认证端点

- **严重程度**: 严重
- **所在位置**: 第6.4节第397行（"前端使用 fetch + ReadableStream 或 EventSource API 消费流数据"），以及第6.9节第454行（"SSE 代理转发策略与 6.4 节医师对话接口一致"）
- **问题描述**:
  文档在描述前端如何消费SSE流时，将EventSource API列为可用选项之一。但这是一个**事实性技术错误**：浏览器标准的EventSource API不支持自定义HTTP请求头（如 `Authorization: Bearer <token>`）。而 `/api/chat/doctor/:id` 和 `/api/assistant/chat` 两个SSE端点均标注"认证：是"，即要求JWT Token认证。若实现者选择EventSource方案，将无法携带Authorization头，导致请求被401拦截。

  该错误在第6.9节通过"与6.4节一致"的引用被传播到了AI助手端点。这是一个会让实现者走入死胡同的技术陷阱。

- **改进建议**: 删除EventSource API选项，明确推荐仅使用 `fetch + ReadableStream` 方式消费SSE流。可补充一句技术说明："因认证SSE端点需在请求头中携带JWT Token，而EventSource API不支持自定义请求头，故仅推荐使用fetch API实现流式消费。"

### 问题 3：PUT /api/plan/adjust、PUT /api/user/profile、PUT /api/user/password 三个端点请求体完全未定义

- **严重程度**: 一般
- **所在位置**: 第6.5节（PUT /api/plan/adjust）、第6.2节（PUT /api/user/profile、PUT /api/user/password）
- **问题描述**:
  这三个PUT端点的请求体参数均未定义：
  - PUT /api/plan/adjust：仅描述为"根据用户反馈调整方案"，但"用户反馈"是什么格式？是自然语言文字段，还是结构化JSON（如指定哪几个方案项要修改、修改内容是什么）？
  - PUT /api/user/profile：描述为"修改当前用户个人信息（用户名、头像等）"，但可修改字段清单、每个字段的类型、头像字段是传文件还是传路径均未说明。且与POST /api/upload/avatar（第6.14节）的关系不清晰——修改头像是先调上传接口再调此接口，还是上传接口内部已处理？
  - PUT /api/user/password：描述为"修改密码（需提供旧密码验证）"，但请求体字段（old_password、new_password）未给出名称和类型，前后端校验规则也无出处。

  这些端点是用户基本操作的核心接口，实现者无法仅凭一句话描述编码。

- **改进建议**: 为三个PUT端点各增加请求体参数表（格式参照6.4节），明确字段名、类型、必填性和约束规则。PUT /api/user/profile与POST /api/upload/avatar的协作流程应以简要文字说明。

### 问题 7：多医师场景下的会话ID（conversation_id）管理模型存在歧义

- **严重程度**: 一般
- **所在位置**: 第4.2节"会话管理"段落（第84行）
- **问题描述**:
  文档描述："前端在 localStorage 中维护'当前活跃对话的 conversation_id'"（单数）。但平台至少配置3位不同科室的医生（验收标准4.2明确要求），每位医生在Dify上是独立的聊天助手应用，拥有独立的对话空间。当前模型存在两个歧义：

  1. **切换医生后原对话上下文是否保留？** localStorage仅存储一个conversation_id，用户与医生A对话后切换到医生B，医生B的新conversation_id会覆盖医生A的。当用户返回医生A时，若不传conversation_id则创建新会话（丢失上下文）；若用户希望继续之前的对话，只能手动从历史会话列表中恢复。这种"每次进入默认新会话"的行为是否为设计意图？

  2. **localStorage的存储键命名未明确。** 是全局单一键（如 `current_conversation_id`）还是按医生ID区分（如 `current_conversation_doctor_{id}`）？前者会导致覆盖问题，后者可自然支持多医生并行对话。文档未做任何说明。

  同样问题适用于AI助手（第4.8节），但AI助手只有一个Agent实例，所以单键方案可接受。多医生场景则需要明确。

- **改进建议**: 在第4.2节会话管理段落末尾补充分辨方案：建议localStorage按医生ID区分存储键（`conversation_doctor_{id}`），使得每位医生的对话上下文独立保持。或在文档中明确"每次进入医生对话默认创建新会话"是有意设计（如产品决策为简化交互），并说明历史对话始终可通过会话列表恢复。

### 问题 8：会话列表端点措辞为"建议新增"，实现者无法确定是否为必需功能

- **严重程度**: 轻微
- **所在位置**: 第4.2节第84行（"建议新增 GET /api/chat/doctor/:id/conversations 端点"）、第4.8节第203行（"建议新增 GET /api/assistant/conversations 端点"）
- **问题描述**:
  两个会话列表端点均使用"建议新增"这一建议性措辞，而非确定性定义。但从功能完整性看：（1）如果没有会话列表端点，用户无法恢复历史对话，每次只能新建会话——这与第4.2节"支持完整的对话历史记录查看"的需求描述相矛盾；（2）第6.4节和第6.9节的API表中已经列出了这两个端点，说明它们已被纳入接口规格。功能描述中的"建议"措辞与API表中的确定存在形成矛盾。

- **改进建议**: 将第4.2节和第4.8节中的"建议新增"修改为"系统需提供"，与API表中的端点定义保持一致。

### 问题 9：SQLite单点故障与3服务器部署架构的可靠性矛盾

- **严重程度**: 轻微
- **所在位置**: 第7.4节"部署架构"
- **问题描述**:
  部署架构使用3台服务器，但SQLite是文件型数据库，仅存在于服务器1上。服务器2和服务器3仅运行Nginx做反向代理和静态资源负载均衡。如果服务器1宕机，Express服务和SQLite数据库同时不可用，服务器2和3只能返回502 Bad Gateway。3台服务器在实际数据可用性上等同于1台，两台运行服务器提供的冗余仅限于静态资源分发层面。

  对于学生实训项目，这并非严重缺陷，但文档在第7.3节可靠性中声称"打卡数据、用户信息等关键数据写入后不应丢失"，而单SQLite实例的故障风险与此承诺相矛盾。至少应标注此限制。

- **改进建议**: 在第7.4节末尾增加一条说明，明确指出SQLite为单实例文件数据库，不具备主从复制或自动故障转移能力。服务器1宕机时，API服务和数据访问将全部中断。如需提升数据层可靠性，可考虑定期备份SQLite文件或迁移至支持复制的数据库（超出当前项目范围，但应让读者知晓此限制）。
