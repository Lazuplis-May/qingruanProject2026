## 问题清单

### 问题1：`/api/admin/execute` 端点权限模型与第5节双路径设计矛盾

- **问题描述**：第5节（数据需求）将 `/api/admin/execute` 设计为同时服务管理员和普通用户（经AI助手）的Text2SQL执行入口，并明确说明普通用户操作时通过Token user_id进行行级权限约束。但第6.10节（管理相关API）将该端点的认证要求定义为"是 + admin"（即JWT校验通过且role字段必须为admin）。两条规则互斥——若执行第6.10节的鉴权逻辑，普通用户通过AI助手触发的所有Text2SQL操作（个人信息查询、打卡记录查询、健康建议生成等）将收到403 Forbidden响应；若按第5节设计放行普通用户，则第6.10节的"需role=admin"约束失效。

- **所在位置**：
  - 第5节"数据需求"中"AI驱动的Text2SQL路径"段落（第299行附近）："调用 Express 的 `/api/admin/execute` 端点执行……AI 助手在对话中为普通用户执行……操作时也可走此路径"
  - 第6.10节"管理相关"API表（第420行）：`POST /api/admin/execute` 的认证列标注为"是 + admin"

- **严重程度**：**严重**。该矛盾直接影响后端接口的鉴权中间件实现——实现者面临二选一的架构决策，无论选择哪一方都会导致另一节描述的功能失效。若选第6.10节方案（仅admin），则AI助手对普通用户的数据库操作能力全部失效；若选第5节方案（放宽为行级权限），则需重新定义端点鉴权策略并修改第6.10节。

- **改进建议**：
  1. 方案A（推荐）：将 `/api/admin/execute` 的认证要求修改为"是"（JWT认证即可，不强制role=admin），在Express端点内部根据 `Token.role` 进行分级鉴权——role=admin时可操作任意表/行，role=user时仅允许操作当前用户本人的数据（通过WHERE子句注入user_id约束或SQL解析校验）。同步更新第6.10节认证列和第4.10节管理员鉴权描述。
  2. 方案B：拆分为两个端点——`POST /api/admin/execute`（需role=admin，用于管理员智能管理）和 `POST /api/assistant/execute`（需登录即可，Express内部施加行级权限约束，用于AI助手代理普通用户操作）。同步更新第5节数据路径描述。
  3. 无论采用何种方案，需在第4.10节"管理员鉴权"段落中补充说明普通用户经AI助手进行Text2SQL操作时的权限边界和鉴权流程。

### 问题2：用户与医生的头像/图片字段缺少上传与管理机制

- **问题描述**：以下数据表均定义了图片相关字段，但全文未定义任何图片上传API端点、存储策略（本地文件系统/Base64/外部URL）、静态资源访问路径或文件类型/大小限制：
  - `users` 表含"头像"字段（第284行），第4.3节个人中心编辑功能含"修改头像"（第95行），第6.2节 `PUT /api/user/profile` 标注可修改"头像等"（第335行）
  - `doctor_information` 表含"头像"字段（第285行），初始化数据要求预置医生含头像（第305行）
  - `diabetes_types` 表含"图片"字段（第287行）
  - `articles` 表含"封面"字段（第286行）

  对于头像（用户/医生），前端实现者无法确定：应使用 `<input type="file">` 上传图片文件，还是提供URL文本输入框让用户粘贴图片链接。对于后端实现者，无法确定：是否需要实现multipart/form-data解析、文件存储路径、静态文件托管路由。这导致前后端在头像功能上的接口契约完全空白。

- **所在位置**：
  - 第5节数据需求（第284-287行）：各表图片字段定义
  - 第6.2节用户相关API（第335行）：PUT /api/user/profile 含头像修改但无上传端点
  - 第6节全文：无任何图片上传或静态资源服务端点

- **严重程度**：**一般**。缺失不影响核心AI功能的实现，但前后端实现者在开发个人中心和医生展示模块时会遇到明确的阻塞点。对于实训项目，头像可用默认占位图临时替代，但API接口的缺失意味着该功能在需求阶段即被搁置，后期补充可能牵动前后端接口契约变更。

- **改进建议**：
  1. 明确头像/图片的存储策略（推荐方案：用户头像通过 `POST /api/upload/avatar` 上传至服务器本地目录，返回访问URL存入对应表的头像字段；医生头像和糖尿病类型图片在初始化SQL脚本中以相对路径引用预置图片文件；文章封面由Dify工作流生成时以外部URL形式填充）。
  2. 在第6节新增"文件上传"子节，至少定义头像上传端点（方法、路径、请求Content-Type、响应格式、文件大小和类型限制）。
  3. 在7.4节部署架构中补充静态资源（上传图片）的Nginx托管路径说明。

### 问题3：Dify工作流输出到数据库的持久化写入流程为隐式约定

- **问题描述**：以下三个Express端点触发Dify工作流生成AI内容后，生成的内容需要写入数据库以便后续查询接口（GET类端点）返回，但Express端点内部的"Dify响应解析→数据结构化→数据库INSERT"这一关键流程从未被明确描述：
  - `POST /api/risk/predict`（第6.3节）：触发风险预测工作流，返回风险评分与建议。但 `GET /api/risk/history`（同节）需从 `user_risk_info` 表读取历史记录，意味着预测端点必须将每次预测结果写入 `user_risk_info`。写入时机（工作流返回后同步写入？）、写入字段映射（Dify输出JSON到表字段的对应关系）均未定义。
  - `POST /api/plan/generate`（第6.5节）：触发生成饮食/运动方案工作流。但 `GET /api/plan/current`（同节）需从 `life_plans` 表读取方案数据，意味着生成端点必须将Dify输出的结构化方案解析后逐条INSERT到 `life_plans` 表（含type、order、time、title、content字段）。Dify工作流输出格式与表字段的映射关系未定义。
  - `POST /api/articles/generate`（第6.7节）：触发文章生成工作流。但 `GET /api/articles` 和 `GET /api/articles/:id`（同节）需从 `articles` 表读取，意味着生成端点必须将文章写入数据库。写入字段包括标题、封面、正文（Markdown）、分类等，对应关系未定义。

  第5节数据路径仅区分了"常规CRUD"和"AI驱动的Text2SQL"两条路径，但上述三个端点不属于Text2SQL路径（它们调用的是Dify工作流API而非Text2SQL工具），也不在常规CRUD路径列举的范围内。它们属于未命名的第三条路径："Express触发Dify工作流→解析输出→写入数据库"，这条路径的写入环节是隐式的。

- **所在位置**：
  - 第5节数据需求（第297-301行）：两条数据路径的定义，未覆盖工作流输出的持久化路径
  - 第6.3节（风险预测API）、第6.5节（生活方案API）、第6.7节（健康资讯API）

- **严重程度**：**一般**。有经验的开发者可从"GET端点需要数据→POST端点必须写入"的逻辑推断出写入行为，但写入的触发时机、Dify输出格式与表结构的映射、写入失败的回滚策略均未定义，实现者需要自行设计这部分关键逻辑。

- **改进建议**：
  1. 在第5节数据路径说明中增加第三条路径："AI内容生成持久化路径"，描述Express端点触发Dify工作流后解析输出并写入数据库的标准流程。
  2. 为每个涉及持久化的POST端点补充"服务端处理流程"简述（1-2句话），至少说明：接收请求→调用Dify工作流→解析响应→写入对应数据表→返回结果给前端。
  3. 明确Dify工作流输出的JSON结构与数据库表字段的映射关系（可在Dify工作流清单表中增加"输出字段→数据库表字段映射"列），或标注该映射留待概要设计阶段定义。

### 问题4：AI助手Agent和管理员Agent的API Key存储位置未定义

- **问题描述**：文档对Dify API Key的管理采用了两种模式：（1）医生聊天助手的API Key（chat_token）存储在 `doctor_information` 表中，Express运行时从数据库读取后调用Dify API（第5节，第285行；第6.4节SSE代理策略），此模式完整且可操作；（2）AI助手Agent（`diabetes-assistant-agent`）和管理员Agent（`admin-manager-agent`）以及各工作流（`diabetes-risk-prediction`等）的API Key存储位置完全未提及。Express在代理 `/api/assistant/chat`（第6.9节）、`/api/admin/execute`（第6.10节）以及 `/api/dify/workflow/:workflow_id` 和 `/api/dify/agent/:agent_id`（第6.11节）时，必须持有对应Dify应用的API Secret才能调用Dify API。由于Agent和工作流的API Key不属于用户级数据（不像医生chat_token那样按医生区分），无法沿用数据库存储模式，但文档未给出替代方案。

- **所在位置**：
  - 第5节数据需求：仅定义了doctor_information表的chat_token字段（第285行）
  - 第6.9-6.11节：AI助手、管理、Dify代理端点均未说明API Key来源
  - 第6.11节Dify工作流清单（第444-458行）：提到了各工作流的标识和触发端点，但未提API Key管理

- **严重程度**：**一般**。后端实现者无法确定这些API Key应通过环境变量、配置文件还是数据库存储。考虑到医生chat_token已建立"数据库存储"的先例，其他API Key的缺失会引发实现者的困惑——是沿用数据库模式（建新表或字段）还是改用环境变量模式。

- **改进建议**：
  1. 在第5节或第7.4节（部署架构）中明确：系统级API Key（AI助手Agent、管理员Agent、各工作流）通过环境变量或 `.env` 配置文件管理，区别于存储在数据库中的用户级API Key（医生chat_token）。
  2. 给出环境变量的推荐命名规范，如 `DIFY_AGENT_ASSISTANT_KEY`、`DIFY_AGENT_ADMIN_KEY`、`DIFY_WORKFLOW_RISK_KEY` 等，或统一使用 `DIFY_API_BASE_URL` + 各工作流/Agent的 `app_id` 和 `api_secret` 映射配置。
  3. 在第7.4节部署架构的环境依赖中补充"需配置的Dify API Key清单"。

### 问题5：未登录用户点击FAB按钮后的交互行为未定义

- **问题描述**：第4.8节（第199行）描述AI智能助手FAB"位于主界面右下角，用户从任意页面均可一键唤起助手对话窗口"。第4.10节访问控制矩阵（第258行）将"AI智能助手（FAB入口可见）"标注为"公开访问"，但"AI智能助手（发送对话消息）"标注为"需登录"。这意味着FAB按钮对所有用户可见（包括未登录用户），但对话功能需要认证。然而，未登录用户点击FAB后应发生什么——是弹出对话窗口但显示登录提示、直接跳转到登录页、还是按钮视觉上可见但点击无响应——文档未在任何位置定义此交互行为。前端实现者需要自行决定此路径，可能导致与其他"需登录"页面的路由守卫行为不一致。

- **所在位置**：
  - 第4.8节AI智能助手（第199-206行）：FAB入口描述，但仅覆盖已登录用户的交互
  - 第4.10节访问控制矩阵（第258-259行）：标注了FAB可见性与对话权限的差异，但未定义未登录点击行为

- **严重程度**：**轻微**。不影响核心功能实现，但交互细节的缺失意味着不同前端开发者可能采用不同处理方式，影响用户体验一致性。

- **改进建议**：在第4.8节验收标准或第4.10节访问控制矩阵中补充一行说明，如："未登录用户点击FAB后弹出对话窗口，窗口内展示登录引导提示（含'去登录'按钮，点击跳转至登录页），不直接展示对话输入框。"

### 问题6：列表类GET端点缺少分页参数规范

- **问题描述**：以下返回列表数据的API端点均未定义分页参数（page、pageSize、limit、offset等），也未说明默认返回条数和最大条数限制：
  - `GET /api/articles`（第6.7节）：文章列表，支持分类筛选但无分页
  - `GET /api/doctors`（第6.4节）：医生列表
  - `GET /api/punch/list`（第6.6节）：打卡记录列表，支持日期和类型筛选但无分页
  - `GET /api/assistant/advice`（第6.9节）：健康建议历史列表
  - `GET /api/admin/logs`（第6.10节）：管理员操作日志列表

  对于打卡记录和操作日志，数据量随使用时间线性增长，缺少分页将导致单次请求返回全部数据，影响前端渲染性能和服务端查询效率。

- **所在位置**：
  - 第6.4节（医师咨询API）：GET /api/doctors
  - 第6.6节（打卡API）：GET /api/punch/list
  - 第6.7节（健康资讯API）：GET /api/articles
  - 第6.9节（AI助手API）：GET /api/assistant/advice
  - 第6.10节（管理API）：GET /api/admin/logs

- **严重程度**：**轻微**。对于实训项目的数据规模（单组使用、数据量有限），缺少分页不会造成严重的性能问题，但不利于培养规范化的API设计习惯，且后期若需扩展为多用户系统则成为明确的性能瓶颈。

- **改进建议**：为上述列表端点统一补充分页查询参数（推荐 `page` 和 `pageSize`，默认page=1、pageSize=20），并在响应格式中说明是否包含总条数字段（`total`）以支持前端分页组件。

### 问题7：缺少环境配置规范（环境变量/配置文件）

- **问题描述**：文档多处引用了需要在部署时配置的敏感信息和环境相关参数，但全文未定义统一的环境配置方式（如 `.env` 文件、配置模块或启动参数）。需要配置的项至少包括：
  - JWT签名密钥（第4.10节"服务端使用密钥签名"）
  - Dify API Base URL（第7.4节"Express通过HTTP请求调用Dify API"）
  - AI助手Agent和管理员Agent的Dify API Key（见问题4）
  - 各Dify工作流的API Key
  - SQLite数据库文件路径
  - Express服务端口（第6节"默认端口3000"）
  - Nginx静态文件目录路径（第7.4节）

  对于实训项目，缺少统一配置规范意味着每个开发者可能将API Key硬编码在源码中（造成安全风险和代码合并冲突），或在交接时遗漏关键配置项导致部署失败。

- **所在位置**：全文（配置项分散在4.10、6、7.4节）

- **严重程度**：**轻微**。实训场景下配置项数量有限，开发者可通过口口相传或README补充。但从需求文档完整性角度，应至少指出需要配置的关键项和推荐管理方式。

- **改进建议**：在第7.4节（部署架构）末尾或第8节（范围与边界）中新增"环境配置"子节，列出关键配置项清单（名称、用途、是否敏感），推荐使用 `.env` 文件管理敏感配置并通过 `dotenv` 加载，非敏感配置（如端口、文件路径）可提供默认值。
