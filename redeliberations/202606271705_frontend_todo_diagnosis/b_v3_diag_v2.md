## 2. 上一轮审查问题核验

上一轮审查（`b_v3_diag_v1.md`）识别的6个问题，经本轮逐条对照实际源代码（`useApi.ts`、`useHomeApi.ts`、`Punch.vue`、`LifePlan.vue`、`authStore.ts`）验证，**全部确认成立**。以下逐条列出验证结果。

### 问题1: G14 响应拦截器修复方案（方案B）与现有 getErrorMessage 函数不兼容

- **问题描述**: 诊断报告 G14 推荐方案B（`useApi.ts` 响应拦截器中 `Promise.reject(new Error(res.data.message || '请求失败'))`）产生的 Error 对象不含 `response` 属性。现有 `getErrorMessage()`（Punch.vue:64、LifePlan.vue:103）的提取逻辑首行检查 `'response' in err`，纯 Error 对象无法通过此检查，导致后端真实错误消息被 fallback 字符串吞没。

- **源码验证**: 
  - `useApi.ts:20` 响应拦截器 success 分支为 `(res) => res`，不检查 `success` 字段。
  - `Punch.vue:63-77` `getErrorMessage` — `'response' in err` → 提取 `axiosErr.response?.data?.error?.message` 或 `.data?.message` → 否则返回 fallback。
  - `new Error('...')` 不含 `response` 属性 → 直接走 fallback 分支。
  
- **所在位置**: `a_v3_diag_v1.md` 第476-491行（G14 修复建议方案B代码块）、第858-864行（8.3(c) 未识别此断裂点）

- **严重程度**: 中 — 10个 API 函数的错误提示路径受影响。

- **改进建议**: 
  (1) 在 G14 方案B中构造类 AxiosError：`const err = new Error(res.data.message || '请求失败'); (err as any).response = { data: res.data }; Promise.reject(err)`。
  (2) 或与 G8 合并修复——抽取 `getErrorMessage` 时增加纯 Error 的 message 提取分支。
  (3) 在 8.3(e) 交互风险表中增加 G14↔G8 条目。

### 问题2: S5a 隐藏后端 API 依赖，优先级表标注为"无前置依赖"不准确

- **问题描述**: 诊断报告 8.2 节优先级表将 S5a 标注为"无前置依赖"。但 `useHomeApi.ts` 当前仅含 `getDoctors`、`getArticles`（列表）、`getDiabetesTypes`、`getDiabetesType`（单数，用于糖尿病类型），不存在 `getArticle(id)` 单篇文章详情函数。

- **源码验证**: 已通读 `src/composables/useHomeApi.ts` 全文（73行），确认仅4个 API 函数，无 `getArticle`。

- **所在位置**: `a_v3_diag_v1.md` 第127行（S5a 修复建议）、第805行（优先级表）、第682行（技术可行性评估）

- **严重程度**: 中 — 影响优先级排序准确性和工时估算。

- **改进建议**: 
  (1) 8.2 节 S5a"前置依赖"列标注"待确认：后端 GET /api/articles/:id 接口是否存在"。
  (2) 第5节 S5a 复杂度评估从"低"修正为"低至中，取决于后端 API 就绪状态"。
  (3) S5a 修复建议中明确 `getArticle(id)` 的预期请求/响应格式。

### 问题3: S8 sessionStorage 迁移缺少跨标签页同步的具体方案

- **问题描述**: 诊断报告第216行仅以一句"如 BroadcastChannel"提及跨标签页同步，无具体实现方案。sessionStorage 迁移后每个新标签页将因缺失 token 而强制跳转登录页，形成显著 UX 退化。

- **源码验证**: `authStore.ts:12-13` — ref 初始化从 localStorage 读取 token/role，迁移至 sessionStorage 后新标签页初始化时 token 为 null，`isLoggedIn` computed 返回 false，路由守卫将重定向至 /login。

- **所在位置**: `a_v3_diag_v1.md` 第216行（S8 边界条件）

- **严重程度**: 中 — 影响修复决策的可执行性。

- **改进建议**: 
  (1) 在 S8 修复建议中增加"跨标签页同步（可选增强）"子节，给出 BroadcastChannel 最小实现方案。
  (2) 明确标注跨标签页同步为可选增强，核心修复可独立交付。
  (3) 在 8.3 节副作用评估中增加 S8 的跨标签页 UX 影响量化说明。

### 问题4: S1/S2/S8 修复建议中缺少统一的缓存清理函数实现

- **问题描述**: 8.3(e) 交互B识别了 S8↔S1/S2 的清理依赖，但各修复建议中均未落实。S8 的 `clearAuth()` 修改清单不含清理 `qrzl_home_cache`/`qrzl_plan_cache` 的操作。修复者需跨章节整合信息才能实现完整清理。

- **源码验证**: `authStore.ts:65-74` `clearAuth()` — 仅移除 `token/role/user/must_change_password` 四个键。迁移后若 S1/S2 已将缓存写入 sessionStorage，登出后旧用户数据残留。

- **所在位置**: `a_v3_diag_v1.md` 第200-212行（S8 函数修改清单）、第58-63行（S1）、第76-79行（S2）

- **严重程度**: 低 — 8.3节已从宏观识别，但各修复建议未同步更新。

- **改进建议**: 
  (1) 在 S1/S2 修复建议中各增加一条：暴露 `clearHomeCache()`/`clearPlanCache()` 函数。
  (2) 在 S8 的 `clearAuth()` 修改清单中增加调用上述清理函数（或直接执行 `sessionStorage.removeItem`）。

### 问题5: 8.3(e) 交互C（G7↔G14）的因果逻辑不准确

- **问题描述**: 8.3(e) 交互C声称"G14 修复后可能产生 null 输入"给 `renderMarkdown`。实际代码中，G14 导致 reject → Store catch 设置 `analysisError` → 模板 `v-else-if="store.analysisError"` 渲染错误降级 UI → `v-else-if="store.analysis"`（含 Markdown 渲染）不被进入。

- **源码验证**: `Punch.vue:172-190` 模板结构为 `v-if="store.analysisLoading"` → `v-else-if="store.analysisError"` → `v-else-if="store.analysis"`，三者互斥。G14 触发的错误进入第二个分支，Markdown 渲染分支（第三个分支）不会被渲染。

- **所在位置**: `a_v3_diag_v1.md` 第882行（交互风险表条目C）、第833行（脚注†C）

- **严重程度**: 轻微 — 空值防御建议本身无错，但因果逻辑链不成立，降低交互风险表作为集成测试指导的可信度。

- **改进建议**: 
  (1) 将条目C的"交互性质"改为"G7 抽取为通用工具函数后，未来可能被其他路径以 null 调用。建议在抽取时即增加空值防御，不特定关联 G14。"
  (2) 或将条目C从"修复间交互风险"重新分类。

### 问题6: G15 修复建议的提示文案与 S7 修复后行为存在语义矛盾
