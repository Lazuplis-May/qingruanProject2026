### 中严重度

2. **问题定级与原始 todo.md 存在多处差异但未给出解释** — 经质询报告核实，todo.md 中13个严重问题有10个被诊断报告重新定级，差异远不止最初列举的2例（S3和G5）。无声的重新定级导致修复者以哪份文档的定级为准来决定修复顺序的问题。改进：(a) 对与 todo.md 定级不同的条目，在诊断结论中显式说明重新定级的理由；(b) G5 建议将诊断结论调整为"不成立（设计文档应更新以匹配更优实现）"；(c) 在总体评估中增加一行对比说明诊断报告定级与原始定级的差异及原因。

4. **S8 诊断对 sessionStorage 短期缓解方案评估不完整，可能阻塞渐进式改进** — 质询报告发现诊断报告中列举的 sessionStorage 安全优势第3项"不会随每次 HTTP 请求自动发送"用于对比 localStorage 时是无效论据——localStorage 同样不具备自动 HTTP 发送能力，此属性是 Cookie 专有特征。该论据不影响核心判断但削弱了论据严谨性，建议删除该项或明确标注为"与 Cookie 对比的优势，但与 localStorage 相同"。改进：(a) 区分短期缓解（sessionStorage，即时生效、无后端依赖）和根治方案（HttpOnly Cookie，需后端协同）的有效性层级；(b) 明确说明 sessionStorage 虽不能根治但确实降低了攻击面（不跨标签页共享、标签页关闭后自动清除），建议作为过渡措施立即实施；(c) 补充 token 迁移到 sessionStorage 后 authStore 和路由守卫的联动修改点（authStore.ts 中读取 localStorage 的相关代码，router/index.ts 中 token 相关的路由守卫逻辑）。

5. **修复优先级排序未标注问题间的依赖关系和批处理可行性** — 第8.2节给出线性优先级排序但未标注：(a) S6 修复依赖 S5 先完成（路由不存在时修复跳转目标将直接触发404）；(b) S1 和 S2 同属 sessionStorage 缓存模式，可用同一模板实现，批处理可减少上下文切换；(c) G7、G8、G12 均属抽取公共工具函数，可在一次重构中统一处理；(d) S3 和 S7 均在 Punch.vue + punchStore.ts 中，修改区域重叠，分两次修改变更冲突风险高。改进：(a) 在优先级表中增加"前置依赖"列标注阻塞关系（S6→S5）；(b) 标注可批处理的问题组（S1+S2、G7+G8+G12、S3+S7）；(c) 在第7节根因分析与第8.2节优先级排序之间添加交叉引用映射关系。
