### 问题1: S5b 对 chatStore 和 SSE 机制的就绪状态未经实际代码验证（严重）

- **问题描述**: 诊断报告对 S5b（DoctorChatView.vue + /consultation/doctor/:id 路由）的技术可行性评估和复杂度判断基于未经验证的依赖假设。报告称 S5b "依赖 chatStore 和 SSE 机制就绪"，但从未实际读取 `src/stores/chatStore.ts` 以验证其状态。经实际代码审查发现，chatStore 是一个骨架实现——`conversations` 为空数组 `ref([])`，`abortActiveConnection()` 为空函数体，不含任何 SSE 连接、消息收发、对话管理的实现代码。`src/types/sse.ts` 中定义的 `SSEMessageEvent`、`SSEErrorEvent`、`ChatMessage` 等类型完全未被 chatStore 引用。设计文档 4.2 节（第3471行）明确要求 chatStore 管理"AI对话状态 (conversation_id, 消息列表, fabOpen, SSE连接状态)"，但实际实现与设计之间的差距远超诊断报告的认知。

- **所在位置**:
  - 第5节技术可行性评估 S5b 行：复杂度评估为"中高"——实际应为"高"或"极高"，因其隐含了对整个 chatStore SSE 系统的实现需求
  - S5b 修复建议（第140-145行）：仅提及"依赖 chatStore"而未说明 chatStore 实际需要从头实现
  - 8.2 节优先级表 S5b 行（第840行）：未标注 chatStore 本身需先完成的前置工作
  - 第7节根因分析"路由拆分未跟进"（第762-768行）：仅覆盖组件和路由缺失，未覆盖 chatStore 实现缺失这一更深层的系统性问题

- **严重程度**: 严重。此问题导致：(a) S5b 的工作量和复杂度被系统性低估——实际需要先实现完整的 chatStore（SSE 连接管理、消息收发、多医生会话路由），再实现 DoctorChatView.vue 组件；(b) 技术可行性评估"42项问题均无阻塞性技术障碍"对 S5b 而言过于乐观——chatStore 的 SSE 实现是整个项目中未完成的子系统，非简单的前端组件补充；(c) 优先级排序 P0 标签可能误导项目管理者在缺少 chatStore 基础的情况下启动 S5b。

- **改进建议**:
  1. 读取 `src/stores/chatStore.ts` 和 `src/types/sse.ts`，在 S5b 修复建议中明确 chatStore 当前状态为"骨架实现，需从头构建 SSE 通信层"
  2. 将 S5b 拆分为两个子任务：S5b-1（实现 chatStore SSE 核心——EventSource 连接、消息收发、conversation_id 管理、fabOpen 状态、多医生会话路由）和 S5b-2（实现 DoctorChatView.vue 组件 + 路由注册），并标注 S5b-1 是 S5b-2 的前置依赖
  3. 在技术可行性评估中将 S5b-1 的复杂度调至"高"（涉及 SSE 协议实现、断线重连、消息流式渲染），S5b-2 复杂度保持"中高"
  4. 在 8.2 节优先级表中将 S5b-1 标注为独立 P0 项，S5b-2 降为 P1（依赖 S5b-1）

### 问题4: S5a 缺少后端 API 不可用时的降级实现路径（中）

- **问题描述**: 诊断报告正确识别了 `useHomeApi.ts` 中不存在 `getArticle(id)` 函数，并标注为 [†D] 依赖。但修复建议仅停在"需确认后端是否已有接口"层面，未提供后端 API 不存在时的降级方案。如果后端 `GET /api/articles/:id` 接口尚未实现，修复者将面临阻塞。实际上存在可行的降级路径：使用现有 `getArticles()` 列表 API 获取全部文章后在客户端按 ID 筛选，或新增 `getArticle(id)` 函数并暂时 stub 后端响应。诊断报告应至少提供一个 fallback 方案，使前端修复不因后端就绪状态而完全阻塞。

- **所在位置**:
  - S5a 修复建议（第126-132行）：仅要求"确认"后端 API 是否存在，未提供后端不可用时的替代路径
  - 8.2 节脚注 [†D]（第869行）：仅描述依赖关系，未提供解除阻塞的替代方案

- **严重程度**: 中。在前后端分离开发场景中，前端修复不应完全阻塞于后端 API 就绪状态。缺少降级方案降低了修复的可操作性。

- **改进建议**:
  1. 在 S5a 修复建议中增加"后端 API 不可用时的降级方案"子项：若 `GET /api/articles/:id` 不存在，可先用 `getArticles()` 全量拉取后在客户端 `find(a => a.id === id)` 筛选，包装为 `getArticle(id)` 函数，待后端接口就绪后仅替换函数实现而无需改组件
  2. 明确 `getArticle(id)` 的预期返回类型（建议复用 `Article` 类型，或新建 `ArticleDetail` 扩展类型）

### 问题5: S8 BroadcastChannel 跨标签页同步代码示例存在模块级初始化副作用（低）

- **问题描述**: S8 的"跨标签页同步（可选增强）"代码示例在模块顶层（Store 定义作用域内、函数体外）执行 `new BroadcastChannel('qrzl_auth_sync')`。这在以下场景中存在风险：(a) 单元测试环境或 SSR 环境中 `BroadcastChannel` API 不存在——虽有 try-catch 静默降级，但在测试环境中意外的 DOM API 调用可能导致测试框架报错；(b) 如果在 Pinia Store 初始化完成前收到 `onmessage` 事件并调用 `syncFromStorage()`，此时 `token`/`role`/`user` ref 可能处于未完全初始化的中间状态。更稳健的做法是延迟初始化——仅在首次 `setAuth()` 或 `clearAuth()` 调用时创建 BroadcastChannel，或在 `onmessage` 处理中增加 Store 就绪状态检查。

- **所在位置**:
  - S8 跨标签页同步代码示例（第220-233行）：`new BroadcastChannel(...)` 在模块顶层立即执行

- **严重程度**: 低。此方案标注为"可选增强"，不影响核心修复。但若修复者直接复制代码示例，可能在生产环境的边缘场景中触发问题。

- **改进建议**:
  1. 将 `new BroadcastChannel(...)` 改为懒初始化：`let channel: BroadcastChannel | null = null`；在需要广播/接收时通过 getter 函数 `getChannel()` 按需创建（首次调用时初始化）
  2. 在 `onmessage` 处理器中增加 `if (!token || !role)` 等状态就绪检查，避免在 Store 未完全水合时执行 `syncFromStorage()`
