## 质询要点（CHALLENGED 时存在）

### 质询要点1

- **问题**：S5a 张力分析的结论"诊断无法从现有文档中确定答案"与设计文档实际内容不符。设计文档第6节（接口测试规范）第6224行明确记载 `GET /api/articles/:id 返回完整文章（含 content），views 计数+1`，这是可直接解答"文章详情页是否应含正文"问题的设计证据。
- **原因**：此遗漏影响修复者对文章详情页实现范围的判断。诊断报告给出的两种可能性（独立详情API含content vs 仅元数据展示页）中，方案(a)已被设计文档第6节确认——后端应返回完整文章内容、含 content 字段。诊断报告未引用该证据，导致修复者需要额外进行诊断报告已经可以完成的判断（与设计方/后端确认 API 行为），增加不必要的沟通成本。虽然不影响核心诊断结论（S5a 组件/路由缺失仍需修复），但降低了修复者直接使用报告的效率。
- **建议方向**：(1) 检索设计文档第6节（接口测试用例/冒烟测试清单）中与 `/api/articles/:id` 相关的 API 契约条目；(2) 将该条目作为确定证据更新张力分析的结论——明确方案(a)（独立详情API含content）为设计意图，修复时需新建 `ArticleDetail extends Article { content: string }` 类型；(3) 将设计文档第6节纳入诊断报告的引用范围，补充到 S5a 的设计依据中。

### 质询要点2

- **问题**：S5b-1 的前端 API composable 层创建需求未显式识别，与 S5a 的 API 依赖分析标准不一致。S5a 分析明确检查了 `useHomeApi.ts` 中是否存在 `getArticle(id)` 函数并标注 [†D]，但 S5b-1 分析未检查代码库中是否存在 chat 相关的 API composable（实际不存在）。
- **原因**：这不改变核心诊断结论（chatStore 需从头构建），但导致 S5b-1 的工作范围评估中缺少一个显式的 API 层前置子任务。修复者在实施 S5b-1 时需自行发现 `useChatApi.ts`（或等价的 chat API 函数）需从零创建——包括 `POST /api/chat/doctor/:id` 的请求构造逻辑（message body、conversation_id 参数、Authorization header）、响应流读取、与 chatStore 的接口对接。对标 S5a 的 [†D] 分析精度，S5b-1 应得到同等的 API 层依赖检查。
- **建议方向**：(1) 在 S5b-1 修复建议的"需修改文件"中增加 API composable 层的创建任务（如新建 `src/composables/useChatApi.ts` 或在 chatStore 中内联 fetch 调用——需明确技术选择）；(2) 在 8.2 节优先级表 S5b-1 行"前置依赖"列中标注 API composable 层创建需求；(3) 考虑将 chat API composable 的创建与 G14（响应拦截器 success 检查）建立交叉引用——G14 先于 S5b-1 完成可使新创建的 chat API 函数自动受益于响应拦截器的错误处理。

### 质询要点3

- **问题**：根因2"路由拆分未跟进"对 S5a 和 S5b 的完成度层级差异做了同质化概括。S5a 为3层缺失（组件文件+API函数+类型字段），S5b 为5层缺失（chatStore骨架+Consultation占位+组件文件+API函数+SSE通信层），两者在修复工作量和复杂度上存在数量级差异（4-8h vs 32-48h），但根因分析层面未做区分。
- **原因**：详细逐项诊断已充分区分了两者的差异，根因层面的同质化概括不影响诊断结论的准确性。但修复者在初读根因章节时可能形成"两个问题同等严重"的初步印象，需阅读详细诊断条目后才能准确理解差异。
- **建议方向**：(1) 在根因2描述中增加对 S5a 和 S5b 完成度层级差异的简要概括（如"S5b 的缺失程度远大于 S5a——涉及 Store 骨架、入口页占位、SSE 通信层三个额外缺失维度"）；(2) 在根因到问题映射表中为 S5a 和 S5b 增加复杂度差异的脚注标注。
