## 2. 质量问题逐项描述

### 问题1: G14 响应拦截器修复方案（方案B）与现有 getErrorMessage 函数不兼容

- **问题描述**: G14 的推荐修复方案（方案B，在 `useApi.ts` 响应拦截器中统一 reject `success:false` 响应）会导致后端错误消息在 UI 层丢失。

  方案B的实现代码为：
  ```
  Promise.reject(new Error(res.data.message || '请求失败'))
  ```
  但现有 `getErrorMessage()` 函数（LifePlan.vue:102-109 / Punch.vue:63-77）的错误消息提取逻辑检查的是 AxiosError 结构：
  ```
  if (err && typeof err === 'object' && 'response' in err) {
    const axiosErr = err as { response?: { data?: { error?: { message?: string }; message?: string } } }
    // ...
  }
  ```
  方案B产生的 `new Error(...)` 是纯 Error 对象，不含 `response` 属性，因此 `getErrorMessage` 会跳过整个 AxiosError 提取分支，直接返回 fallback 字符串（如 `'操作失败，请稍后重试'`）。后端返回的实际错误消息（如 `'业务限流'`、`'参数校验失败'`）将完全丢失。

  这导致了一个矛盾的结果：G14 修复本意是让后端错误消息可见（从静默空态转为显式错误提示），但方案B的实现方式反而使错误消息被屏蔽，用户看到的所有 `success:false` 错误都显示为相同的泛化文案。

- **所在位置**:
  - G14 修复建议 方案B 代码块，`a_v3_diag_v1.md` 第476-491行
  - 8.3(c) G14副作用分析，第858-864行（该节声称"UI 展示错误提示"但未识别此断裂点）
  - 8.3(e) 交互风险表缺少 G14↔G8 交互条目

- **严重程度**: 中 — 影响 10 个 API 函数的所有错误提示路径，导致后端错误消息系统性地不可见。修复者可行动但会产出有缺陷的结果。

- **改进建议**:
  (1) 在 G14 修复建议的方案B中，将 `new Error(res.data.message)` 替换为一个携带原始响应信息的错误对象。推荐两种方式：
    - **方式一（最小改动）**: 构造一个类 AxiosError 对象：`const err = new Error(res.data.message || '请求失败'); (err as any).response = { data: res.data }; Promise.reject(err)` — 使现有 `getErrorMessage` 的 `'response' in err` 检查通过，后续 `.data.message` 可正常提取。
    - **方式二（更规范）**: 与 G8（getErrorMessage 抽取）合并修复。先抽取统一的 `getErrorMessage`，在其中增加对纯 Error 的 fallback 处理：在 `'response' in err` 检查前增加 `if (err instanceof Error && err.message && err.message !== '请求失败') return err.message`，使不含 response 的 Error 对象也能展示其 message。
  (2) 在 8.3(e) 交互风险表中增加 G14↔G8 条目，标注"G14 改 Error 构造函数 → G8 getErrorMessage 需兼容新 Error 格式"。
  (3) 在 8.3(c) G14副作用分析中补充此前端集成断裂点，与"需与后端确认 API 契约"并列作为缓解措施前提。

### 问题2: S5a 隐藏后端 API 依赖，优先级表标注为"无前置依赖"不准确

- **问题描述**: 诊断报告 8.2 节优先级表将 S5a 标注为"无前置依赖"且复杂度为"低，可独立完成"。但 S5a 的修复建议正文（第127行）提到"需确认后端是否已有 `GET /api/articles/:id` 接口，若不存在需新增 `useHomeApi.ts` 中的 `getArticle(id)` 函数"。

  经核实 `src/composables/useHomeApi.ts` 当前代码：该文件仅包含 `getDoctors`、`getArticles`（列表，复数）、`getDiabetesTypes`、`getDiabetesType`（单数，但用于糖尿病类型而非文章）四个函数。**不存在 `getArticle(id)` 用于获取单篇文章详情的 API 函数。**

  这意味着 S5a 实际存在一个隐含的前置依赖：需要后端提供（或已存在但前端未接入）`GET /api/articles/:id` 接口。如果后端该接口不存在，S5a 的实现复杂度将从"低"上升为"中"（需协调后端新增 API）；如果后端接口存在但前端未接入，需先确认接口响应格式后才能写 `useHomeApi.ts` 中的对应函数。

- **所在位置**:
  - S5a 修复建议正文，`a_v3_diag_v1.md` 第127行
  - 8.2节优先级表，第805行（S5a 行标注"无前置依赖"）
  - 5节技术可行性评估，第682行（S5a 评估为"复杂度低，可独立完成"）

- **严重程度**: 中 — 影响优先级排序的准确性。如果后端 API 不存在，S5a 的实际工作量可能翻倍，P0 优先级的执行时间预估将失准。

- **改进建议**:
  (1) 在 8.2 节优先级表中 S5a 的"前置依赖"列标注"待确认：后端 `GET /api/articles/:id` 接口是否存在（若不存在需新增或与后端协调）"。
  (2) 在第5节技术可行性评估中，将 S5a 的复杂度描述从"复杂度低，可独立完成"修改为"复杂度低至中：前端组件和路由工作量为低；若需新增后端 API 接口，整体复杂度为中"。
  (3) 在 S5a 修复建议正文中，将 `getArticle(id)` 函数的接口契约（预期请求路径、响应格式）明确写出，以便后端开发者并行工作。

### 问题3: S8 sessionStorage 迁移缺少跨标签页同步的具体方案

- **问题描述**: S8 诊断建议将 token/role/user 从 localStorage 迁移到 sessionStorage 作为"短期缓解方案（建议立即实施）"。报告在第216行边界条件中提到了"跨标签页场景需重新登录或通过其他机制同步（如 BroadcastChannel）"，但仅此一行提及，未给出任何具体的 BroadcastChannel 实现方案。

  跨标签页打开是常见用户行为（如从首页 Ctrl+点击"生活方案"在新标签页打开）。实施 sessionStorage 迁移后，每个新标签页都会因缺少 token 而强制跳转登录页。这是从"安全风险存在但用户无感"到"每次新标签页都要重新登录"的显著 UX 退化。

  对于标记为"建议立即实施"的方案，缺少对此 UX 影响的量化说明和缓解路径，降低了修复决策的可操作性——修复者无法评估"立即实施"的代价是否可接受。

- **所在位置**: S8 修复建议边界条件，`a_v3_diag_v1.md` 第216行
- **严重程度**: 中 — 影响修复决策。修复者可能因不了解 UX 代价而言目实施，或相反因担心 UX 退化而搁置整个修复。

- **改进建议**:
  (1) 在 S8 修复建议中增加"跨标签页同步（可选增强）"子节，描述 BroadcastChannel 的最小实现方案：
    - 在 `authStore` 初始化时创建 `new BroadcastChannel('qrzl_auth_sync')`
    - `setToken`/`setAuth` 成功后通过 channel 广播 `{ type: 'auth_update', token, role, user }`
    - 其他标签页收到消息后调用 `setAuth()` 同步登录态
    - `clearAuth()` 时广播 `{ type: 'auth_clear' }`，其他标签页同步登出
    - 在 `onUnmounted`（或 app 级别）关闭 channel
  (2) 明确标注 BroadcastChannel 同步为"可选增强"，核心修复（token 迁移到 sessionStorage）可独立交付，跨标签页同步作为后续迭代的渐进增强。
  (3) 在副作用评估 8.3 节中增加 S8 的跨标签页 UX 影响说明。

### 问题4: S1/S2/S8 修复建议中缺少统一的缓存清理函数实现

- **问题描述**: 8.3(e) 交互风险表（条目B）正确识别了 S8（sessionStorage 迁移）与 S1/S2（sessionStorage 缓存）之间的清理依赖："S8 的 `clearAuth()` 需同步清理 S1/S2 的缓存键"。8.3(b) 也指出"token 失效重新登录后旧用户缓存仍残留在 sessionStorage 中"。

  但在 S1、S2、S8 各自的修复建议中，均未包含统一的缓存清理函数的具体实现。S8 的修复建议列举了所有需迁移的函数和键，但没有在 `clearAuth()` 的修改清单中包含清理 `qrzl_home_cache` 和 `qrzl_plan_cache` 的操作。S1/S2 的修复建议专注于缓存读写逻辑，未提及需要暴露一个清理入口供 `clearAuth()` 调用。

  这意味着修复者如果严格按照三个条目的修复建议依次实施，会在完成 S1+S2 和 S8 后留下一个残留缺陷：登出/401 后旧用户的缓存数据仍可被新登录用户读取（标签页未关闭场景）。

- **所在位置**:
  - S8 修复建议（函数修改清单表格，`a_v3_diag_v1.md` 第200-212行）中 `clearAuth()` 行未包含清理 home/plan 缓存键的操作
  - S1 修复建议（第58-63行）和 S2 修复建议（第76-79行）未提及需要暴露清理函数
  - 8.3(b) 副作用评估（第851-856行）和 8.3(e) 条目B（第881行）识别了问题但未下钻到各修复建议中

- **严重程度**: 低 — 8.3节已从宏观角度识别此问题，但各修复建议未同步更新。修复者需要跨章节整合信息才能实现完整修复。

- **改进建议**:
  (1) 在 S1 修复建议中增加一条："在 `homeStore` 中导出 `clearHomeCache()` 函数（执行 `sessionStorage.removeItem('qrzl_home_cache')`），供 `authStore.clearAuth()` 在登出/401时调用"。
  (2) 在 S2 修复建议中增加对应的 `clearPlanCache()` 函数。
  (3) 在 S8 修复建议的 `clearAuth()` 修改清单中增加："调用 `homeStore.clearHomeCache()` 和 `lifePlanStore.clearPlanCache()`（如果 S1/S2 已实现），清除旧用户的页面缓存数据"。
  (4) 或者统一方案：在 `clearAuth()` 中直接执行 `sessionStorage.removeItem('qrzl_home_cache')` 和 `sessionStorage.removeItem('qrzl_plan_cache')`，不依赖 Store 暴露函数（更简单，不引入跨 Store 依赖）。

### 问题5: 8.3(e) 交互C（G7↔G14）的因果逻辑不准确

- **问题描述**: 8.3(e) 交互风险表条目C描述 G7（Markdown渲染抽取）与 G14（success字段拦截器）的交互性质为："G14 修复后可能产生 null 输入，G7 抽取时需同步增加空值防御"。这段因果分析是不准确的。

  实际执行流程为：G14 拦截器 reject → Store catch 块设置 `error.value` → 组件的 viewMode 切换到错误态（如 Punch.vue:181 `v-else-if="store.analysisError"`）→ 渲染错误降级 UI。在错误态下，模板中的 `v-else-if="store.analysis"` 分支（含 Markdown 渲染）根本不会被进入。因此 G14 修复并不会"产生 null 输入"给 `renderMarkdown` ——错误态和数据态是互斥的模板分支。

  `renderMarkdown` 的空值防御仍然是有价值的防御性编程实践（防止未来代码变更引入 null 输入），但其与 G14 之间的因果关系在当前代码结构下不成立。将不存在因果关系的修复对标注在交互风险表中，可能导致修复者将精力投入到错误的集成测试场景上。

- **所在位置**: 8.3(e) 交互风险表条目C，`a_v3_diag_v1.md` 第882行

- **严重程度**: 轻微 — 不影响修复方案正确性（空值防御建议本身无错），但交互风险表的因果逻辑有误，降低了该表作为集成测试指导的可信度。

- **改进建议**:
  (1) 将条目C的"交互性质"从"G14 修复后可能产生 null 输入"修改为"G7 抽取的 `renderMarkdown()` 作为通用工具函数，未来可能被其他路径（含错误恢复路径）以 null/非字符串参数调用。建议在抽取时即增加空值防御，不特定关联 G14。"
  (2) 将条目C从"修复间交互风险"重新分类为"抽取通用函数时的防御性编程建议"，或在交互性质列中标注"因果关系弱，属预防性措施"。

### 问题6: G15 修复建议的提示文案与 S7 修复后行为存在语义矛盾

- **问题描述**: G15 诊断结论认为 loadMore 后 AI 分析不变量预期行为，修复建议为在分析区增加提示文案"分析基于所有打卡记录"。但 S7 修复后，日期筛选变更会触发 `fetchAnalysis()` 重新拉取（`setFilter` → `fetchList` + `fetchAnalysis`），此时分析数据是基于筛选后日期范围的，而非"所有打卡记录"。

  如果同时实施 S7 和 G15 的修复建议，分析区的提示"分析基于所有打卡记录"将与 S7 的实际行为（分析基于筛选日期范围）产生语义矛盾，用户可能在修改日期筛选后看到提示说"所有记录"但数据实际只覆盖了30天。

- **所在位置**: G15 修复建议，`a_v3_diag_v1.md` 第499行

- **严重程度**: 低 — 仅为提示文案的措辞问题，不影响功能正确性。但反映了修复建议之间的协调不够紧密。

- **改进建议**: 将提示文案从"分析基于所有打卡记录"修改为"分析基于当前筛选范围内的打卡记录"或"分析统计范围为上述日期的打卡数据"，使其与 S7 修复后的日期感知分析行为一致。

## 3. 审查总结

经审查，诊断报告 v3 在以下维度表现良好：
- **事实准确性**: 所有引用的代码位置、设计文档行号、接口签名均通过实际读取代码验证，无事实错误。
- **证据链完整性**: 42个诊断条目均有设计依据引用和代码证据，因果关系清晰。
- **结构完整性**: 覆盖了需求要求的全部诊断维度（一致性检查、技术可行性、逻辑完整性、根因分析、优先级排序）。
- **迭代响应质量**: 两轮修订系统性地解决了5+10个反馈问题，v3修订说明详实可追溯。

发现的6个质量问题集中于修复方案的可操作性维度：
- **中等（3个）**: G14拦截器与getErrorMessage不兼容、S5a隐藏后端API依赖、S8缺失跨标签页同步方案。这三个问题可能导致修复者在实施后产生新的缺陷或无法评估修复代价。
- **低（2个）**: S1/S2/S8修复建议缺少统一清理函数、G15提示文案与S7行为矛盾。这两个问题会增加修复者的信息整合成本或产生UI层面的次级偏差。
- **轻微（1个）**: 8.3(e)交互C因果逻辑不准确。不影响修复正确性但降低交互风险表的分析精度。

上述问题均可在诊断报告内通过局部修订解决（补充兼容性说明、调整依赖标注、增加实现细节），无需推翻整体结论。
