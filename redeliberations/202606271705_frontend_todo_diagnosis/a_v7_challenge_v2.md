## 质询要点（CHALLENGED 时存在）

- **问题**：S12诊断结论"两条打卡路径间的一致性由后端API契约保证"缺乏后端写入行为（同步/异步）的直接证据支撑，诊断自身标记该验证为"可选"但未在主要结论中反映此不确定性
- **原因**：S12的诊断结论"无需代码修改"取决于后端POST /api/punch为同步写入的前提。若后端为异步队列写入，两条打卡路径将出现时间窗口内的不一致，需前端引入状态直通或乐观更新机制。诊断未从设计文档、API文档或后端代码中提取关于写入语义的证据，使核心结论建立在一个未经验证的假设上
- **建议方向**：(a) 查阅设计文档中`POST /api/punch`的接口设计说明（3.2节）和测试规范（6节），寻找关于写入同步性的描述；(b) 如果设计文档未明确写入语义，应标记此问题为"待后端确认"而非"可选验证"，并在结论中反映条件性——"若后端为同步写入则无需代码修改，若为异步写入需补充前端状态直通"；(c) 考虑在修复建议中增加防御性措施（如Punch.vue的onMounted中增加短暂延迟后重试fetchList），作为对后端写入延迟的通用防护

- **问题**：S8"设计对齐"标签与立即代码修改建议之间存在语义张力，修复者可能困惑于变更顺序
- **原因**：诊断标签"设计对齐"传递的信号（问题在设计，代码正确）与修复建议"立即实施sessionStorage迁移"（代码修改）之间存在方向性矛盾。虽然报告在正文中说明了sessionStorage迁移作为"短期缓解方案"的合理性（不改设计、仅改善安全态势），但标签层面的不一致可能使修复者在规划工作时难以判断"先改设计还是先改代码"
- **建议方向**：(a) 将诊断标签从"设计对齐"调整为更精确的描述如"代码符合设计但设计存在安全缺陷——建议代码先行缓解后修订设计"，明确标注变更顺序；(b) 或在修复建议中明确步骤顺序：第一步实施sessionStorage迁移（代码先行，不影响设计合规性），第二步更新设计文档明确sessionStorage选择及其安全权衡

- **问题**：G14响应拦截器修复方案缺少分阶段部署的防护措施，一次性全局拦截可能误伤现有调用方依赖的"静默null"语义
- **原因**：诊断在8.3(c)副作用评估中识别了从"静默空态→显式错误"的风险变化，但未在修复建议层面提供渐进式部署方案。直接部署拦截器reject可能中断已依赖`null`空数据渲染逻辑的UI路径（如列表为空时依赖`v-if="data"`而非`v-if="error"`的判断链）
- **建议方向**：(a) 建议修复者使用分阶段部署——第一阶段在拦截器中添加`console.warn`收集所有`success:false`发生频率和场景，确认无误报后第二阶段切换为reject；(b) 在第一阶段日志收集中特别关注`GET /api/plan/current`（设计文档设计为"无方案时返回data=null + success=true"，需确认是否有异常场景返回success=false + data=null）；(c) 在8.3(c)副作用评估中补充此分阶段部署建议

- **问题**：G14修复方案B（响应拦截器统一处理）的新增Error对象构造方式（`err.response = { data: { message: ... } }`）与各Store中现有的catch块逻辑的兼容性未做逐Store审计
- **原因**：诊断推荐在`useApi.ts`响应拦截器中构造带`response`属性的Error对象以兼容`getErrorMessage`函数，但未验证三个Store（homeStore/lifePlanStore/punchStore）中各自的catch块是否均通过`getErrorMessage`消费错误对象、是否存在直接访问`err.response.data.message`的代码路径、是否存在不经过`getErrorMessage`的自定义错误处理逻辑。如果在某个Store中存在绕过`getErrorMessage`的错误处理分支，拦截器构造的Error对象格式可能不被正确消费
- **建议方向**：在G14修复建议中增加前置审计步骤——在三个Store的catch块中搜索所有错误处理路径，确认它们均通过`getErrorMessage`（或等价的兼容函数）提取错误消息，不存在硬编码的`err.response.data.message`访问
