### 问题1: 汇总统计表与 G5 详细诊断结论存在逻辑矛盾

- **问题描述**: 诊断报告第1节汇总表显示"不成立"条目为0（严重0 + 一般0 = 0），但 G5 详细诊断正文第339行明确写道"本问题不成立——代码实现优于设计文档规定。建议将诊断结论从'确认'调整为'不成立（设计文档应更新以匹配更优实现）'。" v2 修订说明第857行声称"G5 诊断结论调整为'不成立（设计文档应更新以匹配更优实现）'"，但实际只修改了正文内的"诊断结论调整"段落文案，未同步修改 G5 条目头部的诊断结论标签（第334行仍为"**确认**"），也未更新第1节汇总表。这导致汇总表（0个不成立）与详细诊断（1个不成立）之间存在数据矛盾。修复者若只看汇总表会认为所有42个问题均需代码修复，而实际上 G5 仅需设计文档更新。

- **所在位置**: 
  - 第1节汇总表（第15-20行）：`| 不成立 | 0 | 0 | 0 |`
  - G5 详细诊断（第334行）：头部仍标注"确认"
  - G5 详细诊断（第339行）：正文称"本问题不成立"
  - 修订说明（第857行）：声称已调整但未实际更新汇总表

- **严重程度**: 中 — 不影响问题诊断的正确性，但导致修复者对问题的整体分布产生偏差认知（以为无"不成立"条目），可能将"无需代码修改"的条目纳入修复计划。

- **改进建议**: (1) 将 G5 条目头部的诊断结论从"确认"改为"不成立（设计文档应更新以匹配更优实现）"；(2) 将第1节汇总表"不成立"行更新为 `| 不成立 | 0 | 1 | 1 |`（G5），并将"确认"行从37调整为36；(3) 在 G5 正文中删除"诊断结论调整"段落（该段落是修订过渡期的元注释，v2 正式发布后应直接呈现最终结论而非"建议调整"）。

### 问题2: S8 sessionStorage 迁移修复建议遗漏关键代码修改点

- **问题描述**: S8 修复建议要求将 `token`/`role`/`user` 从 localStorage 迁移到 sessionStorage，但仅显式列出了 `setToken`/`clearAuth`/`syncFromStorage` 三个函数。经代码核查，`authStore.ts` 中有 **19 处 localStorage 操作点**分布在以下位置，修复建议未覆盖的函数包括：

  (a) **`setAuth()` 函数（第35-42行）**: 写入 `token`/`role`/`user` 到 localStorage — 这是登录和 token 恢复的核心路径，若漏改将导致 token 同时写入 sessionStorage（由 `setToken` 写入）和 localStorage（由 `setAuth` 写入），产生双存储分裂；
  
  (b) **`fetchProfile()` 函数（第100-101行）**: 写入 `user`/`role` 到 localStorage — 用户刷新个人信息后，role 和 user 将回写到 localStorage，与已迁移到 sessionStorage 的 token 产生存储介质不一致；
  
  (c) **`setProfile()` 函数（第108行）**: 写入 `user` 到 localStorage — 同上；
  
  (d) **初始化 ref 声明（第12-24行）**: `token`/`role`/`user`/`mustChangePassword` 四个 ref 的初始值从 localStorage 读取 — 若漏改，组件初始化时 ref 为 null（sessionStorage 中无数据），即使 `syncFromStorage` 改为读 sessionStorage 也无法恢复（因为初始化时已设 null）；
  
  (e) **`login()` 函数（第83行）**: 写入 `must_change_password` 到 localStorage — 此项虽按规定保留在 localStorage，但修复者需知道此函数无需修改（修复建议已说明，但未与不改项做交叉确认）。

  当前修复建议虽在宏观意图上正确（"将 token/role/user 从 localStorage 迁移到 sessionStorage"），但在"可操作性"层面存在遗漏——开发者按照建议中显式列出的函数逐一修改后，`setAuth`/`fetchProfile`/`setProfile` 中的 localStorage 写入仍会执行，产生运行时状态分裂。

- **所在位置**: S8 修复建议（第195-200行），对照代码 `src/stores/authStore.ts` 第12-113行全部19处 localStorage 操作。

- **严重程度**: 中 — 按当前建议实现会产生 token 存储介质不一致的运行时缺陷（部分路径写 sessionStorage、部分路径写 localStorage），导致路由守卫在某些场景下无法读取 token。

- **改进建议**: (1) 在 S8 修复建议中补充完整的函数修改清单：`setToken()`、`setAuth()`、`syncFromStorage()`、`clearAuth()`、`fetchProfile()`、`setProfile()`、`login()`（仅 must_change_password 保留 localStorage），以及初始化 ref 声明（第12-24行）；(2) 用表格列出每个函数中需要修改的 localStorage 键和对应的 sessionStorage 替换，修改键和不修改键用不同标注区分；(3) 补充一行说明"建议在 authStore.ts 中全文搜索 `localStorage` 并逐处判断是否迁移，确保无遗漏"。

### 问题3: G14 严重程度说明行仍保留 v1 残留数字

- **问题描述**: G14 诊断条目严重程度行（第436行）写有"共涉及 3 个 API composable 文件约 7 个函数"，但下方详细证据列表（第437-448行）明确列出了10个函数（useHomeApi 4个 + useLifePlanApi 4个 + usePunchApi 2个 = 10个）。内部审议（`a_v2_challenge_v1.md` 第17行）已指出此不一致："G14诊断条目的严重程度说明行中，文字'共涉及 3 个 API composable 文件约 7 个函数'与下方详细证据列出的10个函数不一致……此数字差异不影响诊断结论"。该问题被定性为轻微，但 v2 修订并未修复。作为在严重程度描述行（读者最先阅读的高度概括位置）中的数字错误，会削弱报告可信度——读者看到"约7个"后向下翻到详细列表却发现是10个，会对其他数字的准确性产生疑虑。

- **所在位置**: G14 诊断条目严重程度行（第436行）："共涉及 3 个 API composable 文件约 7 个函数"

- **严重程度**: 低 — 不影响诊断结论的正确性（详细证据列表和修复建议均准确），但位于高度可见的位置降低了报告的数字可信度。

- **改进建议**: 将"约7个函数"改为"10个函数"，或删除数量描述（改为"共涉及 3 个 API composable 文件中全部 API 函数"），避免与详细证据列表产生数字矛盾。

### 问题4: 缺少修复间交互风险评估

- **问题描述**: 诊断报告 8.3 节对4个高风险修复点进行了独立的副作用评估（每个修复点单独分析其影响），但未评估修复之间的交互风险。以下修复对存在潜在的交互冲突：

  (a) **G14（success字段拦截器）+ S9（fetchAnalysis竞态保护）**: G14 在 axios 响应拦截器的成功分支中插入 `Promise.reject`，改变了所有 API 调用的错误传播路径。S9 在 `fetchAnalysis()` 中参考 `fetchList()` 的 `requestId` 快照模式增加竞态保护，但 `fetchList()` 的 catch 块通过 `snapshot !== requestId.value` 判断丢弃旧错误。G14 引入新的 reject 路径后，`fetchAnalysis()` 的 catch 块中 `analysisError.value` 的设置逻辑与 `requestId` 快照的交互需重新验证——旧响应的 success:false reject 被新请求的 requestId 快照过滤后是否仍会设置 analysisError；

  (b) **S8（sessionStorage迁移）+ S1/S2（sessionStorage缓存）**: 两者共享 sessionStorage 命名空间。S8 修复建议中提供两种缓存清理方案（`clearAuth` 中全量清理 / 缓存 key 加 token hash 前缀），但 S1/S2 修复建议未提及与 S8 的交互。如果修复者先实施 S1/S2（创建 `qrzl_home_cache` / `qrzl_plan_cache` 键），后实施 S8（在 `clearAuth` 中增加全量清理），需确保清理的键名列表与 S1/S2 中创建的键名一致；如果修复顺序相反，S1/S2 需感知 S8 已建立的清理机制；

  (c) **G7（Markdown渲染抽取）+ G14（success字段拦截器）**: G7 抽取 `renderMarkdown` 为公共函数，但如果 G14 在拦截器中增加 success:false reject，`createPunch` API（useLifePlanApi.ts 第61行）失败后的错误消息可能影响 Swal 弹窗的错误提示文案来源。虽然此交互风险极低，但 G7+G14 涉及其用的错误处理链（`getErrorMessage` 也建议合并抽取），建议交叉标注。

- **所在位置**: 报告未覆盖此维度。8.3节仅覆盖单点副作用。

- **严重程度**: 低 — 有经验的开发者通常能自行识别此类交互，但缺乏标注会增加修复顺序决策的时间和出错概率。

- **改进建议**: (1) 在 8.3 节末尾增加"修复交互风险"子节，列出上述3组交互及建议的修复顺序（如 G14 应在 S9 之前实施，S8 应在 S1/S2 之前实施以建立缓存清理机制）；(2) 在 8.2 节优先级表中对存在交互风险的修复对增加脚注标注（如 G14 行脚注 "与 S9 共享错误处理路径，建议先实施 G14"）。

### 问题5: 所有42个条目均呈现为"需要某种行动"，缺少"无代码修改需求"条目的显式标注

- **问题描述**: 诊断报告将全部42个条目归类为"确认"（37个）、"设计对齐"（4个）、"部分确认"（1个）。然而，其中有多个条目的实际诊断结论是不需要代码修改：

  - **G5**: 结论为"代码实现优于设计，仅需设计文档更新"
  - **G4**: 结论为"双模式并存可接受，建议加注释说明意图"
  - **S12**: 三元判断明确"不需要代码修改"
  - **S13**: 结论为"代码符合设计，内部矛盾在设计文档层面"

  这些条目与需要实质代码修改的条目（S1/S2/S5/S7/S9/G14等）在汇总表中处于同一分类下（"确认"或"设计对齐"），修复者无法从汇总表或优先级排序中快速区分"需要编写新代码"与"仅需确认/更新文档"。在42个条目的工作量面前，这种区分直接影响修复计划的制定效率。

- **所在位置**: 第1节汇总表（第15-20行）、第8.2节优先级排序表（第780-796行）。

- **严重程度**: 低 — 不影响诊断准确性，但降低了修复计划阶段的效率。

- **改进建议**: (1) 在第1节汇总表中增加一行"无需代码修改"维度统计（如 G4/G5/S12/S13 共4个条目）；(2) 在第8.2节优先级表中对无需代码修改的条目增加"行动类型"列标注（代码修复/设计文档更新/仅确认）；(3) 在 8.1 节影响面分级中，将"无需代码修改"条目从低危级中分离为独立行（或在低危级中标注）。

- **证据链稳固**: 经代码核查，关键诊断条目（S1/S2/S5/S7/S8/G14）的代码引用与实际情况一致。设计文档行号引用经抽查（第3466/3474/3482/3779行）均准确。
- **修复建议详实**: 每个条目包含修改文件、关键逻辑、边界条件和验证方法，修复者可以直接按照建议实施。

### 遗留问题

v2 诊断报告的主要遗留问题集中在 **修订完整度** 和 **修复间交互分析** 两个维度：

1. **G5 修订不完整**（问题1）: 修订说明声称调整但未同步更新汇总表，产生"声称 v2 已修复但实际残留"的矛盾——这是本次审查发现的最显著的逻辑不一致。
2. **S8 修复建议覆盖面不足**（问题2）: 遗漏 `setAuth`/`fetchProfile`/`setProfile` 和初始化 ref 的 localStorage 操作点，按当前建议实施会产生 token 存储分裂。
3. **G14 数字残留未清理**（问题3）: 内部审议已指出但未修复，虽轻微但在报告的高度可见位置。
4. **修复间交互风险未覆盖**（问题4）: 独立副作用评估完整，但修复对之间的交叉影响缺少分析。
5. **"无需修改"条目混编**（问题5）: 不直接影响修复质量，但增加修复者筛选和排期的认知成本。

### 审查结论

v2 诊断报告在修复建议、依赖标注、子任务分解、验证方法、副作用评估五个维度上相比 v1 有质的提升，v1 中"指导修复者行动"维度的系统性不足已基本解决。本次审查新发现的5个问题集中在修订完整度（G5汇总表未更新、G14数字残留）和实施致密性（S8遗漏修改点、修复间交互缺失）层面。建议优先修复问题1（G5汇总表矛盾）和问题2（S8修改清单补全），两者若按当前状态交付将导致实质性的实施错误或规划偏差。
