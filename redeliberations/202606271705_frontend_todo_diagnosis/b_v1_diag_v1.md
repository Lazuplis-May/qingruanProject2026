# 诊断报告质量审查 v1

> **审查对象**: `a_v1_diag_v1.md` (前端代码审查问题诊断报告 v1)
> **审查视角**: 需求响应充分度、整体深度与完整性、可操作性（修复者行动指引、副作用评估、优先级合理性）
> **审查日期**: 2026-06-27

## 审查发现

### 问题1: 缺失具体修复建议，直接违反需求明确要求

- **问题描述**: 诊断报告末尾声明"本报告定位问题根因至具体代码位置和触发条件，不包含修复步骤或代码补丁"，但 `requirement.md` 明确要求产出"问题清单（含严重程度、证据引用、修复建议）"。报告提供了问题描述、证据链和根因分析，唯独在每个诊断条目中缺失了"修复建议"字段。相比之下，原始 `todo.md` 每一条都带有"建议修复"段落。诊断报告仅告知修复者"改哪里"和"为什么改"，但未告知"怎么改"，导致修复者需要自行设计修复方案，增加了修复不一致和引入新问题的风险。
- **所在位置**: 报告第519行（末尾声明）及第2节、第3节全部42个诊断条目均无"修复建议"字段。
- **严重程度**: 高 — 直接违反需求规格，严重降低报告的可操作性。
- **改进建议**: 在每个诊断条目末尾追加"修复建议"字段，至少包含：(a) 建议的修改文件和函数，(b) 关键实现逻辑简述，(c) 需注意的边界条件。对于"设计对齐"类问题（S8/S13），修复建议可指向设计文档更新而非代码修改。

### 问题2: 问题定级与原始 todo.md 存在多处差异但未给出解释

- **问题描述**: 诊断报告对多个问题的严重程度重新定级，但与原始 `todo.md` 的定级不一致且未说明理由。例如：S3（默认近30天日期）在 todo.md 中标记为"严重"，诊断报告降级为"中"；G5（打卡弹窗顺序）todo.md 标记为"一般"且承认代码更合理，诊断报告分类为"确认"却未建议将其降级为"不成立"或"设计文档需更新"。这种无声的重新定级会导致修复者困惑：应以哪份文档的定级为准来决定修复顺序？
- **所在位置**: S3 诊断（第51-58行）、G5 诊断（第208-212行），以及其他存在定级差异的条目。
- **严重程度**: 中 — 不产生事实错误但造成优先级决策混乱。
- **改进建议**: (1) 对于与 todo.md 定级不同的条目，在诊断结论中显式说明重新定级的理由；(2) 对于 G5 这类"代码优于设计"的条目，建议将诊断结论调整为"不成立（设计文档应更新以匹配更优实现）"；(3) 在总体评估（第8节）中增加一行对比说明诊断报告定级与原始定级的差异及原因。

### 问题3: G14（API success 字段检查缺失）严重程度被系统性低估

- **问题描述**: G14 描述的是所有 API composable 函数（`useHomeApi.ts`、`useLifePlanApi.ts`、`usePunchApi.ts` 共7个函数）在解包 `res.data.data` 时未检查 `res.data.success` 字段。当后端返回 HTTP 200 但 `success: false` 时，所有数据管道将静默地将 `null` 作为正常数据向下游传递。经代码验证，axios 响应拦截器（`useApi.ts:19-41`）仅处理 HTTP 401 错误，不检查 `success` 字段。这意味着此问题的影响面覆盖全部7个 API 端点，下游 Store 和 View 组件处理 `null` 时可能产生运行时崩溃或白屏。诊断报告将其列为"一般问题"且放在优先级第4档，远低于其实际风险级别。
- **所在位置**: G14 诊断（第263-268行），优先级排序（第511行将 G14 放在第4档）。
- **严重程度**: 中 — 虽然实际触发取决于后端行为（后端是否可能返回 HTTP 200 + success:false），但作为系统性故障点，其潜在破坏力远超"一般"级别。
- **改进建议**: (1) 将 G14 严重程度上调至"中"或以上；(2) 在修复建议中明确需要修改的文件清单（至少3个 API composable 文件共约7个函数）；(3) 补充说明后端 API 契约中 success:false + HTTP 200 的发生场景（如业务限流、参数校验失败等），评估实际触发概率；(4) 考虑在 `useApi.ts` 的响应拦截器中统一处理 `success: false` 情况，减少各 API 函数的重复检查代码。

### 问题4: S8 诊断对 sessionStorage 短期缓解方案评估不完整，可能阻塞有意义的渐进式改进

- **问题描述**: S8 涉及 JWT Token 存储在 localStorage 的 XSS 窃取风险。诊断报告正确识别了这是设计层面决策的后果，但随后分析称"代码修复（如切换到 sessionStorage）只能降低非持久会话风险，无法根本解决 XSS 窃取问题（攻击者仍可在会话期间读取）"。此论断忽略了 sessionStorage 相比 localStorage 的三项安全优势：(a) 遵循同源策略且不跨标签页共享（降低攻击面），(b) 标签页关闭后自动清除（缩短凭据暴露窗口），(c) 不会随每次 HTTP 请求自动发送（与 Cookie 不同）。todo.md 给出的修复建议是两步走——短期切 sessionStorage + 长期改 HttpOnly Cookie，这是务实的渐进式策略。诊断报告将两者混为一谈，可能导致修复者认为短期措施无价值而选择"等待后端改造"的不作为状态。
- **所在位置**: S8 诊断（第109-119行），尤其是第119行"代码修复（如切换到 sessionStorage）只能降低非持久会话风险，无法根本解决 XSS 窃取问题"。
- **严重程度**: 中 — 诊断推理正确但结论引导方向可能有误，存在误导修复者放弃短期可操作改进的风险。
- **改进建议**: (1) 区分短期缓解（sessionStorage，即时生效、无后端依赖）和根治方案（HttpOnly Cookie，需后端协同）的有效性层级；(2) 明确说明 sessionStorage 虽不能根治但确实降低了攻击面，建议作为过渡措施立即实施；(3) 补充 token 迁移到 sessionStorage 后 authStore 和路由守卫的联动修改点（`authStore.ts:12,39`、`router/index.ts:79,119` 中读取 `localStorage` 的相关代码）。

### 问题5: 修复优先级排序未标注问题间的依赖关系和批处理可行性

- **问题描述**: 第8.2节给出了线性优先级排序（S5 > S1/S2/S3/S6/S7 > S4/S9/S11 > ...），但未标注以下关键关系：(a) S6（文章点击跳转 /news 而非 /news/article/:id）的修复依赖 S5（/news/article/:id 路由不存在）先完成——路由不存在时修复跳转目标将直接触发404；(b) S1 和 S2 同属 sessionStorage 缓存模式，可用同一模板实现，批处理可减少上下文切换；(c) G7（safeContentHtml）、G8（getErrorMessage）、G12（escapeHtml）均属"抽取公共工具函数"，可在一次重构中统一处理；(d) S3 和 S7 均在 `Punch.vue` + `punchStore.ts` 中，修改区域重叠，分两次修改变更冲突风险高。不标注这些关系的优先级排序会导致修复者按线性顺序工作，产生不必要的往返和冲突。
- **所在位置**: 第8.2节（第508-515行），第7节根因分析（第450-487行）。
- **严重程度**: 中 — 降低修复效率但不会导致修复错误。
- **改进建议**: (1) 在优先级表中增加"前置依赖"列标注阻塞关系（S6→S5）；(2) 标注可批处理的问题组（S1+S2、G7+G8+G12、S3+S7）；(3) 第7节根因分析已做到"按根因归组"，但与第8.2节优先级排序之间缺少交叉引用，建议添加映射关系。

### 问题6: S12 诊断缺少明确的"是否需要修复"的结论性判断

- **问题描述**: S12 涉及 LifePlan 内打卡状态（`completedMap`）与 Punch 页面打卡列表（`records`）的两套独立状态。诊断报告正确描述了架构（间接一致性模型，通过后端 API 串联），但结论为"功能正确性由后端 API 契约保证"，未明确回答"这到底是不是一个需要修复的问题"。todo.md 的修复建议是"确认后端 API 一致性；在文档中明确两条路径的数据一致保障机制"——这两个建议是明确的行动指令。诊断报告的描述性结论让修复者无法判断：(a) 是应该关闭此问题（设计如此），(b) 还是应该增加前端状态共享，(c) 还是仅需补充文档说明。从可操作性视角，这是诊断最模糊的条目之一。
- **所在位置**: S12 诊断（第152-162行）。
- **严重程度**: 低 — 不影响功能正确性，但影响修复决策效率。
- **改进建议**: 在 S12 诊断结论中明确给出三元判断：(a) 是否需要代码修改（不需要——当前实现符合设计的选择），(b) 是否需要设计文档补充（可选——在数据流文档中注明间接一致性模型和前端体验影响），(c) 是否需要验证（可选——确认后端 POST /api/punch 的写入在 GET /api/punch/list 的读取之前已生效）。

### 问题7: S5（最高优先级问题）缺乏子任务分解，修复者无从下手

- **问题描述**: S5 被列为最高优先级（阻塞级），但诊断报告仅指出两条路由和两个组件文件缺失，给出了中等复杂度的评估。经代码验证，`DoctorChatView.vue` 需要实现：SSE 流式对话（需集成 `chatStore`）、医生上下文加载（通过路由参数 `:id` 查询）、消息渲染（marked+DOMPurify）、对话历史管理、免责声明弹窗。`ArticleDetailView.vue` 需要实现：文章详情 API 调用、Markdown 正文渲染、封面/元信息展示、返回导航。两个组件的实现工作量和复杂度差异很大，但诊断将其归为同一问题，未做任何子任务分解。作为最高优先级的阻塞性问题，缺乏分解将导致修复者无法评估工作量、无法并行分配、无法追踪进度。
- **所在位置**: S5 诊断（第73-87行），技术可行性评估（第410行）。
- **严重程度**: 低 — 不改变问题定性但降低最高优先级任务的可执行性。
- **改进建议**: (1) 将 S5 拆分为 S5a（`ArticleDetailView.vue` + `/news/article/:id` 路由，复杂度低）和 S5b（`DoctorChatView.vue` + `/consultation/doctor/:id` 路由，复杂度中高）；(2) 分别列出两个组件的依赖项（需使用的 Store、API 函数、现有可复用模式）；(3) 标注 S5a 可独立完成，S5b 依赖 chatStore 和 SSE 机制的就绪状态。

### 问题8: 缺少修复后的验证方法建议

- **问题描述**: 诊断报告为每个问题提供了代码位置和触发条件的精确描述，但未提供修复后如何验证的建议。对于 sessionStorage 缓存（S1/S2），验证方法是检查浏览器 DevTools Application 面板的缓存键值和过期逻辑；对于路由缺失（S5），验证方法是直接访问 URL 并检查是否正常渲染而非404；对于 API 竞态（S9），验证方法是快速连续切换筛选条件并检查最终状态是否正确。缺少验证指引意味着修复者需要自行设计测试方案，增加修复不完全或引入退化的风险。
- **所在位置**: 全报告范围内，每个诊断条目均缺失。
- **严重程度**: 低 — 不直接影响修复质量但增加了验证遗漏的概率。
- **改进建议**: 对每个诊断条目的修复建议追加"验证方法"子项，1-2句话描述修复后如何确认问题已解决。对于高风险条目（S5/S8/G14）的验证方法应更详细。

## 整体质量评估

### 优势

- **证据链扎实**: 每条诊断均有设计文档行号引用和代码文件位置，经抽样验证（S1/S2/S5/S7/G14/S8），引用准确无误。
- **覆盖完整**: 42项 todo 问题全部得到逐项诊断，无遗漏项。
- **根因分析有深度**: 第7节识别了5个系统性根因，超越了逐项罗列问题的层面。
- **技术判断准确**: 经代码验证，涉及事实性判断的关键条目（S5路由缺失、S1/S2缓存遗漏、S7分析未联动、G14无success检查）均与代码实际情况一致，无事实错误。

### 核心缺陷

诊断报告最显著的不足在于**可操作性层面**：(1) 缺失修复建议，直接违反需求要求的"修复建议"字段；(2) 优先级排序未体现依赖关系和批处理可行性；(3) 最高优先级问题缺乏子任务分解；(4) 部分条目的结论性判断不够明确（如S12到底是需要修复还是设计如此）。

### 审查结论

报告在"问题识别"和"证据链构建"维度质量较高，但在"指导修复者行动"维度存在系统性不足。主要改进方向：为每个条目补充修复建议、标注优先级依赖关系、拆分高复杂度阻塞性问题为子任务、为关键条目补充验证方法。
