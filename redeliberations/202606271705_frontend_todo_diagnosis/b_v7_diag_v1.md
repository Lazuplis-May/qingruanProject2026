## 审查发现

### 问题1. G11 修复建议使用推测性行号定位，降低可操作性（一般）

- **问题描述**: G11（LifePlan.vue form 使用 reactive + null，空字符串可能漏过校验）的修复建议将目标函数定位为"`validateForm()` 函数（推测在 ~158-180行区域）"，使用"推测"措辞且范围跨23行。经实际代码读取，`validateForm()` 函数位于 `src/views/LifePlan.vue:157-163`，仅7行，包含4个校验条件。推测范围与实际位置存在偏差（起始行差1行，范围差3倍），且"推测"措辞向修复者传递"诊断者未确认代码位置"的信号，削弱信任度。
- **所在位置**: 诊断报告 G11 修复建议（"推测在 ~158-180行区域"）
- **严重程度**: 一般
- **改进建议**: (1) 将"推测在 ~158-180行区域"替换为精确行号"第157-163行"；(2) 修复建议中的代码片段与第158行 `form.age == null`、第160行 `form.height == null`、第160行 `form.weight == null` 逐行对应；(3) 对全部42个条目中涉及文件行号定位的修复建议，系统性地将"推测"/"约"/"~"等不确定性措辞替换为经实际代码读取确认的精确行号。若因文件版本不确定无法给出精确行号，应显式标注"行号基于当前快照，实施前请确认"而非使用"推测"。

### 问题2. G24 page-enter 动画全局提取方案未分析两页面动画效果差异导致的统一化副作用（一般）

- **问题描述**: G24 修复建议提出将 `page-enter` 动画定义从 Home/LifePlan 的 `<style scoped>` 中提取到全局样式文件。经实际代码读取，Home.vue 的 `page-enter` 动画为 `@keyframes pageEnter { opacity 0→1 + translateY(8px→0) }`（淡入+上滑），LifePlan.vue 的 `page-enter` 动画为 `@keyframes fadeIn { opacity 0→1 }`（纯淡入，无位移）。两者动画效果不同。若统一提取为全局样式，必须选择其中一种作为全局定义——这意味着另一页面的动画效果将被改变（要么 LifePlan 多了上滑位移，要么 Home 丢失了上滑位移）。诊断报告未识别这一差异，未给出两种动画效果的选择建议或保留组件级覆盖的方案。
- **所在位置**: 诊断报告 G24 修复建议；8.3(e) 交互E（全局样式提取）未提及动画差异
- **严重程度**: 一般
- **改进建议**: (1) 在 G24 修复建议中增加"动画效果差异分析"子项，列出 Home（fadeIn+translateY）和 LifePlan（fadeIn only）的当前动画差异；(2) 给出两种统一的选项：选项A——全局定义使用 fadeIn+translateY（Home 行为，LifePlan 行为变更），选项B——全局定义使用纯 fadeIn（LifePlan 行为，Home 行为变更），选项C——保留组件级覆盖（全局定义纯 fadeIn 为基础，Home 在 scoped 中额外覆盖 translateY）；(3) 在 8.3(e) 交互E 中补充动画效果差异的风险说明。

### 问题3. S8 sessionStorage 迁移的跨标签页 UX 退化在副作用评估中未量化影响面（一般）

- **问题描述**: S8 sessionStorage 迁移将导致每个新标签页（含 `Ctrl+Click` 链接、外部链接跳转、用户手动新标签页打开）因缺失 token 而强制跳转登录页。8.3(b) 副作用评估识别了"token 失效后缓存残留"风险并给出了缓解措施，但未评估 sessionStorage 迁移本身造成的正向 UX 退化。影响面包括：(a) 所有 `target="_blank"` 或 `Ctrl+Click` 打开的页面——用户在新标签页中看到登录页而非目标页面；(b) 用户从外部链接（如邮件通知）点击进入——总是需要登录；(c) 与 localStorage 时代相比，登录频率显著增加。BroadcastChannel 可选增强标注了解决方向但明确标记为"可选"，核心修复独立交付时这些 UX 退化将实际发生。当前报告未帮助修复者判断：该 UX 退化是否在可接受范围内、是否需要将 BC 增强从"可选"升级为"强建议"、是否有产品层面的权衡需要决策。
- **所在位置**: 诊断报告 8.3(b) 副作用评估、S8 修复建议"跨标签页同步（可选增强）"
- **严重程度**: 一般
- **改进建议**: (1) 在 8.3(b) 中增加"跨标签页 UX 退化量化"子项，列出受影响的用户场景（新标签页打开、外部链接进入、浏览器恢复会话）及影响频次估计；(2) 明确建议将 BroadcastChannel 增强从"可选增强"升级为"强建议——若无跨标签页同步，S8 迁移后的用户体验退化显著，建议在核心修复完成后的同一迭代内交付 BC 增强"；(3) 在 8.3(g) 跨标签页场景验证中标注"若未实现 BC 增强，标签页B跳转登录页为预期行为但应记录为用户体验退化项"。

### 问题4. G14 两阶段部署策略缺少日志收集期间受影响 UI 路径的影响面分析（一般）

- **问题描述**: G14 v12 新增的两阶段部署策略（第一阶段 `console.warn` 日志收集，第二阶段切换为 `Promise.reject`）是有效的风险缓解措施。但报告未分析第一阶段日志收集期间，当前已存在的"success:false 静默 null 传递"问题在哪些具体 UI 路径上持续发生。修复者处于两难：(a) 日志收集期间，现有的静默 null 问题仍在影响最终用户——报告应让修复者知晓这期间的已知风险面；(b) 特别关注的 `GET /api/plan/current` 被提及但未给出在该期间若发现误报（如"无方案时返回 success=false + data=null"）的处理预案。两阶段策略缺少"日志收集期间的风险告知"和"异常发现后的处置决策树"，削弱了修复者的行动指引完整性。
- **所在位置**: 诊断报告 G14 修复建议"分阶段部署建议（v12新增）"
- **严重程度**: 一般
- **改进建议**: (1) 在分阶段部署建议中增加"日志收集期间的已知风险面"子项——列出当前受静默 null 影响的 UI 路径（列表 blank 态、Store error 不更新、后端 message 不展示），说明日志收集期间这些问题仍然存在，团队需接受此期间的风险敞口；(2) 补充"日志收集后决策树"——若发现 `GET /api/plan/current` 返回 `success: false + data: null`（误报），应如何在拦截器中增加排除逻辑（如按 URL pattern 白名单），而非直接进入第二阶段全局 reject；(3) 标注日志收集期的建议时长——"一个迭代周期"过于模糊，建议给出具体天数或版本窗口（如"至少涵盖一个完整的测试周期，包含至少一次回归测试"）。

### 问题5. P4 优先级层内部15+条目缺少执行排序指引（轻微）

- **问题描述**: 诊断报告为 P2 层提供了内部执行排序（G14 最先，S4+S11 次之，S8 最后），但 P4 层包含 15+ 条目（G1/G2/G4/G5/G9-G11/G13/G15-G23/G26-G29/S12/S13），仅标注了 G24+G25 的批处理关系，未提供任何内部排序指引。P4 层内部存在隐式依赖：(a) G23（删除 api.ts 死代码）应在 G9（合并 DiabetesTypeView 接口）之后执行——G9 可能将接口移至 `types/api.ts`，若先删除死代码再移入接口需二次修改；(b) G16（marked.parse 异步模式标注）与 G7（Markdown 渲染抽取）相关——若已抽取 `useMarkdown.ts`，G16 的标注仅需修改一处而非两处；(c) G26（enumLabels 类型约束）与 G20/G21/G27（Store 接口重构）独立但建议在 Store 重构稳定后再修改 enumLabels（避免 Store 重构期间引入新的枚举使用点而未受类型约束保护）。不提供排序指引，修复者在面对15+条目时缺乏"从哪个开始"的指引。
- **所在位置**: 诊断报告 8.2 节优先级表 P4 行（第990-996行）、8.2.1 节未覆盖 P4 层排序
- **严重程度**: 轻微
- **改进建议**: (1) 在 8.2 节 P4 区域增加简短的内部排序建议——如"G7/G8/G12 先完成（工具抽取影响后续条目），G23 在 G9 之后执行，G26 建议在所有 Store 修改稳定后执行，其余条目无强依赖可按任意顺序"；(2) 或在 8.2.1 节"并行化策略"中增加 P4 层的简短排序段落。

### 问题6. 工时汇总段落未区分单人/多人场景，与并行化策略的数值可能造成混淆（轻微）

- **问题描述**: 8.2 节预估工时汇总描述了全部42项修复"总预估工时约 66-100h（8.3-12.5人天）"。8.2.1 节并行化策略给出了三人并行 41-64h 的对比数字。两个数字并列但没有在汇总段落中标注：66-100h 对应纯串行单人开发，41-64h 对应三人并行（含关键路径分析）。项目管理者若仅阅读工时汇总段落（第982行），可能以 66-100h 作为排期依据，忽略了并行后工期可缩短至 41-64h 的信息——该信息仅出现在 150 行之后的 8.2.1 节中，且汇总段落未做交叉引用。
- **所在位置**: 诊断报告 8.2 节预估工时汇总段落（第982行）
- **严重程度**: 轻微
- **改进建议**: 在工时汇总段落末尾增加一句交叉引用——"以上为纯串行单人估算。若采用多人并行策略（详见 8.2.1 节），三人并行后预估总工期可缩短至 41-64h（关键路径为 S5b-1 20-28h），工期缩短约 35-36%。"

## 审查总结

本次审查从**可操作性**和**副作用完备性**两个内部审议未充分覆盖的维度，识别了6个质量问题（4个一般 + 2个轻微）：

- **可操作性问题**（问题1、问题4、问题5、问题6）：G11 推测性行号降低了修复者的直接行动能力；G14 两阶段部署缺少日志收集期间的风险告知和处置决策树；P4 层15+条目缺少执行排序指引导致修复者缺乏起点参考；工时汇总段落未与并行策略建立交叉引用。
- **副作用完备性问题**（问题2、问题3）：G24 全局动画提取未识别 Home 和 LifePlan 的动画效果差异；S8 跨标签页 UX 退化在副作用评估中未被量化。

内部审议已覆盖的维度（技术可行性、证据充分性、标签一致性、优先级依赖、修复间交互风险、并行化策略、诊断范围声明等）经核查均已在 v12 中得到充分处理，本轮不再重复验证。
