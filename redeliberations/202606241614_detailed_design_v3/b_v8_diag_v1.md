# 第8轮详细设计质量诊断报告

## 诊断发现的问题

### 问题1：风险预测缺失值估算公式遗漏
- **问题描述**：`a_v8_tech_v2.md` 第 5.3.1 节提到了“缺失值估算节点：waist/systolic_bp为空则按公式估算”，但整个详细设计文档中均未提供具体的估算公式。这使得开发人员无法实现该节点逻辑，违反了设计文档的深度和完整性要求。
- **所在位置**：第 5.3.1 节（Dify工作流节点编排逻辑）
- **严重程度**：严重
- **改进建议**：在 5.3.1 节或相关补充说明段落完整补充需求文档（4.4节）中规定的腰围和收缩压估算公式（包含按性别区分的基准公式以及 BMI 调整因子等具体计算规则），以指导后续开发落地。

### 问题2：打卡分析工作流缺少解析失败兜底策略
- **问题描述**：需求 4.7 节要求打卡分析由模型生成。在详细设计的第 5.2.4 节中定义了 `punch-analysis` 工作流的输入变量和“输出结构定义”。但在早先迭代中，针对 Dify 模型可能输出不稳定结果的情况，系统已经为 `diabetes-risk-prediction` 和 `health-article-generator` 工作流补充了三层降级解析兜底策略。`punch-analysis` 同样输出结构化 JSON 并用于最终渲染，却遗漏了这一兜底处理机制，可能引发前端解析异常。
- **所在位置**：第 5.2.4 节（punch-analysis 工作流）
- **严重程度**：一般
- **改进建议**：在 5.2.4 节补充“输出格式与解析失败处理策略”段落，对齐其他工作流的三层降级解析框架，定义当 JSON 解析失败或输出非结构化文本时的默认返回结构及错误响应。

### 问题3：admin_logs 审计日志表存在被删除的风险与列名校验错误
- **问题描述**：根据需求 4.9 节，管理员操作日志必须“不可删除”。但在详细设计 7.3.4 节的 `validateRowLevelPermission` 表分类及规则中，将 `admin_logs` 错误归类为了“用户私有表（含 user_id 列）”，且依据该分类规则允许执行带有 `user_id = operatorId` 的 `UPDATE/DELETE` 操作。这导致两个致命缺陷：1) `admin_logs` 表 DDL（2.2节）使用的是 `operator_id` 而不是 `user_id`，导致校验逻辑列名错位抛错；2) 即使列名匹配，允许操作者修改或删除自己的操作审计记录严重违背了“不可删除”的安全底线。
- **所在位置**：第 7.3.4 节（SQL权限控制伪代码 - validateRowLevelPermission）
- **严重程度**：严重
- **改进建议**：在 7.3.4 节的表分类中为 `admin_logs` 单独建立规则，明确规定对 `admin_logs` **严禁执行任何 INSERT、UPDATE 和 DELETE 操作**（写操作由Express中间件内部自动完成）；若允许Agent查询日志，则 `SELECT` 语句中需强制校验 `operator_id = operatorId`。

## 诊断总结
本次审查重点关注了内部审议中易被忽略的系统级约束与业务完整性。发现的核心问题集中在业务公式漏记、部分工作流容错架构未统一对齐，以及涉及数据安全的审计日志权限拦截规则出现了严重的设计漏洞。建议在后续修订中重点修复这些漏洞以确保技术方案的严密性。
