## 质量诊断发现

### 问题 1（严重）：POST /api/articles/generate 分类阶段响应结构缺失

**问题描述**：第 6.7 节 POST /api/articles/generate 端点设计了两阶段工作流——不传 `category` 时进入"分类生成阶段"返回推荐分类标签列表，传入 `category` 时进入"文章生成阶段"返回完整文章。文档为文章生成阶段定义了正常响应结构（与文章详情一致，见第 963 行），但**分类阶段的响应结构从未定义**。实现者无法确定分类生成阶段的返回 JSON 格式——是 `["饮食指导", "运动指南", ...]` 简单字符串数组，还是 `[{ "name": "饮食指导", "description": "..." }]` 结构化对象数组，抑或其他格式。这是一个阻塞前端分类选择 UI 实现的缺口。

**所在位置**：第 6.7 节，第 919-925 行（请求参数表）及第 963 行（仅定义了文章阶段的响应结构）

**严重程度**：严重 —— 阻塞前后端并行开发，实现者无法确定分类阶段的前端渲染数据源格式。

**改进建议**：（1）在第 6.7 节补充分类生成阶段的正常响应 JSON 结构示例，明确定义分类标签列表的数据格式（字段名、类型、每个分类对象包含的属性）；（2）说明分类标签列表是否包含描述文本（如"适合您的饮食建议"等引导语）、是否按优先级排序、标签数量上限；（3）标注该响应结构为基于 Dify 工作流输出字段的推断，实际以工作流实际输出为准。

### 问题 2（严重）：PUT /api/plan/adjust 的 plan_id 参数类型与 POST /api/plan/generate 响应中的 plan_id 类型不一致

**问题描述**：第 6.5 节存在类型冲突——PUT /api/plan/adjust 请求体参数表中 `plan_id` 类型标注为 `string`（第 853 行："需调整的方案ID"），而 POST /api/plan/generate 正常响应 JSON 中 `plan_id` 的类型为 `number`（第 864 行：`"plan_id": 1`）。此外，文档定义 `plan_id` 为"方案组 ID"（同一批生成的所有方案项共享此 ID），但 PUT /api/plan/adjust 的描述仅笼统地说"需调整的方案"，未明确 `plan_id` 是指方案组 ID 还是单个方案项 ID。如果前端将 number 类型的 plan_id 作为参数传递给期望 string 的端点，可能因类型校验失败导致调整请求被拒绝。

**所在位置**：第 6.5 节，第 853 行（参数表，plan_id 类型为 string）与第 864 行（响应 JSON，plan_id 类型为 number）

**严重程度**：严重 —— 可能导致运行时类型校验错误，且 plan_id 语义不明确影响方案调整流程的正确实现。

**改进建议**：（1）统一 plan_id 的类型定义——推荐使用 `number`（与 life_plans 表的 INTEGER 主键类型一致）；（2）在 PUT /api/plan/adjust 参数说明中明确 `plan_id` 为"方案组 ID（即 POST /api/plan/generate 返回的 plan_id，代表同一批生成的全部方案项）"；（3）补充说明调整行为是"整个方案组的替换"（当前活跃方案组中的所有 is_active 项标记为 FALSE，新生成方案组标记为 TRUE）。

### 问题 4（一般）：Markdown 渲染安全性（XSS）完全未提及

**问题描述**：文档在 4.6 节（第 295 行）和 8.1 节（第 1310 行）明确前端将使用 `marked`（Markdown 解析器）将文章正文从 Markdown 转换为 HTML 进行渲染。AI 生成的文章正文由 DeepSeek 大模型产出，内容不可控——理论上可能包含恶意 `<script>` 标签、`onerror` 事件处理器等 XSS 攻击向量。然而全文未提及任何 HTML 净化（sanitization）步骤（如 DOMPurify、sanitize-html 等），也未在 7.3 节安全性条款中涉及前端渲染安全。

本平台的用户输入点极少（主要为表单字段），AI 生成内容是最主要的 XSS 攻击面。marked 库本身不执行 HTML 净化——其默认行为是将原始 HTML 标签原样输出。

**所在位置**：第 4.6 节第 295 行（Markdown 渲染描述）、第 8.1 节第 1310 行（marked 作为独立 npm 依赖引入）、第 7.3 节（安全性条款未涉及前端渲染安全）

**严重程度**：一般 —— 安全缺口，但可通过在详细设计或开发阶段补充 DOMPurify 等净化步骤来解决。

**改进建议**：（1）在 8.1 节 marked 引入描述中同步提及 HTML 净化库（推荐 DOMPurify），明确"marked 仅负责 Markdown→HTML 转换，HTML 净化由 DOMPurify 在渲染前处理"；（2）在 7.3 节安全性条款中增加一条："AI 生成内容的 HTML 渲染安全——前端在渲染 Markdown 转换后的 HTML 前，须经 HTML 净化器移除潜在恶意代码（如 script 标签、事件处理器等），防止 XSS 攻击"。

### 问题 6（一般）：life_plans 表 plan_type 的 'other' 枚举值无对应业务场景

**问题描述**：第 5 节 life_plans 表定义中 plan_type 字段的 CHECK 约束为 `CHECK(plan_type IN ('diet', 'exercise', 'other'))`（第 611 行），其中 `'diet'` 对应饮食方案、`'exercise'` 对应运动方案——这两个值在 4.5 节生活方案功能中有完整的业务场景描述。但 `'other'` 枚举值在全文中从未出现——4.5 节仅描述饮食和运动两类方案，6.5 节 POST /api/plan/generate 只提及"饮食和运动方案"，Dify 工作流 life-plan-generator 的输出描述同样仅覆盖饮食和运动。`'other'` 是一个已定义但无业务归宿的枚举值，实现者无法确定：（1）`'other'` 类型的方案项何时生成；（2）前端应如何渲染 `'other'` 类型的方案项（什么图标、什么标题、什么样式）；（3）打卡分析中 `'other'` 类型应如何归类统计。

**所在位置**：第 5 节第 611 行（life_plans 表 plan_type CHECK 约束）

**严重程度**：一般 —— 数据库约束定义了合法值，但业务功能未覆盖该值，实现者在处理 Dify 工作流输出时需要防御性编程，面对 `'other'` 值缺乏处理依据。

**改进建议**：（1）若 `'other'` 是为未来扩展预留的，应在表定义中标注"预留值，当前版本不使用"，并在 CHECK 约束旁加注释说明；（2）若 `'other'` 确实可能由 Dify 工作流产生（如方案包含无法归类为饮食或运动的建议项），应在 4.5 节补充 `'other'` 类型的说明（示例、前端渲染方式、打卡分析归类规则）；（3）若 `'other'` 是设计错误，应将其从 CHECK 约束中移除。

### 问题 7（一般）：Dify Agent Text2SQL 工具回调的 api_key 配置存在循环依赖

**问题描述**：第 5 节和 6.10 节定义的双认证模式中，Dify Agent 的 Text2SQL 工具在回调 `/api/admin/execute` 时需在请求体中携带 `api_key` 字段（其值须与 Express 环境变量 `DIFY_SERVICE_API_KEY` 一致，Express 以此验证请求来源）。这意味着 Dify Agent 的工具配置中需要预先填入 `DIFY_SERVICE_API_KEY` 的值——这是一个配置循环：Express 的 `.env` 文件和 Dify Agent 的工具回调配置都需要持有同一个密钥值。然而：

- 文档未说明 Dify Agent 工具配置中的 `api_key` 值如何填入——是由开发者在 Dify 平台 UI 上手动复制粘贴，还是由部署脚本自动注入。
- 文档未说明密钥轮换时（若需更换 `DIFY_SERVICE_API_KEY`），需要同步更新 Dify Agent 工具配置——遗漏此步骤将导致所有 AI 助手的 Text2SQL 操作静默失败（Express 返回 403）。
- 部署复杂度被低估——这个跨平台密钥同步是常规 Web 应用部署中不会遇到的额外操作点。

**所在位置**：第 5 节第 622-626 行（双认证模式描述），第 6.10 节第 1034-1036 行（api_key 参数说明）

**严重程度**：一般 —— 不影响需求正确性，但增加了部署和运维的复杂度，且密钥不同步时的故障排除成本高。

**改进建议**：（1）在 5 节双认证模式说明后增加一条"部署注意事项"——明确指出 Dify Agent Text2SQL 工具配置中的 `api_key` 固定值需与 Express 环境变量 `DIFY_SERVICE_API_KEY` 保持一致，密钥轮换时需在两侧同步更新；（2）建议在 `.env.example` 模板文件中为 `DIFY_SERVICE_API_KEY` 添加注释提示"此值需同步配置至 Dify 平台的 Agent 工具回调模板中"；（3）补充密钥轮换时的故障排查指引方向（若 AI 助手对话返回"操作执行失败"，首先检查 DIFY_SERVICE_API_KEY 两侧是否一致）。

### 问题 8（轻微）：检验按钮行为与推荐组件库 Vant 4 的默认行为矛盾

**问题描述**：第 7.1 节交互状态规范中规定"表单提交按钮在存在校验错误时保持可点击——点击后焦点自动跳转至第一个错误字段并展示其错误提示"（第 1211 行）。然而文档第 8.1 节推荐使用 Vant 4 作为移动端 UI 组件库（第 1310 行），Vant 4 的 Form 组件默认行为与上述规定相反——当表单存在校验错误时，Submit 按钮自动进入 `disabled` 状态（通过 `submit` 事件回调返回的 `Promise.reject` 触发），用户点击无效。

实现者面临着"遵循需求文档规定"还是"遵循组件库默认行为"的选择——两者实现路径不同：前者需要自定义覆盖 Vant 的 disabled 逻辑，后者可以直接使用 Vant 默认行为（代码量更少、用户预期更一致）。文档未给出优先级的指引。

**所在位置**：第 7.1 节第 1211 行（检验按钮行为描述）与第 8.1 节第 1310 行（Vant 4 推荐）

**严重程度**：轻微 —— 可在开发阶段协商解决，但增加了前端开发者的决策成本和可能的返工。

**改进建议**：（1）若"保持可点击"行为是设计意图，在交互状态规范中增加一条说明"若组件库默认行为与此规范冲突，以本规范为准，通过覆盖组件库样式或行为实现"；（2）若可接受 Vant 默认行为（disabled 按钮），将规范修订为"按钮在表单存在校验错误时显示为禁用态（遵循所选组件库的默认行为），用户在修正所有错误后方可点击提交"；（3）在 UI 组件库选型评估维度（第 8.1 节第 1310 行）中增加"表单校验行为的可定制性"作为评估项。

### 问题 9（轻微）：风险预测表单中"重新填写"按钮的清除机制与 sessionStorage 备份策略存在空白

**问题描述**：第 4.4 节表单数据跨步骤持久化方案（第 263 行）规定"用户提交成功或点击'重新填写'按钮时清除 Pinia store 和 sessionStorage 中的表单数据"。但文档在描述 sessionStorage 作为页面刷新恢复的备份（第 261 行："在用户刷新页面或意外离开后返回时，从 sessionStorage 恢复数据"）的同时，缺少对以下边界场景的处理说明：

- 用户点击"重新填写"后，如果页面刷新，sessionStorage 是否仍有残留数据触发恢复？清除操作的时序（先清 Pinia 还是先清 sessionStorage）是否影响恢复逻辑的触发条件判断？
- 如果用户刷新页面后（从 sessionStorage 恢复了数据），该数据是否应视为"已填写但未提交"的状态，从而阻止"重新填写"后刷新导致的数据残留？

这是一个实现层面的细微时序问题，但涉及表单数据的可靠性——错误的清除时序可能导致用户以为数据已清除，刷新后却发现旧数据仍然回填。

**所在位置**：第 4.4 节第 259-265 行（表单数据跨步骤持久化描述）

**严重程度**：轻微 —— 实现者大概率能自行处理此边界情况，但明确的时序约定可避免不同实现者对同一场景的不同处理。

**改进建议**：（1）在第 4.4 节持久化方案中补充一条时序约定："点击'重新填写'时，先清除 sessionStorage，再重置 Pinia store，以确保刷新后不会从 sessionStorage 恢复已清除的数据"；（2）在 riskFormStore TypeScript 示例（第 484-544 行）的 `clearFormData` 方法中体现此时序（当前示例已正确实现此顺序，但文档正文描述可明确引用）。
