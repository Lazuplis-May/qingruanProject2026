根据判定标准：诊断报告包含严重和一般等级的问题，且质询报告确认了这些问题确实存在（LOCATED），满足 RETRY 条件。

## 需要解决的问题

- **问题描述**：Dify 平台 `{{user}}` 变量透传能力未实证验证即作为核心架构基础——核心架构假设（AI-to-database 授权管道依赖 Dify Agent 工具回调中 `{{user}}` 的动态透传）在未经验证的情况下被直接用作架构设计前提，且该验证被推迟至开发环境搭建阶段。若验证未通过，主方案和备选方案均无法工作，存在显著返工风险。
- **所在位置**：第 5 节"Dify 平台能力验证任务"段落（文档第 627 行附近）
- **严重程度**：严重
- **改进建议**：（1）将 Dify 平台能力验证前置为门禁任务，在概要设计启动前完成；（2）在验证完成前明确标注所有依赖此能力的功能模块（4.8 AI 智能助手的 Text2SQL 操作、4.9 智能管理）为"待 Dify 平台能力验证后确认的技术方案"；（3）补充验证未通过时的完整替代架构方案（如 Express 代理层在转发 Dify Agent 请求前预查询用户数据并注入对话上下文）。

- **问题描述**：7x24 可用性承诺与 SQLite 单实例架构存在不可调和的事实矛盾——第 7.2 节要求"系统应保持 7x24 小时连续运行能力"，但第 7.4 节同时承认 SQLite 为文件型数据库，不具备主从复制或自动故障转移能力，服务器宕机时 API 服务和数据访问将全部中断。SQLite 单实例架构无法满足 7x24 可用性 SLA，且文档未给出宕机恢复时间目标（RTO）。
- **所在位置**：第 7.2 节"性能"第 4 条（文档第 1182 行）与第 7.4 节"SQLite 数据可靠性说明"（文档第 1228 行）
- **严重程度**：严重
- **改进建议**：（1）将"7x24 小时连续运行能力"修改为符合 SQLite 单实例架构实际可用性的表述，或明确标注 RTO；（2）在第 7.4 节补充与服务可用性声明的交叉引用，明确标注当前架构下的可用性约束是已知的架构取舍；（3）若实训项目对可用性无硬性要求，建议直接删除 7x24 声明。

- **问题描述**：Dify 工作流 `health-article-generator` 的 `user_id` 参数在专用端点中的传递路径未显式说明——POST /api/articles/generate 作为专用端点，其服务端处理流程仅笼统描述"调用 Dify health-article-generator 工作流"，未显式说明 Express 如何将 JWT 中的 user_id 映射为工作流 inputs 中的 `user_id` 字段。通用代理端点的说明不适用于专用端点。
- **所在位置**：第 6.7 节 POST /api/articles/generate 端点与第 6.11 节 Dify 工作流清单汇总表
- **严重程度**：一般
- **改进建议**：（1）在 6.7 节服务端处理流程中显式补充 JWT user_id → Dify 工作流 inputs.user_id 的映射步骤；（2）在第 6 节导言"请求参数映射说明"表格中增加 JWT Token.user_id → Dify 工作流 inputs.user_id / Dify Agent `user` 参数的映射约定行；（3）同步检查其他依赖 JWT user_id 自动填充的专用端点。

- **问题描述**：通用 Dify 代理端点与专用代理端点的职责边界未定义——两组端点的关系、调用路径（前端是否直接调用通用端点）、专用端点是否内部复用通用端点逻辑、通用端点是否仅供内部使用等问题均未明确，影响实现决策和代码复用。
- **所在位置**：第 6.11 节"通用 Dify 代理端点"与第 6.4/6.9/6.10 节各专用端点
- **严重程度**：一般
- **改进建议**：（1）在第 6.11 节明确通用代理端点的定位，建议标注为"Express 内部代理层使用的通用转发端点，由各专用端点内部调用，不直接暴露给前端"；（2）更新 Dify 工作流清单汇总表的"触发端点"列，对由专用端点间接触发的工作流标注实际的专用端点路径；（3）在架构层面明确分层：前端 → 专用 API 端点 → 通用 Dify 代理端点 → Dify API。

- **问题描述**：修订说明章节存在逆时序排列——第 17 节（v1→v2，最早修订批次）被插入在第 16 节（v6→v7）和第 18 节（v7→v8）之间，破坏时间线一致性，实现者按章节顺序阅读会产生"前面刚定义、后面又删除"的困惑。
- **所在位置**：第 10-18 节修订说明（文档第 1385 行起）
- **严重程度**：一般
- **改进建议**：（1）将所有修订说明按时间线从旧到新排列（v1→v2 至 v7→v8）；（2）在修订说明总起段前增加阅读指引。

- **问题描述**：Dify Agent 工具链定义完全推迟至概要设计阶段——Agent 能调用哪些工具、每个工具的输入输出 schema、权限边界等信息直接影响前端交互设计，当前在需求阶段未列出任何 Agent 预期工具清单，意味着前后端开发者进入概要设计时仍需返回需求层面做大量澄清。
- **所在位置**：第 6.11 节 Dify 工作流清单汇总表、第 4.8 节 AI 智能助手、第 4.9 节智能管理
- **严重程度**：一般
- **改进建议**：（1）在需求文档中至少列出每个 Agent 的"预期工具清单"，标注每个工具的用途、输入输出类型和权限约束；（2）为每个工具标注对前端交互的影响类型；（3）若确实无法确定完整清单，至少应明确工具定义的产出时间节点和责任人。

- **问题描述**：枚举字段的具体字符串值在全文中未统一约定——`diabetes_history`、`family_history`、`gender` 等枚举字段在三处描述的精度不一致，前后端开发者无法独立确定枚举值的具体字符串（中文还是英文）、是否需要中英文映射、是否需要白名单校验。
- **所在位置**：第 4.4 节风险预测、第 6.3 节 API 参数表、第 5 节 user_risk_info 表
- **严重程度**：一般
- **改进建议**：（1）在第 6.3 节各枚举参数说明中明确约定枚举值字符串（推荐与 Dify 工作流 inputs 变量保持一致），标注"其他值将被服务端拒绝"；（2）在第 5 节表定义为枚举字段补充 CHECK 约束或枚举值说明；（3）同步检查 `family_history`、`gender` 等类似字段。

- **问题描述**：前端 UI 组件库选型缺失——移动端优先的 Web 应用全文未涉及 UI 组件库选型（如 Vant、Element Plus、Naive UI 等），导致设计系统参数无法与组件库主题变量对接、移动端特有交互组件（Tabbar、ActionSheet、DatetimePicker 等）的实现方式悬而未决、开发者可能误认为所有 UI 组件需手动实现。
- **所在位置**：第 8.1 节"在范围内"、第 7.1 节"用户界面"
- **严重程度**：一般
- **改进建议**：（1）在第 8.1 节明确前端 UI 组件库选型（推荐 Vant 4 作为移动端优先的 Vue3 组件库）；（2）在第 7.1 节设计系统参数中补充与所选组件库主题变量的映射关系；（3）若推迟选型至少应在 8.1 节声明"待概要设计阶段确定"并列出评估维度。
