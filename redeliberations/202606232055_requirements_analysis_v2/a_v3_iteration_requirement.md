## 当前审查结果

第 2 轮质量审查报告（`b_v2_diag_v2.md`）经质询确认（LOCATED），共发现 8 个质量问题。以下按严重程度从高到低汇总：

### 严重问题

**问题 1：API 正常响应数据结构普遍缺失**
- 第 6 节定义了 20+ 个 API 端点及其请求体参数，但几乎所有端点的正常响应数据结构（HTTP 200 response body JSON schema）均未定义。文档仅在第 6.12 节定义了分页响应外壳格式和第 6.13 节定义了通用错误响应格式。以下关键端点响应体从文档中无法获知：POST /api/auth/login（JWT Token 字段名？role/userInfo 是否同响应返回？）、POST /api/risk/predict（风险评分/风险等级的字段名和取值枚举？）、GET /api/doctors（data 数组元素结构？）、POST /api/plan/generate（饮食/运动方案 JSON 结构？）、GET /api/articles/:id（文章详情字段？）、POST /api/assistant/chat（SSE data 字段结构？）
- 影响：前后端开发者无法独立并行开发——前端不知道 API 返回什么 shape 的数据，无法编写 TypeScript 接口类型和 mock 数据；后端开发者不知道 Express 端点应返回什么结构的 JSON
- 改进建议：（1）为核心 POST 端点补充响应体 JSON schema 示例；（2）为所有 GET 列表端点补充 data 数组元素的字段定义；（3）在 6.4/6.9 节补充 SSE 事件 data 字段的预期结构说明；（4）或在第 6 节导言部分统一声明各端点完整响应数据结构在概要设计阶段补充，并标注为明确的待办事项

### 一般问题

**问题 2：验收标准中的时间指标缺乏测量起点定义**
- 4.2 节"3 秒内开始收到流式回复"、4.4 节"10 秒内展示风险等级"、4.5 节"15 秒内开始返回方案"、4.6 节"15 秒内生成并展示文章"、4.8 节"唤起时间不超过 1 秒"、4.9 节"10 秒内完成并返回操作结果"——均未定义计时起点（用户触发操作时刻 vs 服务端收到请求时刻）。7.2 节已有"合理预期而非硬性约束"声明，但未交叉引用至各验收标准
- 改进建议：为每个时间指标补充计时起点定义（推荐统一采用用户触发操作的时刻）；区分前端计时和服务端计时的场景；对依赖外部 AI 服务的指标标注"基于 DeepSeek API 一般响应时间的推断，实际值可能受 API 负载和网络条件影响"

**问题 3：腰围/收缩压"未提供"的判定逻辑存在隐含歧义**
- 用户在表单中输入 0 vs 留空——两者是否同等处理？前端校验规则和服务端估算触发条件均未定义 null/undefined/0/空字符串的处理策略
- 改进建议：明确"未提供"判定规则（null/undefined/空字符串触发估算，0 视为无效输入并提示用户修正或留空）；在 6.3 节 waist/systolic_bp 参数说明中补充有效范围和校验错误码；前端添加 placeholder 提示"留空将自动估算"

**问题 4：多医生场景下 SSE 连接管理策略未定义**
- 用户与医生 A 对话（SSE 连接已建立），切换到医生 B——医生 A 的 SSE 连接是否关闭？前端同时维持的 SSE 连接数上限？后台对话的消息是否继续接收和缓存？当前文档仅覆盖 conversation_id 的存储恢复，未涉及实时 SSE 连接生命周期管理
- 改进建议：补充 SSE 连接管理策略——切换医生时关闭当前 SSE 连接；仅当前可见对话维持活跃连接；后台对话消息在用户切回时通过历史会话恢复

**问题 5：方案调整后旧方案关联的打卡记录处理策略未定义**
- 方案调整时新方案"替换"旧方案，但旧方案关联的打卡记录通过 plan_id 外键关联——物理删除将导致打卡记录 plan_id 悬空，逻辑过期需新增 is_active 字段
- 改进建议：明确方案替换策略，推荐采用"逻辑过期"（life_plans 表新增 is_active 字段）而非物理删除；打卡分析默认仅统计活跃方案项，历史分析可包含非活跃记录

**问题 6：user_risk_info 表字段与风险预测 API 参数不完全对应**
- diabetes_history（API 必填参数）在 user_risk_info 表中无直接对应字段；第 4.6 节健康资讯数据源描述中引用了表中不存在的"BMI"字段（表中仅有身高和体重）
- 改进建议：在 user_risk_info 表中明确 diabetes_history 的存储方式（独立字段或 JSON 字段）；将 4.6 节"BMI"修正为"身高和体重"或"基于身高和体重计算的 BMI"

### 轻微问题

**问题 7：`marked.js` 使用了非标准 npm 包名（第 1 轮遗留）**
- npm 官方包名为 `marked`，文档中两处（第 291 行、第 992 行）使用 `marked.js`，第 1 轮已识别但未修正
- 改进建议：将 `marked.js` 统一改为 `marked`

**问题 8：Dify 平台能力验证的备选方案缺少关键实现细节**
- 备选方案的 session_id 传递机制与主方案面临相同的 `{{user}}` 变量透传依赖；映射表的生命周期管理和存储位置未覆盖
- 改进建议：补充备选方案的 session_id 传递细节；或明确声明备选方案的完整设计留待概要设计阶段补充
