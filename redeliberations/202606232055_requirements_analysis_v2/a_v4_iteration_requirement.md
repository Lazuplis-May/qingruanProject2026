## 当前审查结果

本轮审查（第3轮v2修订版诊断报告，质询结论为 LOCATED，诊断结论全部确认）发现 **11 个问题**，按严重程度分级如下：

### 严重问题（2个）

1. **Dify 平台 `{{user}}` 变量透传能力未实证验证即作为核心架构基础**（问题1，严重）
   - 位置：第 5 节"Dify 平台能力验证任务"段落（第 627 行附近）
   - 描述：Dify Agent Text2SQL 工具回调中 `{{user}}` 变量的动态透传是整个 AI-to-database 授权管道的基石。第 5 节明确承认若 Dify 平台不支持此能力，"主方案和备选方案均无法工作"。然而该能力验证被推迟至"开发环境搭建阶段"执行，意味着当实现者进入概要设计乃至编码阶段时，核心架构假设可能被推翻。管理员通过智能管理界面操作数据库、普通用户通过 AI 助手触发 Text2SQL 操作——这两大功能路径均依赖此能力。
   - 建议：（1）将 Dify 平台能力验证前置为门禁任务（gate task），在概要设计启动前完成；（2）在验证完成前，在文档中明确标注所有依赖此能力的功能模块（4.8 AI 智能助手的 Text2SQL 操作、4.9 智能管理）为"待 Dify 平台能力验证后确认的技术方案"；（3）补充验证未通过时的完整替代架构方案，例如 Express 代理层在转发 Dify Agent 请求前预查询用户数据并注入对话上下文，使 Agent 回调不再需要携带 user_id。

2. **7x24 可用性承诺与 SQLite 单实例架构存在不可调和的事实矛盾**（问题2，严重）
   - 位置：第 7.2 节"性能"第 4 条（第 1182 行）与第 7.4 节"SQLite 数据可靠性说明"（第 1228 行）
   - 描述：第 7.2 节明确要求"系统应保持 7x24 小时连续运行能力"。第 7.4 节同时承认 SQLite 为文件型数据库，"不具备主从复制或自动故障转移能力。当服务器 1 发生宕机、硬件故障或磁盘损坏时，API 服务和数据访问将全部中断"。两条陈述构成直接矛盾。备份解决的是数据丢失风险（灾难恢复），而非服务可用性（高可用）。文档未给出宕机恢复时间目标（RTO）。
   - 建议：（1）将 7.2 节"7x24 小时连续运行能力"修改为符合 SQLite 单实例架构实际可用性的表述，例如"系统在设计运行时段内保持可用，计划内维护窗口除外"或明确标注 RTO；（2）在第 7.4 节补充与服务可用性声明的交叉引用；（3）若实训项目对可用性无硬性要求，建议直接删除 7x24 声明。

### 一般问题（6个）

3. **Dify 工作流 `health-article-generator` 的 `user_id` 参数在专用端点中的传递路径未显式说明**（问题3，一般）
   - 位置：第 6.7 节 POST /api/articles/generate 端点与第 6.11 节 Dify 工作流清单汇总表
   - 描述：通用代理端点的说明（"user 参数由 Express 从 JWT Token 中提取后自动填充"）位于通用代理端点，POST /api/articles/generate 是专用端点而非通过通用代理端点路由，其服务端处理流程仅笼统描述"调用 Dify health-article-generator 工作流"，未显式说明 Express 如何将 JWT 中的 user_id 映射为工作流 inputs 中的 `user_id`。
   - 建议：（1）在 6.7 节服务端处理流程描述中显式补充 Express 从 JWT Token 提取 user_id → 传入 Dify 工作流 inputs.user_id；（2）在第 6 节导言参数映射说明表格中增加 JWT Token.user_id → Dify 工作流 inputs.user_id / Dify Agent `user` 参数的映射约定行；（3）同步检查 POST /api/plan/generate 等其他依赖 JWT user_id 自动填充的专用端点。

4. **通用 Dify 代理端点与专用代理端点的职责边界未定义**（问题4，一般）
   - 位置：第 6.11 节"通用 Dify 代理端点"与第 6.4/6.9/6.10 节各专用端点
   - 描述：文档同时定义了通用代理端点和专用端点，但两组端点的关系从未被明确说明：专用端点内部是否复用通用端点的代理逻辑、前端应调用专用端点还是通用端点、通用端点是否仅供内部使用不应暴露给前端、若前端可以直接调用通用端点则某些专用端点是否冗余。
   - 建议：（1）在第 6.11 节开头标注通用代理端点的定位为"Express 内部代理层使用的通用转发端点，由各专用端点内部调用，不直接暴露给前端"；（2）更新 Dify 工作流清单汇总表的"触发端点"列，对由专用端点间接触发的工作流标注实际的专用端点路径；（3）在架构层面明确分层：前端 → 专用 API 端点（业务语义层）→ 通用 Dify 代理端点（转发层）→ Dify API。

5. **修订说明章节存在逆时序排列，影响文档可读性**（问题5，一般）
   - 位置：第 10-18 节修订说明（第 1385 行起）
   - 描述：第 17 节（v1→v2，最早修订批次）被插入在第 16 节（v6→v7）和第 18 节（v7→v8）之间，破坏时间线一致性。实现者按章节顺序阅读修订历史，会先看到 v6→v7 的修订内容（含 postMessage 协议定义、iframe 子页面路由规范等），然后读到 v1→v2 的修订内容（声明 iframe 架构已被移除），产生"前面刚定义、后面又删除"的困惑。
   - 建议：（1）将所有修订说明按时间线从旧到新排列：v1→v2 → v2→v3 → ... → v7→v8；（2）在修订说明总起段前增加阅读指引。

6. **Dify Agent 工具链定义完全推迟至概要设计阶段，AI 助手与智能管理的交互设计缺少关键约束**（问题6，一般）
   - 位置：第 6.11 节 Dify 工作流清单汇总表、第 4.8 节 AI 智能助手、第 4.9 节智能管理
   - 描述：Agent 挂载的工具链决定 AI 助手和智能管理功能边界——Agent 能调用哪些工具、每个工具的输入输出 schema、工具的权限边界——这些信息直接影响前端交互设计。当前在需求阶段将所有 Agent 工具定义推迟至概要设计阶段，意味着前后端开发者进入概要设计时仍需返回需求层面做大量澄清。
   - 建议：（1）在需求文档中至少列出每个 Agent 的"预期工具清单"，标注每个工具的用途、输入输出类型和权限约束；（2）为每个工具标注对前端交互的影响类型（是否需要特殊渲染/是否需要二次确认/是否仅后端执行）；（3）若确实无法在需求阶段确定完整清单，至少应明确 Agent 工具定义的产出时间节点和责任人。

7. **枚举字段的具体字符串值在全文中未统一约定，阻塞前后端协议对齐**（问题7，一般）
   - 位置：第 4.4 节风险预测（病史状态选择）、第 6.3 节 API 参数表（`diabetes_history` 参数）、第 5 节 user_risk_info 表（`diabetes_history` 字段）
   - 描述：三处描述的精度不一致——第 4.4 节使用中文自然语言（"健康、糖尿病前期、已确诊"），第 6.3 节使用"如"字暗示为示例而非穷举，第 5 节无约束说明。此外 `family_history`（有/无）、`gender`（男/女）等字段存在相同的约定缺失问题。前后端开发者无法独立确定：枚举值是中文还是英文、是否需要中英文映射、是否需要白名单校验。
   - 建议：（1）在第 6.3 节各枚举参数说明中明确约定枚举值字符串（推荐与 Dify 工作流 inputs 变量保持一致，如 `"healthy"` / `"prediabetes"` / `"diagnosed"`），并标注"其他值将被服务端拒绝"；（2）在第 5 节表定义为枚举字段补充 CHECK 约束或枚举值说明；（3）同步检查 `family_history`、`gender` 等类似枚举字段，一并规范化。

10. **前端 UI 组件库选型缺失，移动端优先的界面开发缺少关键依赖声明**（问题10，一般）
   - 位置：第 8.1 节"在范围内"（前端构建方式描述）、第 7.1 节"用户界面"（设计系统基础参数）
   - 描述：文档明确平台为"移动端优先的 Web 应用"，定义了完整的设计系统基础参数，但全文未涉及前端 UI 组件库的选型。影响：（1）移动端特有交互组件（Tabbar、ActionSheet、DatetimePicker、PullRefresh 等）从零开发工作量极大；（2）设计系统参数（色值、字号、间距）需与组件库主题变量对接——若组件库未确定，CSS 变量方案无法确定；（3）8.1 节仅提及 Swiper 和 marked 两个 npm 依赖，易误导开发者认为其他 UI 组件需手动实现。
   - 建议：（1）在第 8.1 节明确前端 UI 组件库选型（推荐 Vant 4 作为移动端优先的 Vue3 组件库）；（2）在第 7.1 节设计系统参数中补充与所选组件库主题变量的映射关系；（3）若有意推迟选型至概要设计阶段，至少应在 8.1 节声明"移动端 UI 组件库选型待概要设计阶段确定"并列出评估维度。

### 轻微问题（3个）

8. **功能验收标准的时间指标对 DeepSeek API 负载的"合理预期"免责声明存在详细程度不一致**（问题8，轻微）
   - 位置：第 4.2/4.4/4.5/4.6/4.9 节各验收标准段落
   - 描述：各验收标准中免责声明长度不一——部分指标后附加较长免责声明文本，部分仅简短标注，增加了验收标准的解释歧义。
   - 建议：（1）在第 7.2 节统一定义"合理预期而非硬性约束"的语义，各验收标准仅引用此统一定义；（2）统一各验收标准中免责声明的措辞，采用统一模板。

9. **SSE 事件格式的交叉引用链存在间接跳转**（问题9，轻微）
   - 位置：第 6.4 节（医师对话）、第 6.9 节（SSE 事件结构说明）、第 6.10 节（管理员对话）
   - 描述：三个 SSE 流式端点的引用关系为"第 6.10 节 → 第 6.4 节 → 第 6.9 节"，形成两跳引用链，增加实现者查找成本。
   - 建议：（1）在第 6.10 节和第 6.4 节均直接引用第 6.9 节；（2）或在第 6.9 节明确标注其 SSE 事件结构适用于文档中所有 SSE 流式端点。

11. **localStorage 跨浏览器标签页的登录态同步行为未定义**（问题11，轻微）
   - 位置：第 4.10 节"Vue3 SPA 的登录态共享"、Pinia authStore 定义
   - 描述：当用户在同一浏览器中打开多个标签页时，若在一个标签页中执行登出操作，其他标签页的 Pinia store 不会自动感知此变更。现代 SPA 应用中多标签页使用是常见场景，若此行为未定义可能导致：标签页 A 登出后标签页 B 仍显示已登录状态且允许操作（实际 API 调用返回 401）。
   - 建议：（1）在第 4.10 节补充跨标签页登录态同步策略：监听 `window.addEventListener('storage', ...)` 事件，当检测到 `token` 或 `role` 键在其他标签页中被修改或移除时，同步更新当前标签页的 Pinia store 状态；（2）将跨标签页同步的具体实现标注为概要设计阶段确定。
