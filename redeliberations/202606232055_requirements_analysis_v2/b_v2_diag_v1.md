## 本轮新发现的质量问题

### 问题 1：路由表与 TypeScript 路由配置代码示例不一致（严重）

**问题描述**：第 1.1 节的路由映射表（第 19-32 行）列出了 12 条路由，包含 `/news` 和 `/news/article/:id` 两条健康资讯路由。但紧随其后的 TypeScript 路由配置示例（第 50-108 行）完全遗漏了这两条路由——示例中 `routes` 数组不含任何 `/news` 相关路由定义。实现者若以 TypeScript 代码示例为起点开始编码，将遗漏整个健康资讯模块的前端路由入口。

**所在位置**：
- 路由表（第 25-26 行）：定义了 `/news` 和 `/news/article/:id`
- TypeScript 路由配置示例（第 50-108 行）：缺少上述路由

**严重程度**：严重。路由配置是前端 SPA 的骨架，代码示例与实际路由表不一致会直接导致实现者在早期阶段遗漏模块入口，后续返工成本高。

**改进建议**：
- 在 TypeScript 路由配置示例中补充 `/news`（指向 `views/NewsView.vue`）和 `/news/article/:id`（指向 `views/ArticleDetailView.vue`，`props: true`）两条路由。
- 或在该示例前增加明确声明："以下为代表性路由示例，完整路由配置包含表中全部 12 条路由，具体懒加载策略在概要设计阶段确定"。

实现者从文档只能获知 `chatStore` 应维护 `Map<doctorId, conversationId>` 映射和 `assistantConversationId`，但这些类型信息的表达是松散的自然语言描述，而非可指导编码的 TypeScript 接口定义。

**所在位置**：
- `chatStore`：第 204 行（医师对话）、第 334 行（AI 助手对话）
- `riskFormStore`：第 255-256 行（跨步骤表单持久化）

**严重程度**：一般。不阻塞需求理解，但实现者在编码时需自行设计这两处 store 的类型结构，可能引入与整体架构不一致的设计。

**改进建议**：
- 为 `chatStore` 补充简化的 TypeScript 接口定义（至少包含 `Map<number, string>` 类型的医生会话映射和 `assistantConversationId` 字段）。
- 为 `riskFormStore` 补充简化的 TypeScript 接口定义（至少含表单字段类型及各步骤字段分组）。

### 问题 3：Vite 开发服务器 API 代理配置未提及（一般）

**问题描述**：文档明确前端使用 Vite 构建工具（第 5 行、第 992-993 行），后端 Express 部署在服务器1的3000端口上。在本地开发环境中，Vite 开发服务器（默认端口 5173）和 Express 后端（端口 3000）运行在不同端口上，前端通过 `/api/*` 相对路径发起请求将因跨域无法到达后端。这需要 Vite 的 `server.proxy` 配置将 `/api` 请求代理至 Express。

文档第 6 节 API 接口规格全部使用相对路径（如 `/api/auth/login`），暗示同域部署，但未交代本地开发阶段的跨域代理方案。实现者若不了解此机制，会在首次启动前端开发服务器时遇到 CORS 错误或 404。

**所在位置**：全文无涉及 Vite 开发代理配置的内容。最接近的提及是第 7.4 节部署架构（生产环境 Nginx 反向代理），但不覆盖本地开发场景。

**严重程度**：一般。不阻塞需求理解，但会影响开发环境搭建阶段的首日体验，可能导致实现者自行添加低质量的临时 CORS 解决方案。

**改进建议**：
- 在第 8.1 节"在范围内"或第 7.5 节"环境配置"中增加一项："本地开发时 Vite 开发服务器需配置 `server.proxy` 将 `/api` 请求代理至 Express 后端（默认 `http://localhost:3000`），具体配置在概要设计阶段写入 `vite.config.ts`"。

### 问题 4：移动端优先声明缺少关键移动 Web 技术规范（一般）

**问题描述**：第 1 节声明"以移动端优先的 Web 应用形态交付"（第 9 行），第 7.1 节以 375px 宽度为设计基准（第 846 行），但以下移动 Web 关键技术点未被覆盖：

1. **viewport meta 标签**：未指定 `<meta name="viewport" content="width=device-width, initial-scale=1.0">` 是移动端 Web 应用的基础配置。
2. **触摸事件处理**：移动端的 300ms 点击延迟（fastclick 或 `touch-action: manipulation`）未提及。
3. **安全区域适配**：iPhone 刘海屏/底部横条的 safe-area-inset 适配（`env(safe-area-inset-bottom)` 等）未提及，可能导致底部 Tab 栏与 iOS 底部手势条重叠。
4. **滚动穿透**：弹窗/对话窗口打开时背景页面的滚动穿透问题未提及。

这些是移动 Web 开发的常见坑点，在需求阶段明确有助于避免实现阶段的返工。

**所在位置**：第 7.1 节"用户界面"（第 844-877 行）和第 7.3 节"产品质量"（第 907-918 行）。

**严重程度**：一般。不阻塞功能实现，但缺少这些规范可能导致移动端体验问题在测试阶段才暴露。

**改进建议**：
- 在第 7.1 节或第 7.3 节中补充移动端 Web 特有的技术要求：viewport 配置、触摸事件优化（`touch-action: manipulation`）、iOS 安全区域适配、弹窗滚动穿透处理。标注具体实现方案在概要设计/UI 设计阶段细化。

### 问题 5：TypeScript 严格级别未指定（一般）

**问题描述**：文档要求前端统一使用 TypeScript（修订指令第 2 条），但未指定 TypeScript 的编译严格级别（`tsconfig.json` 中 `strict: true` 与否）。`strict: true` 将启用 `strictNullChecks`、`noImplicitAny`、`strictFunctionTypes` 等 8 个子选项，对代码风格和类型注解的完整度要求有根本性影响。例如：
- 若 `strict: true`，`authStore` 中的 `token.value` 类型为 `string | null`，使用时必须进行 null 检查；
- 若 `strict: false`，`null` 可隐式赋值给 `string` 类型，类型安全形同虚设。

当前 Pinia authStore 示例（第 388 行 `token = ref<string | null>(...)`）按 strict 模式编写，但全文未做声明，实现者可能选择宽松模式导致类型安全降级。

**所在位置**：第 8.1 节目录结构中提及 `tsconfig.json`（第 993 行）但未说明其内容。

**严重程度**：一般。不阻塞需求理解，但影响代码质量控制基线。

**改进建议**：
- 在第 8.1 节补充一项："TypeScript 编译配置推荐启用 `strict: true`（严格模式），具体 `tsconfig.json` 完整配置在概要设计阶段确定"。

### 问题 6：前端 API 基础 URL 的策略未定义（一般）

**问题描述**：第 6 节所有 API 端点为相对路径（如 `/api/auth/login`），但前端 Axios 实例需要一个 `baseURL`。在开发环境（Vite dev server + proxy）和生产环境（Nginx 同域部署）中，API 基础 URL 可能不同。文档未说明前端如何获取 API 基础 URL——是构建时常量（`import.meta.env.VITE_API_BASE_URL`）、运行时相对路径（依赖同域部署）、还是有其他约定。

这影响 Axios 拦截器的初始化逻辑和 `.env.production` / `.env.development` 文件的变量定义。

**所在位置**：第 6 节 API 接口规格导言（第 533-536 行）提及"前端通过 HTTP 请求调用"但未说明 base URL 策略。第 8.1 节目录结构中提及 `src/api/` 目录用于"API 请求封装模块"（第 993 行）但未说明基础配置。

**严重程度**：一般。实现者需要自行决定，存在环境切换时配置不一致的风险。

**改进建议**：
- 在第 6 节导言或第 8.1 节中明确：前端 API 请求使用相对路径（依赖 Vite 开发代理和生产环境 Nginx 反向代理），或使用环境变量配置 base URL。具体方案在概要设计阶段确定。
