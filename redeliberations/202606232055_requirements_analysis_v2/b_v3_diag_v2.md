## 问题清单

### 问题1：Dify 平台 `{{user}}` 变量透传能力未实证验证即作为核心架构基础

- **所在位置**：第 5 节"Dify 平台能力验证任务"段落（文档第 627 行附近）
- **严重程度**：严重
- **问题描述**：

  文档将 Dify Agent Text2SQL 工具回调中 `{{user}}` 变量的动态透传作为整个 AI-to-database 授权管道的基石。第 5 节明确承认：若 Dify 平台不支持此能力，"主方案和备选方案均无法工作"。然而，该能力验证被推迟至"开发环境搭建阶段"执行，意味着当实现者进入概要设计乃至编码阶段时，核心架构假设可能被推翻。

  具体而言：（1）管理员通过智能管理界面操作数据库、普通用户通过 AI 助手触发 Text2SQL 操作——这两大功能路径均依赖 Dify Agent 能在工具回调请求体中动态引用对话上下文的 user 参数；（2）备选方案（Express 维护 session_id→user_id 映射表）面临相同的 `{{user}}` 变量透传依赖；（3）最终兜底方案 B（管理员手动输入 user_id 或写死测试用户 ID）被标注为"仅适用开发和演示阶段"，与系统作为完整实训项目的定位矛盾。

  从实现者视角看：在核心架构假设未被验证的情况下进入下游阶段，存在显著返工风险。

- **改进建议**：
  （1）将 Dify 平台能力验证前置为门禁任务（gate task），在概要设计启动前完成；
  （2）在验证完成前，在文档中明确标注所有依赖此能力的功能模块（4.8 AI 智能助手的 Text2SQL 操作、4.9 智能管理）为"待 Dify 平台能力验证后确认的技术方案"；
  （3）补充验证未通过时的完整替代架构方案，例如 Express 代理层在转发 Dify Agent 请求前预查询用户数据并注入对话上下文，使 Agent 回调不再需要携带 user_id。

### 问题2：7x24 可用性承诺与 SQLite 单实例架构存在不可调和的事实矛盾

- **所在位置**：第 7.2 节"性能"第 4 条（文档第 1182 行）与第 7.4 节"SQLite 数据可靠性说明"（文档第 1228 行）
- **严重程度**：严重
- **问题描述**：

  第 7.2 节明确要求"系统应保持 7x24 小时连续运行能力"。第 7.4 节同时承认 SQLite 为文件型数据库，"不具备主从复制或自动故障转移能力。当服务器 1 发生宕机、硬件故障或磁盘损坏时，API 服务和数据访问将全部中断"。两条陈述构成直接矛盾——SQLite 单实例架构无法满足 7x24 可用性 SLA。

  第 7.4 节提出的缓解措施仅为"数据库文件的定期备份策略"——备份解决的是数据丢失风险（灾难恢复），而非服务可用性（高可用）。服务器 1 宕机后，即便有最新备份，也需要人工介入恢复，期间系统不可用。文档未给出任何关于宕机恢复时间目标（RTO）的说明。

- **改进建议**：
  （1）将 7.2 节"7x24 小时连续运行能力"修改为符合 SQLite 单实例架构实际可用性的表述，例如"系统在设计运行时段内保持可用，计划内维护窗口除外"或明确标注 RTO（如宕机后 2 小时内恢复）；
  （2）在第 7.4 节"SQLite 数据可靠性说明"中补充与服务可用性声明的交叉引用，明确标注当前架构下的可用性约束是已知的架构取舍；
  （3）若实训项目对可用性无硬性要求，建议直接删除 7x24 声明，避免误导下游测试和验收。

### 问题3：Dify 工作流 `health-article-generator` 的 `user_id` 参数在专用端点中的传递路径未显式说明

- **所在位置**：第 6.7 节 POST /api/articles/generate 端点与第 6.11 节 Dify 工作流清单汇总表
- **严重程度**：一般
- **问题描述**：

  虽然第 6.11 节通用 Dify 代理端点说明中提及"user 参数由 Express 从 JWT Token 中提取后自动填充"，但该说明位于通用代理端点（`/api/dify/workflow/:workflow_id`）的参数表中，POST /api/articles/generate 是专用端点而非通过通用代理端点路由。专用端点的"服务端处理流程"仅笼统描述"调用 Dify health-article-generator 工作流"，未显式说明 Express 如何将 JWT 中的 user_id 映射为工作流 inputs 中的 `user_id` 字段。

  （2）在第 6 节导言"请求参数映射说明"表格中增加一行：Express 端 JWT Token.user_id → Dify 工作流 `inputs.user_id` / Dify Agent `user` 参数的映射约定；

### 问题4：通用 Dify 代理端点与专用代理端点的职责边界未定义

- **所在位置**：第 6.11 节"通用 Dify 代理端点"与第 6.4/6.9/6.10 节各专用端点
- **严重程度**：一般
- **问题描述**：

  第 6.11 节定义了两个通用 Dify 代理端点：`POST /api/dify/workflow/:workflow_id` 和 `POST /api/dify/agent/:agent_id`。同时，文档定义了专用端点（如 `POST /api/chat/doctor/:id`、`POST /api/assistant/chat`、`POST /api/admin/chat`）。Dify 工作流清单汇总表中，各工作流/Agent 的"触发端点"列有些指向专用端点（如 `/api/chat/doctor/:id`），有些指向通用端点（如 `/api/dify/workflow/:workflow_id`）。

  两组端点的关系从未被明确说明：（1）专用端点内部是否复用通用端点的代理逻辑？（2）前端应调用专用端点还是通用端点？（3）通用端点是否仅供内部使用、不应暴露给前端？（4）若前端可以直接调用通用端点 `/api/dify/agent/diabetes-assistant-agent`，则 `POST /api/assistant/chat` 是否冗余？

- **改进建议**：
  （1）在第 6.11 节开头明确说明通用代理端点的定位——建议标注为"Express 内部代理层使用的通用转发端点，由各专用端点内部调用，不直接暴露给前端"；
  （2）同步更新 Dify 工作流清单汇总表的"触发端点"列，对由专用端点间接触发的工作流（如 `health-article-generator` 由 `POST /api/articles/generate` 触发），标注实际的专用端点路径；
  （3）在架构层面明确分层：前端 → 专用 API 端点（业务语义层）→ 通用 Dify 代理端点（转发层）→ Dify API。

### 问题6：Dify Agent 工具链定义完全推迟至概要设计阶段，AI 助手与智能管理的交互设计缺少关键约束

- **所在位置**：第 6.11 节 Dify 工作流清单汇总表（"工具定义在概要设计阶段补充"）、第 4.8 节 AI 智能助手、第 4.9 节智能管理
- **严重程度**：一般
- **问题描述**：

### 问题7：枚举字段的具体字符串值在全文中未统一约定，阻塞前后端协议对齐

- **所在位置**：第 4.4 节风险预测（病史状态选择）、第 6.3 节 API 参数表（`diabetes_history` 参数）、第 5 节 user_risk_info 表（`diabetes_history` 字段）
- **严重程度**：一般（从轻微升级，因同时影响前后端独立开发）
- **问题描述**：

  第 4.4 节病史状态选择步骤描述用户可选"健康、糖尿病前期、已确诊"三种状态（中文自然语言）。第 6.3 节 POST /api/risk/predict 的 `diabetes_history` 参数类型标注为 `string`，说明为"用户的糖尿病病史状态（如：健康、糖尿病前期、已确诊）"——使用"如"字暗示此为示例而非穷举。第 5 节 user_risk_info 表定义中 `diabetes_history` 字段类型为 TEXT，无约束说明。

  三处描述的精度不一致，前端和后端开发者需要就以下问题达成一致才能独立开发：（1）枚举值的具体字符串是中文（"健康""糖尿病前期""已确诊"）还是英文（"healthy""prediabetes""diagnosed"）？（2）若 Dify 工作流的 inputs 变量使用英文枚举值，Express 代理层是否需要做中英文映射？（3）是否需要校验枚举值在白名单内（防止用户绕过前端直接调用 API 传入任意字符串）？

- **改进建议**：
  （1）在第 6.3 节 `diabetes_history` 参数说明中明确约定枚举值字符串（推荐与 Dify 工作流 inputs 变量保持一致，如 `"healthy"` / `"prediabetes"` / `"diagnosed"`），并标注"其他值将被服务端拒绝"；
  （2）在第 5 节 user_risk_info 表定义中为 `diabetes_history` 字段补充 CHECK 约束或枚举值说明；

### 问题8：功能验收标准的时间指标对 DeepSeek API 负载的"合理预期"免责声明存在详细程度不一致

- **所在位置**：第 4.2/4.4/4.5/4.6/4.9 节各验收标准段落
- **严重程度**：轻微
- **问题描述**：

  第 7.2 节性能要求统一声明：AI 内容生成操作的时间指标为"合理预期而非硬性约束"。但在第 4.4 节（风险预测 10 秒）、第 4.5 节（方案生成 15 秒）、第 4.6 节（资讯生成 15 秒）中，每个指标后均附加了较长的免责声明文本，与第 4.2 节医师对话的 3 秒指标（仅简短标注）在详细程度上不一致。

  从验收者视角看：免责声明长的指标可能被默认为"可以完全不达标"，而免责声明短的指标可能被认为"仍需基本达标"。这增加了验收标准的解释歧义。

- **改进建议**：
  （1）在第 7.2 节性能要求中统一定义"合理预期而非硬性约束"的语义（例如"标注为此类的指标不作为验收阻断项，但偏离超过 50% 时需在测试报告中记录原因"），各验收标准仅引用此统一定义而不再各自展开；
  （2）统一各验收标准中免责声明的措辞——推荐采用统一模板，如"此指标为合理预期，不作为硬性验收标准。详见 7.2 节性能要求说明"。

### 问题9：SSE 事件格式的交叉引用链存在间接跳转

- **改进建议**：
  （1）在第 6.10 节和第 6.4 节的 SSE 代理转发策略中均直接引用第 6.9 节（而非通过中间节跳转）；
  （2）或在第 6.9 节明确标注"本节定义的 SSE 事件结构适用于文档中所有 SSE 流式端点（POST /api/chat/doctor/:id、POST /api/assistant/chat、POST /api/admin/chat）"，各端点只需引用"详见 6.9 节"即可。

### 问题10：前端 UI 组件库选型缺失，移动端优先的界面开发缺少关键依赖声明

- **所在位置**：第 8.1 节"在范围内"（前端构建方式描述）、第 7.1 节"用户界面"（设计系统基础参数）
- **严重程度**：一般
- **问题描述**：

  文档明确平台为"移动端优先的 Web 应用"（第1节），在第 7.1 节定义了完整的设计系统基础参数（色值、字号、间距），在第 8.1 节提及 Swiper（轮播）和 marked（Markdown 解析）两个具体的 npm 依赖。然而，文档全文未涉及前端 UI 组件库的选型——即是否使用 Vant（移动端 Vue3 组件库）、Element Plus（桌面端优先）、Naive UI 或自研组件。

  此缺失对前端开发者的影响：（1）移动端优先的界面包含大量移动端特有的交互组件（如底部 Tab 栏、弹出层 ActionSheet、日期选择器、下拉刷新、无限滚动等），这些组件从零开发工作量极大；（2）第 7.1 节定义的设计系统参数（色值体系、字号层级、间距尺度）通常需要与 UI 组件库的默认主题变量体系对接——若组件库未确定，CSS 变量和主题定制的技术方案无法确定；（3）8.1 节仅提及 Swiper 和 marked 两个具体 npm 依赖，容易误导开发者认为其他 UI 组件都需要手动实现。

  此问题与 v1 审查报告问题 6（Agent 工具链定义推迟）性质类似，但属于前端维度的平行缺失——两者都反映了"关键依赖在需求阶段未选定、推迟至下游阶段决策"的模式。

- **改进建议**：
  （1）在第 8.1 节明确前端 UI 组件库选型（推荐 Vant 4 作为移动端优先的 Vue3 组件库，其提供 Tabbar、Dialog、Popup、DatetimePicker、PullRefresh、List 等内置组件，与文档中描述的底部 Tab 栏、弹窗提示、日期筛选、滚动列表等交互需求高度吻合）；
  （2）在第 7.1 节设计系统参数中补充与所选组件库主题变量的映射关系（如 Vant 的 CSS 变量 `--van-primary-color` 映射至文档定义的 `#4A90D9`），确保设计参数可直接落地为代码；
  （3）若有意推迟组件库选型至概要设计阶段，至少应在 8.1 节声明"移动端 UI 组件库选型待概要设计阶段确定"，并列出评估维度（移动端适配完备度、与 Vue3+TS 的兼容性、包体积、社区活跃度），避免开发者在缺乏约束的情况下各自为政。

### 问题11：localStorage 跨浏览器标签页的登录态同步行为未定义

- **所在位置**：第 4.10 节"Vue3 SPA 的登录态共享"、Pinia authStore 定义（第 378-417 行）
- **严重程度**：轻微
- **问题描述**：

  文档使用 localStorage 作为 JWT Token 和用户角色的持久化存储，并通过 Pinia authStore 统一管理登录态。然而，当用户在同一浏览器中打开多个标签页时，若在一个标签页中执行登出操作（`authStore.clearAuth()` → `localStorage.removeItem('token')`），其他标签页的 Pinia store 不会自动感知此变更——因为 Pinia 的 `ref` 读取的是初始化时的 localStorage 值，而 Vue 响应式系统不会跨标签页同步。

  现代 SPA 应用中多标签页使用是常见场景（用户从外部链接打开系统、从通知跳转等）。若此行为未定义，可能出现以下问题：（1）用户在标签页 A 登出，标签页 B 仍显示已登录状态且允许操作（但实际上 Token 已失效，API 调用返回 401）；（2）用户在标签页 A 登录，标签页 B 仍停留在登录页。

  localStorage 的 `storage` 事件（在非当前标签页中触发）可用于检测此类跨标签页变更，但文档未提及此机制。

- **改进建议**：
  （1）在第 4.10 节补充跨标签页登录态同步策略：监听 `window.addEventListener('storage', ...)` 事件，当检测到 `token` 或 `role` 键在其他标签页中被修改或移除时，同步更新当前标签页的 Pinia store 状态；
  （2）将跨标签页同步的具体实现标注为概要设计阶段确定（因涉及 Pinia 插件或 App.vue 全局监听的选择）。

## 整体评价

该文档经过 9 轮迭代修订（v1→v2 至 v7→v8），在以下方面已达到较高质量水平：

**需求响应充分**：经逐条对照验证，requirement.md 的三条修订指令均已落实——前端技术栈声明已统一为 Vue3 + TypeScript + Vite（第1节第5行），四个 TypeScript 代码示例覆盖路由配置、authStore、chatStore、riskFormStore（第1.1节、第4.10节），正文中未检出与 Vue3 SPA 矛盾的技术描述。

**接口规格完整性**：第 6 节已为 20+ 个端点定义了请求体参数和正常响应 JSON 结构（含字段类型和说明），分页响应格式（6.12）和通用错误响应格式（6.13）已统一。存在的主要缺口为：（1）部分参数映射路径不够显式（问题3、问题4）；（2）枚举字段的字符串值未在前后端之间约定（问题7）。

**前端开发支撑度**：路由配置、Pinia Store 定义、设计系统参数（色值/字号/间距）、交互状态规范（加载/空数据/错误）均已覆盖。存在的主要缺口为：（1）UI 组件库选型缺失（问题10）；（2）跨标签页登录态同步未定义（问题11）。

**历史问题修复**：经独立抽样验证，第1-2轮迭代中严重程度为"严重"的 3 个历史问题（技术栈声明未修订、iframe/SPA 架构二义性、API 响应数据结构缺失）均已在当前文档版本中修复。

本轮审查新发现的问题主要集中在：（1）核心架构依赖的实证验证缺失（问题1）；（2）非功能性需求与架构实现之间的可达成性矛盾（问题2）；（3）前端维度在 v1 审查中被系统性忽略的空白（问题10、问题11）；（4）部分接口参数的代理层映射路径不够显式（问题3、问题4）。

建议在进入概要设计阶段前优先解决"问题1"（Dify 平台能力验证前置）和"问题2"（7x24 声明与架构一致性），问题10（UI 组件库选型）建议在概要设计启动前确定以免阻塞前端开发。
