- **审查视角**：实现者视角——需求是否可直接作为开发依据、验收标准是否明确可量化、是否存在隐含的模糊地带
- **审查维度**：需求响应充分度、事实错误/逻辑矛盾、深度与完整性（侧重内部审议未充分覆盖的维度）

## 问题清单

### 问题1：Dify 平台 `{{user}}` 变量透传能力未实证验证即作为核心架构基础

- **所在位置**：第 5 节"平台存在三条数据操作路径"段落末尾，"Dify 平台能力验证任务"段落（文档第 627 行附近）
- **严重程度**：严重
- **问题描述**：

  文档将 Dify Agent Text2SQL 工具回调中 `{{user}}` 变量的动态透传作为整个 AI-to-database 授权管道的基石。第 5 节明确承认：若 Dify 平台不支持此能力，"主方案和备选方案均无法工作"。然而，该能力验证被推迟至"开发环境搭建阶段"执行，意味着当实现者进入概要设计乃至编码阶段时，核心架构假设可能被推翻。

  具体而言：（1）管理员通过智能管理界面操作数据库、普通用户通过 AI 助手触发 Text2SQL 操作——这两大功能路径均依赖 Dify Agent 能在工具回调请求体中动态引用对话上下文的 user 参数；（2）备选方案（Express 维护 session_id→user_id 映射表）面临相同的 `{{user}}` 变量透传依赖，因为 session_id 同样需要通过 Dify Agent 的工具回调传递；（3）最终兜底方案 B（管理员手动输入 user_id 或写死测试用户 ID）被标注为"仅适用开发和演示阶段"，与系统作为完整实训项目的定位矛盾。

  从实现者视角看：在核心架构假设未被验证的情况下进入下游阶段，存在显著返工风险——若 Dify 平台不支持此能力，整个 Text2SQL 授权管道需要重新设计，届时 API 接口规格、鉴权逻辑、前端交互流程都需联动修改。

- **改进建议**：
  （1）将 Dify 平台能力验证从"开发环境搭建阶段"前提至"需求分析阶段"或"架构设计阶段"之前，作为门禁任务（gate task）；
  （2）在验证完成前，在文档中明确标注所有依赖此能力的功能模块（4.8 AI 智能助手的 Text2SQL 操作、4.9 智能管理）为"待 Dify 平台能力验证后确认的技术方案"；
  （3）补充验证未通过时的完整替代架构方案（不仅限于 demo 阶段的写死 user_id），例如：放弃 Text2SQL 自动授权模式，改为 Express 代理层在转发 Dify Agent 请求前预查询用户数据并注入对话上下文——使 Agent 回调不再需要携带 user_id。

### 问题2：7x24 可用性承诺与 SQLite 单实例架构存在不可调和的事实矛盾

- **所在位置**：第 7.2 节"性能"第 4 条（文档第 1182 行）与第 7.4 节"SQLite 数据可靠性说明"（文档第 1228 行）
- **严重程度**：严重
- **问题描述**：

  第 7.2 节明确要求"系统应保持 7x24 小时连续运行能力"。第 7.4 节同时承认 SQLite 为文件型数据库，"不具备主从复制或自动故障转移能力。当服务器 1 发生宕机、硬件故障或磁盘损坏时，API 服务和数据访问将全部中断"。两条陈述构成直接矛盾——SQLite 单实例架构无法满足 7x24 可用性 SLA。

  第 7.4 节提出的缓解措施仅为"数据库文件的定期备份策略"——备份解决的是数据丢失风险（灾难恢复），而非服务可用性（高可用）。服务器 1 宕机后，即便有最新备份，也需要人工介入恢复，期间系统不可用。文档未给出任何关于宕机恢复时间目标（RTO）的说明。

  从实现者视角看：7x24 是一个可被下游测试验证的硬性指标。若测试团队按此标准验收，当前 SQLite 单实例架构将无法通过。实现者在选择技术方案时会陷入两难——按文档要求实现无法达成的 SLA，还是自行降级该需求。

- **改进建议**：
  （1）将 7.2 节"7x24 小时连续运行能力"修改为符合 SQLite 单实例架构实际可用性的表述，例如"系统在设计运行时段内保持可用，计划内维护窗口除外"或明确标注 RTO（如宕机后 2 小时内恢复）；
  （2）在第 7.4 节"SQLite 数据可靠性说明"中补充与服务可用性声明的交叉引用，明确标注当前架构下的可用性约束是已知的架构取舍；
  （3）若实训项目对可用性无硬性要求，建议直接删除 7x24 声明，避免误导下游测试和验收。

### 问题3：Dify 工作流 `health-article-generator` 的 `user_id` 参数在专用端点中的传递路径未显式说明

- **所在位置**：第 6.7 节 POST /api/articles/generate 端点（文档第 913-914 行）与第 6.11 节 Dify 工作流清单汇总表（文档第 1058-1060 行）
- **严重程度**：一般
- **问题描述**：

  第 6.11 节 Dify 工作流清单汇总表定义 `health-article-generator` 工作流的 `user_id` 为必填 inputs 变量（"用于查询用户健康数据，数据来源优先级已在 4.6 节定义"）。但第 6.7 节为该工作流定义的前端可见端点 POST /api/articles/generate 的请求体参数仅包含 `category`（可选），不含 `user_id`。

  虽然第 6.11 节通用 Dify 代理端点说明中提及"user 参数由 Express 从 JWT Token 中提取后自动填充"，但该说明位于通用代理端点（`/api/dify/workflow/:workflow_id`）的参数表中，POST /api/articles/generate 是专用端点而非通过通用代理端点路由。专用端点的"服务端处理流程"（第 914 行）仅笼统描述"调用 Dify health-article-generator 工作流"，未显式说明 Express 如何将 JWT 中的 user_id 映射为工作流 inputs 中的 `user_id` 字段。

  从实现者视角看：实现 POST /api/articles/generate 端点的开发者需要明确知道：（1）JWT 中的 user_id 是否需要显式映射到工作流 inputs 的 `user_id` 字段；（2）字段名是否需要转换（Express 端 `user_id` vs Dify 端 `user` 参数）；（3）若 Dify 工作流的 inputs 变量名与 Express 端的命名不一致，映射表在何处定义。当前文档将这一映射留给开发者自行推断。

- **改进建议**：
  （1）在 6.7 节 POST /api/articles/generate 的服务端处理流程描述中显式补充：Express 从 JWT Token 提取 user_id → 传入 Dify 工作流 inputs.user_id → 工作流根据 user_id 查询用户健康数据；
  （2）在第 6 节导言"请求参数映射说明"表格中增加一行：Express 端 JWT Token.user_id → Dify 工作流 `inputs.user_id` / Dify Agent `user` 参数的映射约定；
  （3）同步检查其他依赖 JWT user_id 自动填充的专用端点（如 POST /api/plan/generate），确保映射路径在各自的处理流程描述中均有提及。

### 问题4：通用 Dify 代理端点与专用代理端点的职责边界未定义，存在实现歧义

- **所在位置**：第 6.11 节"通用 Dify 代理端点"（文档第 1037-1052 行）与第 6.4/6.9/6.10 节各专用端点
- **严重程度**：一般
- **问题描述**：

  第 6.11 节定义了两个通用 Dify 代理端点：`POST /api/dify/workflow/:workflow_id` 和 `POST /api/dify/agent/:agent_id`。同时，文档第 6.4 节定义了专用端点 `POST /api/chat/doctor/:id`，第 6.9 节定义了 `POST /api/assistant/chat`，第 6.10 节定义了 `POST /api/admin/chat`。Dify 工作流清单汇总表（第 6.11 节末尾）中，各工作流/Agent 的"触发端点"列有些指向专用端点（如 `/api/chat/doctor/:id`），有些指向通用端点（如 `/api/dify/workflow/:workflow_id`、`/api/dify/agent/:agent_id`）。

  两组端点的关系从未被明确说明：（1）专用端点内部是否复用通用端点的代理逻辑？（2）前端应调用专用端点还是通用端点？（3）通用端点是否仅供内部使用、不应暴露给前端？（4）若前端可以直接调用通用端点 `/api/dify/agent/diabetes-assistant-agent`，则 `POST /api/assistant/chat` 是否冗余？

  从实现者视角看：实现者无法确定应实现哪些端点、哪些端点之间存在代码复用关系、前端的 API 调用应指向哪个 URL 模式。这可能导致重复实现或遗漏实现。

- **改进建议**：
  （1）在第 6.11 节开头明确说明通用代理端点的定位——建议标注为"Express 内部代理层使用的通用转发端点，由各专用端点内部调用，不直接暴露给前端"；
  （2）同步更新 Dify 工作流清单汇总表的"触发端点"列，对由专用端点间接触发的工作流（如 `health-article-generator` 由 `POST /api/articles/generate` 触发而非直接通过 `/api/dify/workflow/:workflow_id`），标注实际的专用端点路径；
  （3）在架构层面明确分层：前端 → 专用 API 端点（业务语义层）→ 通用 Dify 代理端点（转发层）→ Dify API。

### 问题5：修订说明章节存在逆时序排列，影响文档可读性

- **所在位置**：第 10-18 节修订说明（文档第 1385-2183 行）
- **严重程度**：一般
- **问题描述**：

  修订说明章节的排列顺序为：第 10 节（v2→v3）、第 11 节（v3→v3_rev1）、第 12 节（v3_rev1→v4）、第 13 节（v4→v5）、第 14 节（v5→v5_rev2）、第 15 节（v5_rev2→v6）、第 16 节（v6→v7）、第 17 节（v1→v2）、第 18 节（v7→v8）。

  第 17 节（v1→v2，最早的修订批次）被插入在第 16 节（v6→v7）和第 18 节（v7→v8）之间，破坏了从旧到新（或从新到旧）的时间线一致性。v1→v2 作为奠定文档基线的修订（统一技术栈为 Vue3+TS+Vite、消除 iframe 架构），按时间线应位于所有修订说明之首，而非插入在倒数第二节的位置。

  从实现者视角看：实现者若按章节顺序阅读修订历史，会先看到 v6→v7 的修订内容（含 postMessage 协议定义、iframe 子页面路由规范等），然后读到 v1→v2 的修订内容（声明 iframe 架构已被移除），产生"前面刚定义、后面又删除"的困惑。

- **改进建议**：
  （1）将所有修订说明按时间线从旧到新排列：v1→v2（第 17 节内容）→ v2→v3（第 10 节内容）→ v3→v3_rev1（第 11 节内容）→ ... → v7→v8（第 18 节内容）；
  （2）保持各节编号与时间线一致，避免跳号；
  （3）在修订说明总起段（第 10 节前）增加阅读指引："以下修订说明按修订发生的时间顺序排列，当前文档正文为所有修订合并后的最终版本"。

### 问题6：Dify Agent 工具链定义完全推迟至概要设计阶段，AI 助手与智能管理的交互设计缺少关键约束

- **所在位置**：第 6.11 节 Dify 工作流清单汇总表（文档第 1069 行）、第 4.8 节 AI 智能助手、第 4.9 节智能管理
- **严重程度**：一般
- **问题描述**：

  第 6.11 节 Dify 工作流清单汇总表的说明中明确："Agent 挂载的工具链（Text2SQL 工具、数据库知识库等）在 Dify 平台上配置，工具定义在概要设计阶段补充"。`diabetes-assistant-agent` 和 `admin-manager-agent` 的工具链是决定 AI 助手和智能管理功能边界的关键——Agent 能调用哪些工具、每个工具的输入输出 schema、工具的权限边界——这些信息直接影响前端交互设计。

  例如：若 `diabetes-assistant-agent` 挂载了"查询打卡记录"工具，AI 助手对话界面中是否需要展示打卡记录的富文本卡片？若 Agent 挂载了"生成健康建议"工具，前端是否需要为健康建议内容设计特殊的渲染组件？若 `admin-manager-agent` 挂载了"批量删除"工具，前端是否需要二次确认弹窗？这些交互决策都依赖 Agent 工具链的具体定义。

  当前文档在需求阶段将所有 Agent 工具定义推迟至概要设计阶段，意味着前后端开发者进入概要设计时仍需返回需求层面做大量澄清工作。

- **改进建议**：
  （1）在需求文档中至少列出每个 Agent 的"预期工具清单"——标注每个工具的用途、输入输出类型（不要求完整 schema，但需说明"做什么"和"产出什么"）以及权限约束（用户级/管理员级）；
  （2）为每个工具标注对前端交互的影响类型（是否需要特殊渲染/是否需要二次确认/是否仅后端执行），帮助概要设计阶段评估前端工作量；
  （3）若在需求阶段确实无法确定完整工具清单，至少应明确 Agent 工具定义的产出时间节点和责任人，避免在概要设计阶段因工具定义不明确而阻塞前端开发。

### 问题7：`diabetes_history` 字段的枚举值在全文中未统一约定

- **所在位置**：第 4.4 节风险预测（第 235 行）、第 6.3 节 API 参数表（第 755 行）、第 5 节 user_risk_info 表（第 608 行）
- **严重程度**：轻微
- **问题描述**：

  第 4.4 节病史状态选择步骤描述用户可选"健康、糖尿病前期、已确诊"三种状态（自然语言描述）。第 6.3 节 POST /api/risk/predict 的 `diabetes_history` 参数类型标注为 `string`，说明为"用户的糖尿病病史状态（如：健康、糖尿病前期、已确诊）"——使用"如"字暗示此为示例而非穷举。第 5 节 user_risk_info 表定义中 `diabetes_history` 字段类型为 TEXT，无约束说明。

  三处描述的精度不一致：（1）前端下拉选项的枚举值——是"健康""糖尿病前期""已确诊"还是英文字符串如 `"healthy"`/`"prediabetes"`/`"diagnosed"`？（2）若 Dify 工作流的 inputs 变量使用英文枚举值，Express 代理层是否需要做中英文映射？（3）是否需要校验枚举值在白名单内（防止用户绕过前端直接调用 API 传入任意字符串）？

  从实现者视角看：前后端开发者需要就枚举值的具体字符串达成一致才能独立开发。当前文档仅给出中文自然语言描述，前后端可能各自选择不同的字符串表示（中文 vs 英文 vs 数字编码），导致联调时协议不匹配。

- **改进建议**：
  （1）在第 6.3 节 `diabetes_history` 参数说明中明确约定枚举值字符串（推荐与 Dify 工作流 inputs 变量保持一致），如 `"healthy"` / `"prediabetes"` / `"diagnosed"`，并标注"其他值将被服务端拒绝"；
  （2）在第 5 节 user_risk_info 表定义中为 `diabetes_history` 字段补充 CHECK 约束或枚举值说明；
  （3）同步检查 `family_history`（有/无）、`gender`（男/女）等类似枚举字段是否存在相同的约定缺失问题，一并规范化。

### 问题8：功能验收标准的时间指标对 DeepSeek API 负载的"合理预期"免责声明在个别指标处存在过度软化

- **所在位置**：第 4.2/4.4/4.5/4.6/4.9 节各验收标准段落
- **严重程度**：轻微
- **问题描述**：

  第 7.2 节性能要求统一声明：AI 内容生成操作的时间指标为"合理预期而非硬性约束"。但在第 4.4 节风险预测验收标准（15 秒）、第 4.5 节方案生成验收标准（15 秒）、第 4.6 节资讯生成验收标准（15 秒）中，每个指标后均附加了较长的免责声明文本（如"10 秒为合理预期而非硬性约束，实际耗时受 Dify 工作流执行时间和 DeepSeek API 响应时间影响"），与第 4.2 节医师对话的 3 秒指标（仅简短标注"基于 DeepSeek API 一般响应时间的推断，实际值可能受 API 负载和网络条件影响"）在详细程度上不一致。

  从验收者（测试团队）视角看：若同一个"合理预期"概念在不同模块有不同长度的免责说明，验收者可能对不同指标采取不同的严格程度——免责声明长的指标可能被默认为"可以完全不达标"，而免责声明短的指标可能被认为"仍需基本达标"。这增加了验收标准的解释歧义。

- **改进建议**：
  （1）在 7.2 节性能要求中统一定义"合理预期而非硬性约束"的语义（例如"标注为此类的指标不作为验收阻断项，但偏离超过 50% 时需在测试报告中记录原因"），各验收标准仅引用此统一定义而不再各自展开；
  （2）统一各验收标准中免责声明的措辞长度和详细程度——推荐采用统一模板，如"此指标为合理预期，不作为硬性验收标准。详见 7.2 节性能要求说明"。

### 问题9：SSE 事件格式的交叉引用链存在间接性，增加实现者的查找成本

- **所在位置**：第 6.4 节（文档第 821 行）、第 6.9 节（文档第 980 行）、第 6.10 节（文档第 1017 行）
- **严重程度**：轻微
- **问题描述**：

  三个 SSE 流式端点的 SSE 事件格式引用关系为：第 6.10 节（管理员对话）引用第 6.4 节（医师对话），第 6.4 节再引用第 6.9 节（SSE 事件 data 字段结构说明）。第 6.9 节也引用自身（AI 助手端点）的处理策略与第 6.4 节一致。

  实现 POST /api/admin/chat 端点的开发者需要先跳转到 6.4 节，再跳转到 6.9 节，才能找到 SSE 事件的 data 字段结构定义。这个两跳引用链增加了不必要的查找成本。

- **改进建议**：
  （1）在第 6.10 节和第 6.4 节的 SSE 代理转发策略中均直接引用第 6.9 节（而非通过中间节跳转）；
  （2）或在第 6.9 节明确标注"本节定义的 SSE 事件结构适用于文档中所有 SSE 流式端点（POST /api/chat/doctor/:id、POST /api/assistant/chat、POST /api/admin/chat）"，各端点只需引用"详见 6.9 节"即可。

## 整体评价

该文档经过 9 轮迭代修订（v1→v2 至 v7→v8），在技术栈统一（Vue3+TS+Vite）、架构一致性（去除 iframe 歧义）、接口规格完整性方面已达到较高质量水平。文档对用户修订指令（requirement.md）的三个核心要求（前端技术栈统一、TypeScript 代码示例统一、消除内部矛盾）响应充分。

本轮审查发现的问题主要集中在：（1）核心架构依赖的实证验证缺失（Dify 平台能力未验证即作为架构基础）；（2）非功能性需求与架构实现之间的可达成性矛盾（7x24 vs SQLite 单实例）；（3）部分接口参数的代理层映射路径不够显式；（4）文档结构层面的可读性问题（修订说明逆时序排列、交叉引用链过长）。

建议在进入概要设计阶段前优先解决"问题1"（Dify 平台能力验证前置）和"问题2"（7x24 声明与架构一致性），其余问题可在下游设计阶段迭代修复。
