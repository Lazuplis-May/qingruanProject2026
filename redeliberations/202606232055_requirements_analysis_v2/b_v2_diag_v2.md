## 本轮新发现的质量问题

### 问题 1：API 正常响应数据结构普遍缺失（严重）

**问题描述**：第 6 节定义了 20+ 个 API 端点及其请求体参数，但几乎所有端点的正常响应数据结构（HTTP 200 response body JSON schema）均未定义。文档仅在第 6.12 节定义了列表类端点的分页响应外壳格式（`{data, pagination}`）和第 6.13 节定义了通用错误响应格式（`{error: {code, message}}`）。以下关键端点的响应体从文档中无法获知：

- `POST /api/auth/login`：JWT Token 以什么字段名返回（`token`？`access_token`？）？role 和 userInfo 是否与 Token 在同一响应中返回，还是需要额外请求？
- `POST /api/risk/predict`：风险评分（0-51 分）以什么字段名返回？风险等级（高风险/低风险）的字段名和取值枚举是什么？建议内容以什么结构返回？
- `GET /api/doctors`：医生列表项包含哪些字段（id、name、department、title、avatar？）？分页外壳内的 `data` 数组元素结构未定义。
- `POST /api/plan/generate`：饮食方案和运动方案以什么 JSON 结构返回？是嵌套数组还是扁平列表？方案项的 id、type、order、title、content 字段如何组织？
- `GET /api/articles/:id`：文章详情包含哪些字段（标题、正文 Markdown、标签列表、封面 URL、作者、发布时间、阅读量）？
- `POST /api/assistant/chat`（SSE 流式）：Dify Agent 返回的 SSE 事件格式虽在 Dify 平台侧定义，但 Express 代理层透传后前端实际收到的 `data` 字段结构未做任何说明。

**影响**：缺少响应数据结构意味着前后端开发者无法独立并行开发。前端开发者不知道 API 返回什么 shape 的数据，无法编写 TypeScript 接口类型和 mock 数据；后端开发者不知道 Express 端点应返回什么结构的 JSON，各端点响应格式可能不一致。

**所在位置**：第 6 节各子节（6.1-6.10），几乎所有非 GET 列表端点均缺少响应体定义。

**严重程度**：严重。API 规格是前后端协作的合同，响应数据结构的缺失使得"需求可直接作为开发依据"这一目标在当前文档状态下无法达成。质询报告（`b_v2_challenge_v1.md`）质询点 1 的覆盖率缺口第 4 条已明确指出此问题，本问题是对该质询意见的具体化确认。

**改进建议**：
- 为核心 POST 端点（至少 `/api/auth/login`、`/api/risk/predict`、`/api/plan/generate`）补充响应体 JSON schema 示例。
- 为所有 GET 列表端点补充 `data` 数组元素的字段定义。
- 在 6.4/6.9 节补充 SSE 事件 `data` 字段的预期结构说明（或明确声明其格式由 Dify 平台定义，并引用 Dify API 文档作为参考）。
- 若响应结构因依赖 Dify 工作流输出而无法在需求阶段确定，应在第 6 节导言部分统一声明"各端点的完整响应数据结构（JSON Schema）在概要设计阶段根据 Dify 工作流的实际输出格式补充"，并将此声明作为明确的待办事项标注——而非仅在个别端点处以零散注释形式提及。

### 问题 2：验收标准中的时间指标缺乏测量起点定义（一般）

**问题描述**：文档 4.1-4.9 节定义了多项以时间为量化指标的验收标准，但几乎所有时间指标的测量起点均未定义。开发者无法确定应从哪个事件开始计时、以何种方式测量：

| 验收标准 | 所在位置 | 缺失的起点定义 |
|---------|---------|--------------|
| "用户发送消息后 3 秒内开始收到流式回复" | 4.2 节，第 209 行 | 3 秒计时起点是"用户点击发送按钮"还是"消息到达 Express 服务端"？前者包含网络延迟，后者仅测量服务端处理时间。若为前端计时，网络波动可能使此指标不可稳定复现 |
| "10 秒内展示风险等级和对应的预防/管理建议" | 4.4 节，第 252 行 | 10 秒计时起点是"用户点击提交按钮"还是"服务端收到请求并开始调用 Dify 工作流"？中间包含网络传输、Dify 工作流排队等待时间 |
| "方案内容在 15 秒内开始返回" | 4.5 节，第 283 行 | 同上——计时起点未定义 |
| "15 秒内生成并展示完整的文章" | 4.6 节，第 296 行 | 同上 |
| "助手对话窗口从任意页面可通过 FAB 一键唤起，唤起时间不超过 1 秒" | 4.8 节，第 338 行 | "唤起时间"的定义是"点击 FAB 到对话窗口渲染完成"还是"点击 FAB 到对话窗口开始出现动画"？若窗口使用 CSS transition 动画，应在动画开始还是结束时计时？ |
| "全流程在 10 秒内完成并返回操作结果" | 4.9 节，第 357 行 | 此 10 秒包含 AI 自然语言理解、SQL 生成、数据库执行、结果返回的全链路耗时。考虑到 DeepSeek API 调用本身可能需要数秒、外加网络往返，10 秒是否作为可稳定达成的硬性约束需要验证 |

**影响**：缺少测量起点的时间指标无法被测试人员客观验证。不同开发者对计时起点的解读差异可能导致同一功能在不同人的测试中得出相反的"通过/不通过"结论。

**所在位置**：4.2 节（第 209 行）、4.4 节（第 252 行）、4.5 节（第 283 行）、4.6 节（第 296 行）、4.8 节（第 338 行）、4.9 节（第 357 行）。

**严重程度**：一般。不阻塞功能实现，但使验收标准丧失了可测试性。在测试阶段（特别是并行轨道 A5 性能测试）需要回溯到需求文档确认指标定义时，会发现缺乏明确的测量基准。

**改进建议**：
- 为每个含时间指标的验收标准补充"计时起点"定义。推荐统一采用"用户触发操作（点击/提交）的时刻"作为起点（即前端开始计时），因为这是用户实际感知的等待时间，与用户体验目标一致。
- 区分"前端计时"（含网络延迟）和"服务端计时"（仅服务端处理）的场景。若某指标在前端计时下难以稳定达成，应标注该时间为"合理预期而非硬性约束"（如 7.2 节中已有类似声明"合理预期而非硬性约束"，建议将该声明交叉引用至各验收标准）。
- 对依赖外部 AI 服务的时间指标（如 10 秒/15 秒），建议标注"基于 DeepSeek API 一般响应时间的推断，实际值可能受 API 负载和网络条件影响"。

### 问题 3：腰围/收缩压"未提供"的判定逻辑存在隐含歧义（一般）

**问题描述**：第 4.4 节描述"若用户未填写腰围或收缩压，系统基于以下公式估算替代值"（第 234 行）。但"未填写"与"填写了 0"之间的判定边界未定义。在 Web 表单中，用户可能在腰围输入框中输入 `0`（例如用户不知道自己的腰围但试图填写一个占位值）——此时前端和后端应如何处理？

- 若将 `0` 视为"未提供"并触发估算公式：用户输入 `0` 的结果与留空相同，但用户可能认为系统会使用 0 进行计算。
- 若将 `0` 视为有效值并代入公式：0cm 腰围在医学上无意义，可能导致风险评分失真。

同样的问题适用于收缩压（用户输入 `0` 或 `0mmHg`）。

此外，第 6.3 节 POST /api/risk/predict 请求体参数表中 `waist` 和 `systolic_bp` 标注为"否"（可选），但未说明服务端接收到 `null`、`undefined`、`0`、空字符串 `""` 时应分别如何响应——是触发估算、返回校验错误、还是使用默认值？

**所在位置**：
- 第 4.4 节第 234 行（估算触发条件描述）
- 第 6.3 节第 608-609 行（waist/systolic_bp 参数定义）

**严重程度**：一般。前端开发者需要知道表单校验规则（腰围字段是否允许输入 0？是否需要 placeholder 提示"留空将自动估算"？）；后端开发者需要知道服务端校验和估算逻辑的触发条件。当前歧义可能导致前后端对"未提供"的理解不一致。

**改进建议**：
- 在 4.4 节明确"未提供"的判定规则：建议采用"字段值为 `null`、`undefined` 或空字符串时触发估算，值为有效正数（包括 0，但 0 视为无效输入并提示用户修正或留空）时使用该值"。
- 在 6.3 节 waist/systolic_bp 参数说明中补充：有效范围为 >0 的数值；`null`/未传时触发服务端估算；`0` 或负数返回 `VALIDATION_ERROR`。
- 前端表单的腰围/收缩压输入框应添加 placeholder 提示（如"留空将自动估算"），降低用户输入 `0` 的概率。

### 问题 4：多医生场景下 SSE 连接管理策略未定义（一般）

**问题描述**：第 4.2 节明确提出"平台至少配置 3 位不同科室/领域的医生，每位医生可独立进行多轮对话"，且"切换医生后原对话上下文不丢失"（第 205 行）。但以下与 SSE 连接管理相关的问题未定义：

1. **连接保持策略**：用户与医生 A 对话（SSE 连接已建立，正在流式接收回复），此时切换到医生 B 并发送新消息——医生 A 的 SSE 连接是否应关闭？若保持，用户可能同时与 3 位医生进行流式对话，前端需维护 3 条并发的 SSE 连接（`fetch + ReadableStream`），每一条都在消耗浏览器连接资源。
2. **连接数上限**：平台未定义前端同时维持的 SSE 连接数上限。若用户快速切换医生并每次发起新对话，可能触发浏览器对同一域名的 HTTP 并发连接数限制（HTTP/1.1 通常为 6 个），导致新连接被阻塞。
3. **后台对话的消息接收**：若医生 A 的对话正在流式返回一段长篇回复，用户切换到医生 B 的页面——医生 A 的流式消息是否继续在后台接收并缓存？还是暂停/丢弃？用户切换回医生 A 时，是否能看到完整的已生成回复？

当前文档的会话管理描述（第 205-206 行）仅覆盖 conversation_id 的存储和恢复，未涉及实时 SSE 连接的生命周期管理。

**所在位置**：
- 第 4.2 节"会话管理"段落（第 205-206 行）
- 第 6.4 节 SSE 代理转发策略（第 631 行）

**严重程度**：一般。不阻塞核心对话功能的实现，但缺少此策略可能导致：（1）前端在实现时采用最简方案（切换医生即关闭所有连接），与"对话上下文不丢失"的承诺存在体验落差；（2）连接数未受控时可能触发浏览器限制，导致新对话无法发起，产生难以排查的 bug。

**改进建议**：
- 在 4.2 节"会话管理"段落中补充 SSE 连接管理策略：（1）明确切换医生时是否关闭当前 SSE 连接（建议：关闭，因为用户已离开该对话界面）；（2）明确后台保持的 SSE 连接数上限（建议：仅当前可见对话维持活跃连接，后台对话的消息在用户切回时通过历史会话恢复）；（3）明确用户切回医生 A 时，前端通过 conversation_id 调用 `/api/chat/doctor/:id` 恢复上下文（不发新消息），或在 UI 中展示"点击继续对话"以重新建立 SSE 连接。
- 或在 7.3 节健壮性规范中补充 SSE 连接管理的通用策略。

### 问题 5：方案调整后旧方案关联的打卡记录处理策略未定义（一般）

**问题描述**：第 4.5 节描述用户可通过 `/api/plan/adjust` 对已生成方案提出修改意见，系统根据反馈重新生成方案。第 6.5 节服务端处理流程明确"新生成的方案替换原有方案记录"（第 657 行）。但第 4.5 节和第 4.7 节均强调打卡记录通过 `plan_id` 外键关联到具体的 `life_plans` 方案项——这意味着方案被替换后，旧方案项的 `id` 可能不再存在，与之关联的打卡记录将成为孤立数据（`plan_id` 指向不存在的记录）。

以下场景的具体行为未定义：
- 用户生成了方案 A（含 4 条饮食项、3 条运动项，共 7 条 life_plans 记录），对其中早餐项完成了 3 次打卡。
- 用户对方案提出调整意见，系统生成方案 B（新的 7 条 life_plans 记录，新 id）。
- 方案 A 的记录被"替换"——是物理删除（DELETE FROM life_plans WHERE ...）还是逻辑标记为过期（新增 `is_active` 字段）？若物理删除，关联打卡记录的 `plan_id` 变为悬空外键；若保留，打卡分析时需能区分"当前方案的完成率"和"历史方案的完成率"。

**所在位置**：
- 第 4.5 节方案调整描述（第 265-267 行）
- 第 6.5 节 POST /api/plan/adjust 服务端处理流程（第 657 行）
- 第 4.7 节打卡依从性分析（第 302-303 行）
- 第 5 节 punch_in 表 plan_id 外键定义（第 478 行）

**严重程度**：一般。不影响方案生成和打卡的基本功能，但若采用物理删除方案记录的方式，将导致：（1）打卡记录的 `plan_id` 悬空，外键约束（SQLite 默认不强制但启用后）可能阻止删除；（2）打卡分析功能无法回溯"用户历史上对哪个方案项完成了打卡"，依从性分析的历史趋势数据丢失。

**改进建议**：
- 在 4.5 节或第 5 节明确方案替换的策略：推荐采用"逻辑过期"而非物理删除——life_plans 表新增 `is_active`（BOOLEAN，默认 1）字段，方案调整时将旧方案所有项的 `is_active` 置为 0，新方案项插入且 `is_active` 为 1。打卡分析的"按方案项依从性分析"默认仅统计 `is_active=1` 的方案项，历史分析可通过包含 `is_active=0` 的记录实现。
- 或在修订说明中声明方案替换策略留待概要设计阶段确定。

### 问题 6：user_risk_info 表字段与风险预测 API 参数不完全对应（一般）

**问题描述**：第 5 节 user_risk_info 表定义为"用户ID、年龄、性别、身高、体重、家族史、腰围、收缩压、妊娠状态、原始输入信息、疾病类型"。第 6.3 节 POST /api/risk/predict 请求体包含 10 个参数：`diabetes_history`、`diabetes_type`、`age`、`gender`、`height`、`weight`、`waist`、`systolic_bp`、`family_history`、`pregnancy`。

对照发现以下字段映射关系不明确：

| API 参数字段 | user_risk_info 表字段 | 问题 |
|-------------|----------------------|------|
| `diabetes_history`（必填） | 无直接对应字段 | "健康"/"糖尿病前期"/"已确诊"的病史状态存储在哪个字段？若存储在"原始输入信息"JSON 字段中，则无法直接用 SQL 条件查询"所有病史状态为糖尿病前期的用户" |
| `diabetes_type`（可选） | `疾病类型` | 若 `diabetes_history` 不是"已确诊"，`diabetes_type` 字段在表中存储什么值？NULL 还是空字符串？ |
| age, gender, height, weight, waist, systolic_bp, family_history, pregnancy | 对应中文名列 | 映射关系明确 |

此外，第 4.6 节健康资讯数据源描述中提到"优先取 user_risk_info 表中该用户最新一条记录中的年龄、性别、BMI、家族史、糖尿病类型等字段"——但 user_risk_info 表中并**没有 BMI 字段**（表中存储的是身高和体重，BMI 需要计算）。文档在此处引用了表中不存在的字段名。

**所在位置**：
- 第 5 节 user_risk_info 表定义（第 475 行）
- 第 6.3 节 POST /api/risk/predict 参数表（第 598-611 行）
- 第 4.6 节数据源优先级描述中引用的 BMI 字段（第 288 行）

**严重程度**：一般。`diabetes_history` 的存储位置影响数据库查询设计；BMI 引用错误可能导致开发者在 user_risk_info 表中寻找不存在的列。

**改进建议**：
- 在 user_risk_info 表中明确 `diabetes_history` 的存储方式：（a）新增独立字段 `diabetes_history`（TEXT），或（b）明确声明其存储在"原始输入信息"JSON 字段中。若选择方案（b），需说明通过 JSON 查询的约束（SQLite 的 JSON1 扩展支持 `json_extract`）。
- 将第 4.6 节"BMI"修正为"身高和体重"或"基于身高和体重计算的 BMI"，避免引用表中不存在的字段名。

### 问题 8：Dify 平台能力验证的备选方案缺少关键实现细节（轻微）

**问题描述**：第 5 节"Dify 平台能力验证任务"段落（第 494 行）描述了双认证模式依赖 Dify 平台支持 `{{user}}` 变量透传，以及若平台不支持的备选方案（Express 维护 session_id → user_id 映射表）。但备选方案的以下关键细节未覆盖：

1. **session_id 如何传递至 Dify Agent**：备选方案中，Express 在代理请求的 `user` 参数中传入 session_id（而非 user_id）。但 Dify Agent 的 Text2SQL 工具回调 `/api/admin/execute` 时，仍需将 session_id 携带在回调请求体中——这与主方案面临相同的问题：Dify 需要支持在工具回调中引用 `{{user}}` 变量。如果 Dify 不支持此能力，备选方案同样无法运作。备选方案并未绕过这一平台能力依赖。
2. **映射表的生命周期管理**：映射表条目"在用户登出或 Token 过期时清理"，但 Token 过期可能发生在用户持续使用平台的过程中（如 4.10 节描述的"跨天使用"场景）——此时映射表条目被清理但用户仍在活跃对话中，导致后续 Text2SQL 操作因 session_id 无效而失败。
3. **映射表的存储位置**：描述中使用"Map 或 Redis"，但 SQLite 部署在单台服务器上，Redis 不是已有组件——若使用内存 Map，服务重启后所有映射丢失。

**所在位置**：第 5 节，第 494 行。

**严重程度**：轻微。此问题涉及的是主方案失败后的备选路径，当前阶段（需求分析）的读者可接受备选方案尚未完全展开。但备选方案若存在与主方案相同的平台能力依赖，则验证任务后的决策路径（"主方案失败→启用备选方案"）实际上不可行。

**改进建议**：
- 在备选方案描述中补充说明：Express 生成 session_id 后，在首次代理请求的 `user` 参数中传入 `{session_id, user_id}` 的复合结构（如 `user: "sid_xxx|uid_yyy"`）。Dify Agent 在工具回调时将 `user` 原样携带。Express 的 `/api/admin/execute` 端点从 `user_id` 参数中分离 session_id 并查表获取实际 user_id。此方案同样依赖 Dify 在工具回调中引用 `{{user}}` 变量，若平台不支持，需降级到第三个备选方案（如 Express 在代理层将 user_id 附加到请求头 `X-User-Id` 并由 Nginx 注入回调请求——此路径需额外验证）。
- 或明确声明备选方案的完整设计在概要设计阶段补充，当前仅标注存在降级路径。
