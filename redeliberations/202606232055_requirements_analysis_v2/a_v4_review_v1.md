## 逐维度审查

### 1. 忠实性

**[通过]** 文档准确传达了原始需求中关于糖尿病预治智能助手的核心意图，9大功能模块均被覆盖。前端技术栈已统一为 Vue 3 + TypeScript + Vite，与原始修订指令一致。iframe 架构已被移除，统一为 Vue Router SPA 架构。

**[通过]** Dify `{{user}}` 变量透传能力验证已被提升为门禁任务（gate task），明确规定在概要设计启动前完成，且所有依赖此能力的核心功能模块已标注"待 Dify 平台能力验证后确认"。替代架构方案已补充。此修改准确反映了审查建议中的高优先级要求。

**[通过]** 7x24 可用性声明已被移除，替换为符合 SQLite 单实例架构实际可用性的表述（含 RTO 和实训项目定位说明），与 7.4 节 SQLite 数据可靠性说明形成一致。此修改准确反映了审查建议中的要求。

**[问题-一般]** 枚举字段的具体字符串值在全文中仍未统一约定。第 4.4 节和第 6.3 节中 `diabetes_history` 仍使用中文自然语言"如：健康、糖尿病前期、已确诊"描述，"如"字暗示为示例而非穷举。`gender`（男/女）、`family_history`（有/无）等字段存在相同的约定缺失。前后端开发者无法独立确定枚举的完整取值集合和字符串表示形式。此问题已在第3轮迭代中被识别（问题7），本轮仍未解决。

**[问题-轻微]** 4.8 节健康建议的触发机制已补充"由 Dify Agent 根据对话上下文自主决策"的描述，忠实反映了 Agent 模式的本质特性，但未标注此设计为推断性补充——下游架构师可能将其误解为经过验证的确定行为。

### 2. 清晰性

**[通过]** 通用 Dify 代理端点与专用端点的职责边界已明确定义。第 6.11 节明确标注通用代理端点为"Express 内部代理层的通用转发端点，由各专用端点在服务端内部调用，不直接暴露给前端"，并给出了清晰的架构分层：前端 → 专用 API 端点（业务语义层）→ 通用 Dify 代理端点（转发层）→ Dify API。Dify 工作流清单汇总表的"触发端点"列已标注实际专用端点路径。

**[通过]** JWT Token 中的 user_id 到 Dify 工作流 inputs.user_id 的映射已在多处显式说明。第 6 节导言参数映射说明表格增加了对应行，POST /api/articles/generate（第 6.7 节）和 POST /api/plan/generate（第 6.5 节）的服务端处理流程中均已显式描述 Express 从 JWT Token 提取 user_id → 传入 Dify 工作流 inputs.user_id 的完整路径。

**[问题-一般]** 修订说明章节仍存在逆时序排列。第 17 节（v1→v2，最早修订批次）位于第 16 节（v6→v7）和第 18 节（v7→v8）之间，破坏时间线一致性。实现者按章节顺序阅读修订历史时，会先看到 v6→v7 的修订内容（含 postMessage 协议定义、iframe 子页面路由规范等已废弃内容），然后读到 v1→v2 的修订内容（声明 iframe 架构已被移除），产生"前面刚定义、后面又删除"的困惑。此问题在第3轮迭代中已被识别（问题5），本轮仍未解决。

**[问题-一般]** Dify Agent 工具链定义仍完全推迟至概要设计阶段，AI 助手与智能管理的交互设计缺少关键约束。第 6.11 节和第 5 节仅笼统提及"Text2SQL 工具、数据库知识库等"，未列出任何 Agent 的预期工具清单。Agent 能调用哪些工具、每个工具的输入输出 schema、权限边界——这些信息直接影响前端交互设计（如 AI 助手对话中是否需要二次确认弹窗、哪些操作会产生副作用、哪些操作仅返回查询结果）。需求阶段完全空白意味着前后端开发者进入概要设计时仍需返回需求层面做大量澄清。此问题在第3轮迭代中已被识别（问题6），本轮仍未解决。

**[问题-轻微]** SSE 事件格式的交叉引用链仍存在间接跳转。第 6.10 节（管理员对话）引用"与 6.4 节医师对话接口一致"，第 6.4 节引用"见 6.9 节"，形成 6.10→6.4→6.9 的两跳引用链。虽然 6.9 节已标注其 SSE 事件结构"适用于所有 SSE 流式端点"，降低了实际查找成本，但第 6.10 节的引用路径仍保持间接跳转，与第 6.4 节同为引用方却路径不一致。此问题在第3轮迭代中已被识别（问题9），本轮仅部分缓解。

### 3. 完备性

**[通过]** Dify 平台能力验证的替代架构方案已补充完整，涵盖备选方案的 session_id 传递约束、映射表存储位置（Express 进程内存 Map 或 SQLite 持久化）、映射表生命周期管理（登出清理 + JWT 过期定时清理）、以及退化方案 B 的边界条件。验证未通过时的 fallback 路径已具备基本可执行性。

**[通过]** 用户认证与会话管理已完整覆盖 JWT 无状态机制、Token 过期处理（HTTP 拦截器 + SSE 对话中 + 多步骤表单中途 + 全局同步四种场景）、bcrypt 密码哈希、Pinia store 统一登录态管理。访问控制矩阵已按页面/功能模块定义三级权限边界。

**[问题-一般]** 前端 UI 组件库选型缺失，移动端优先的界面开发缺少关键依赖声明。文档明确平台为"移动端优先的 Web 应用"，定义了完整的设计系统基础参数（色值、字号、间距），但全文未涉及移动端 UI 组件库的选型。影响：（1）移动端特有交互组件（Tabbar、ActionSheet、DatetimePicker、PullRefresh、Toast 等）从零开发工作量极大；（2）设计系统 CSS 变量需与组件库主题变量对接，组件库未确定则 CSS 变量方案无法最终确定；（3）8.1 节仅提及 Swiper 和 marked 两个 npm 依赖，易误导开发者认为其他 UI 组件需手动实现。此问题在第3轮迭代中已被识别（问题10），本轮仍未解决。

**[问题-轻微]** 验收标准中的时间指标免责声明仍存在详细程度不一致。部分指标后附加较长免责声明文本（如 4.2 节的"基于 DeepSeek API 一般响应时间的推断，实际值可能受 API 负载和网络条件影响"），部分仅简短标注（如 4.5 节"15 秒为合理预期而非硬性约束"），增加了验收标准的解释歧义。7.2 节虽已统一定义"合理预期而非硬性约束"的语义，但各验收标准未统一引用此定义。此问题在第3轮迭代中已被识别（问题8），本轮未明显改善。

**[问题-轻微]** localStorage 跨浏览器标签页的登录态同步行为仍未定义。第 4.10 节 Pinia authStore 定义中未涉及 `window.addEventListener('storage', ...)` 事件监听，也未说明多标签页场景下的策略（是接受不同步还是需要同步）。当用户在同一浏览器中打开多个标签页时，若在一个标签页中执行登出操作，其他标签页的 Pinia store 不会自动感知此变更——标签页 A 登出后标签页 B 仍显示已登录状态且允许操作（实际 API 调用返回 401）。此问题在第3轮迭代中被识别（问题11），本轮仍未解决。

## 修改要求（仅 REJECTED 时存在）

- **问题**：修订说明章节存在逆时序排列（第 17 节 v1→v2 插入在第 16 节 v6→v7 和第 18 节 v7→v8 之间）
- **原因**：实现者按章节顺序阅读修订历史时会产生"前面刚定义、后面又删除"的困惑，影响对文档演进过程的理解。虽然不直接导致架构决策错误，但增加了理解文档的成本。
- **建议方向**：将所有修订说明按时间线从旧到新排列：v1→v2 → v2→v3 → v3→v3_rev1 → v3_rev1→v4 → v4→v5 → v5→v5_rev2 → v5_rev2→v6 → v6→v7 → v7→v8。同时建议在修订说明总起段前增加阅读指引。

---

- **问题**：Dify Agent 工具链定义完全推迟至概要设计阶段，AI 助手（diabetes-assistant-agent）和管理员 Agent（admin-manager-agent）的预期工具清单缺失
- **原因**：Agent 挂载的工具链决定 AI 助手和智能管理的功能边界——Agent 能调用哪些工具、每个工具的输入输出 schema、工具的权限边界——这些信息直接影响前端交互设计（如是否需要二次确认弹窗、哪些操作有副作用）。在需求阶段完全空白意味着前后端开发者进入概要设计时仍需返回需求层面做大量澄清，降低并行开发效率。
- **建议方向**：在需求文档中至少列出每个 Agent 的"预期工具清单"（工具名称、用途、输入输出类型、权限约束），并标注每个工具对前端交互的影响类型（是否需要特殊渲染/是否需要二次确认/是否仅后端执行）。若确实无法在需求阶段确定完整清单，至少应明确 Agent 工具定义的产出时间节点和责任人。

---

- **问题**：枚举字段的具体字符串值在全文中未统一约定（diabetes_history、gender、family_history 等字段）
- **原因**：前后端开发者无法独立确定枚举值的完整集合和字符串表示形式（中文还是英文？是否需要中英文映射？是否需要白名单校验？）。第 6.3 节 API 参数表中 "如" 字暗示为示例而非穷举，第 5 节表定义无 CHECK 约束说明，三处描述的精度不一致。前后端协议无法对齐，阻塞并行开发。
- **建议方向**：（1）在第 6.3 节各枚举参数说明中明确约定枚举值字符串（推荐与 Dify 工作流 inputs 变量保持一致，如 `"healthy"` / `"prediabetes"` / `"diagnosed"`），并标注"其他值将被服务端拒绝"；（2）在第 5 节表定义为枚举字段补充 CHECK 约束或枚举值说明；（3）同步检查 `family_history`、`gender` 等类似枚举字段，一并规范化。

---

- **问题**：前端 UI 组件库选型缺失，移动端优先的界面开发缺少关键依赖声明
- **原因**：文档明确平台为"移动端优先的 Web 应用"，定义了完整的设计系统基础参数，但未涉及 UI 组件库选型。移动端特有交互组件（Tabbar、ActionSheet、DatetimePicker、PullRefresh、Toast 等）从零开发工作量极大；设计系统参数需与组件库主题变量对接；8.1 节仅提及两个 npm 依赖易误导开发者。前端开发启动时面临组件选型的不确定性。
- **建议方向**：（1）在第 8.1 节明确前端 UI 组件库选型（推荐 Vant 4 作为移动端优先的 Vue3 组件库）；（2）在第 7.1 节设计系统参数中补充与所选组件库主题变量的映射关系；（3）若有意推迟选型至概要设计阶段，至少应在 8.1 节声明"移动端 UI 组件库选型待概要设计阶段确定"并列出评估维度（如移动端适配完整性、Vue3 兼容性、Bundle 体积、社区活跃度）。

---

- **问题**（轻微）：验收标准中时间指标的"合理预期"免责声明详细程度不一致
- **原因**：各验收标准中免责声明长度不一，增加了验收标准的解释歧义。虽然不影响功能性需求的完整性，但降低了验收标准作为"完成定义"的可操作性。
- **建议方向**：统一各验收标准中免责声明的措辞模板，引用 7.2 节的统一定义，避免在各验收标准中重复大段免责文本。

---

- **问题**（轻微）：SSE 事件格式的交叉引用链存在间接跳转（6.10→6.4→6.9）
- **原因**：虽然 6.9 节已标注其 SSE 事件结构"适用于所有 SSE 流式端点"，降低了实际查找成本，但 6.10 节的引用路径仍与 6.4 节同为引用方却路径不一致。实现者在阅读 6.10 节时需先跳转至 6.4 节，再跳转至 6.9 节，增加了不必要的查找步骤。
- **建议方向**：将第 6.10 节和第 6.4 节中的 SSE 事件结构引用均直接指向第 6.9 节，消除中间跳转。

---

- **问题**（轻微）：localStorage 跨浏览器标签页的登录态同步行为未定义
- **原因**：多标签页场景下，一个标签页登出后其他标签页的 Pinia store 不会自动感知此变更，可能导致标签页 B 仍显示已登录状态，用户尝试操作后收到 401 错误。虽然不阻塞核心功能开发，但影响多标签页使用场景下的用户体验一致性。
- **建议方向**：（1）在第 4.10 节补充跨标签页登录态同步策略：监听 `window.addEventListener('storage', ...)` 事件，当检测到 `token` 或 `role` 键在其他标签页中被修改或移除时，同步更新当前标签页的 Pinia store 状态；（2）将跨标签页同步的具体实现标注为概要设计阶段确定。
