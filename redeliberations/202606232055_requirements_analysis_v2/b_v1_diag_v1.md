### 问题 3：【阻塞性】前端架构存在全局性二义性——iframe 子页面模式与 Vue Router SPA 模式并存

- **问题描述**：文档同时描述了两套互斥的前端架构：（A）第 1 节（第 9 行）"各模块以 iframe 方式嵌入主框架"，配套第 1.1 节完整的 postMessage 跨模块通信协议；（B）第 1.2 节"平台采用 Vue3 SPA 架构，使用 Vue Router 4 管理前端路由"，配套完整的路由映射表和懒加载配置。这两套架构无法共存——如果使用 Vue Router SPA，模块间切换由 Vue Router 的 `<router-view>` 组件完成，不需要 iframe 和 postMessage；如果使用 iframe 子页面，则各模块是独立的 HTML 页面，Vue Router 无法管理它们的路由。
- **所在位置**：
  - iframe 架构声明：第 1 节第 9 行。
  - postMessage 协议规范：第 1.1 节，文档第 10-28 行。
  - Vue Router SPA 声明：第 1.2 节，文档第 30-77 行。
  - 该矛盾贯穿全文——第 4.10 节（第 310 行）同时出现了"Vue3 SPA 架构下，所有页面组件运行在同一 JavaScript 运行时中"和"iframe 子页面登录态共享"两种矛盾的场景描述。
- **严重程度**：阻塞性（Blocking）。前端架构是所有开发工作的基础决策。开发者拿到这份文档无法判断应按哪种模式开始编码——是用 Vue Router 构建单一 SPA，还是为每个模块创建独立的 HTML 页面并通过主框架的 iframe 加载。
- **改进建议**：
  1. 做出明确的技术决策：选择 Vue Router SPA（推荐，与 Vue3+TS+Vite 技术栈一致）或 iframe 多页面架构。
  2. 若选择 Vue Router SPA：删除第 1.1 节全部 iframe 通信机制描述；删除第 1 节"各模块以 iframe 方式嵌入主框架"的表述；将第 4.10 节中的 iframe 登录态共享描述替换为 Pinia store 统一管理方案；删除第 4.4 节中依赖 iframe 架构的 sessionStorage 跨页面持久化前提说明。
  3. 若保留 iframe 架构（与 Vue3 共存）：需明确说明每个 iframe 内部也是一个独立的 Vue3 应用（micro-frontend 模式），并说明主框架如何路由到对应的 iframe 页面。但这会增加大量架构复杂度，需评估对实训项目周期的影响。

## 四、实现者视角评估

### 是否可直接作为开发依据？

**否。** 以下三个未决问题使开发者无法开始编码：

1. **前端项目脚手架不明确**：是使用 `npm create vue@latest` 初始化 Vue3+Vite 项目，还是手工编写 HTML/CSS/JS 文件？文档同时给出了两种矛盾答案。
2. **模块间导航方式不明确**：是使用 Vue Router 的 `<router-link>` 和 `<router-view>`，还是为每个模块写独立 HTML 并通过主框架的 iframe 加载？这两种方式的目录结构、组件拆分逻辑、状态管理方案完全不同。
3. **目录结构不明确**：第 8.1 节推荐了 Vue3 标准目录结构（`src/views/`、`src/components/`、`src/router/` 等），但第 16.14 节的修订记录又推荐了原生开发的目录结构（`pages/`、`css/`、`js/`、`lib/`）。开发者无法确定应按哪种结构组织代码。

### 验收标准是否明确可量化？

验收标准本身较为明确（每个功能模块有 1-2 条含量化指标的标准），但验收标准的前置条件——前端架构——尚未确定，导致部分标准（如 4.1 节"首页在 375px 宽度的移动端视口中完整展示所有区域"）的验证方式取决于架构选择（iframe 模式下需考虑每个 iframe 内部的独立视口，SPA 模式下只需验证单一视口）。

### 是否存在隐含的模糊地带？

存在。最显著的模糊地带是 **iframe 在 Vue3 SPA 中的角色**：如果最终决策是保留 iframe 架构，那么每个 iframe 内部的页面是否也使用 Vue3 构建（形成嵌套 Vue3 应用）？如果每个 iframe 内部也使用 Vue3，那么路由是在主框架层（一个全局 Vue Router）管理还是在每个 iframe 内部（多个独立的 Vue Router）管理？文档对这些问题未提供任何说明。
