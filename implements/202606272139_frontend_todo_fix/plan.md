## 1. 总体策略

### 1.1 修复顺序原则

1. **P0 优先** — 功能阻断级问题（S5a/S5b），用户完全不可用，必须最先修复。
2. **P1 次之** — 高危级问题（S1/S2/S3/S7/S9/S6），影响核心体验但功能可降级运行。
3. **P2 后续** — 中危级问题（G14/S4/S11/S8），功能部分可用但有偏差。
4. **P3/P4 收尾** — 代码质量/可维护性迭代及设计文档更新。

### 1.2 优先级原则

- **硬性依赖优先**: S9 是 S3+S7 的前置条件；S5a 是 S6 的前置条件；S5b-1 是 S5b-2 的前置条件。
- **影响面优先**: G14（success字段检查）影响全部10个API函数，应在早期轮次引入。
- **批处理优先**: 同文件/同模块的修复合并处理（S1+S2、S3+S7、G7+G8+G12），减少上下文切换和合并冲突。
- **Store层先于View层**: 数据接口变更（如 G27 filter迁移）应先于模板修改，避免返工。

### 1.3 增量交付策略

每个里程碑完成后可独立部署，不阻塞后续里程碑。详见 8.2.1 节里程碑规划。

## 6. 风险提示

1. **S5b-1 (chatStore SSE) 为最大风险项**: 20-28h工作量，涉及 fetch+ReadableStream SSE管道、断线重连、流式渲染。建议由最有经验的开发者负责。
2. **Punch.vue 9项修复聚合风险**: G3/G6/G7/G8/G13/G15/G17/G24/G27 集中在同一文件，需同一开发者集中完成。
3. **G14 分阶段部署**: 建议先日志收集1-2周确认 `success: false` 语义，再切换为 reject，避免误伤合法空态。
4. **S8 sessionStorage 迁移的 UX 退化**: 跨标签页场景需重新登录，强烈建议同步实施 BroadcastChannel 增强（约30行代码）。
