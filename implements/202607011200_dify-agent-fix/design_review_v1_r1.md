## 发现

### 发现 1 — [一般] 前置条件声明错误：修改后并非"两种写法均正确"

**位置**：`detail_v1.md` 行为契约 > 前置条件 > 第 1 条

**原文**：
> `.env` 中已配置 `DIFY_API_BASE`（含 `/v1` 后缀或不含均可，修改后两种写法均正确）

**问题**：该声明在事实上是错误的。Dify 官方 Chat Message API 端点为 `/v1/chat-messages`。如果 `DIFY_API_BASE` 不含 `/v1` 后缀（例如配置为 `http://example.com`），修改后的拼接结果将是 `http://example.com/chat-messages`，缺少必要的 `/v1` 路径段，请求同样无法到达正确的 Dify 端点。

修改后的代码能正确工作的**唯一前提**是 `DIFY_API_BASE` 已包含 `/v1` 后缀——正如当前 `.env` 文件中的实际配置 `http://222.241.14.34:56487/v1`。同项目中 `difyService.js` 不追加 `/v1` 的前提也是 `DIFY_API_BASE` 已含此后缀，而非因为 `/v1` 可选。

**影响**：该错误声明可能误导未来维护者——如果有人在阅读此设计文档后认为"两种写法均可"，从而在修改 `.env` 时去掉 `/v1` 后缀，将导致与本次修复前类似的 404 故障（只是缺少一层 `/v1` 而非重复两层）。

**期望修正**：将前置条件修改为准确表述，例如：
> `.env` 中已配置 `DIFY_API_BASE`（必须包含 `/v1` 后缀，如 `http://222.241.14.34:56487/v1`），修改后 URL 拼接为 `{base}/chat-messages`，实际请求路径为 `/v1/chat-messages`。

### 发现 2 — [轻微] 设计文档未声明本轮范围边界

**位置**：`detail_v1.md` 概述

**问题**：需求文档列出了 3 个待修复问题（A: 双 `/v1` 路径拼接、B: 缺失 Agent 代理路由、C: API Key 配置冲突），但本设计文档仅覆盖问题 A，未在任何位置说明"本轮仅处理问题 A，问题 B/C 延后至后续轮次"。虽然任务文件 `task_v1.md` 的修订说明中明确了 R1/R2/R3 三轮规划，但设计文档作为独立交付物缺少此上下文，独立阅读时可能造成"设计未覆盖全部需求"的困惑。

**期望修正**：在概述节末尾增加一行范围声明，例如：
